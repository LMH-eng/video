# 视频上传服务端 + 客户端完整上传逻辑代码梳理

## 整体业务流程

1. 客户端上传窗口选择视频，FFmpeg 抽帧合成 GIF，封装**上传请求包 STRU_UPLOAD_RQ**（携带视频名、GIF 名、文件类型、用户 ID、分类标签 Hobby、文件总大小、唯一 FileId），TCP 发送给服务端；
2. 服务端`UploadRq`接收上传请求，创建 FileInfo 结构体，查询用户名、拼接视频 / GIF 本地存储路径，打开文件句柄存入全局`m_mapFileIDToFileInfo`映射；
3. 客户端分片循环发送`STRU_FILEBLOCK_RQ`文件分片包；
4. 服务端`UploadBlockRq`根据 FileId 查找 FileInfo，分片写入本地文件；文件全部接收完成后入库，前端刷新推荐列表加载所有本地视频文件。

void TcpKernel::UploadRq(int clientfd ,char* szbuf,int nlen)
{
    printf("clientfd:%d UploadRq\n", clientfd);
    // 强制转换为上传请求包头
    STRU_UPLOAD_RQ * rq = (STRU_UPLOAD_RQ *)szbuf;

    info->m_nFileID = rq->m_nFileId;
    info->m_VideoID = 0;
    info->m_nFileSize=rq->m_nFileSize;
    // 拷贝视频文件名、文件后缀
    strcpy( info->m_szFileName , rq->m_szFileName );
    strcpy( info->m_szFileType , rq->m_szFileType );

    // SQL查询用户名：根据用户ID查t_UserData表

        delete info;
        return;
    }
    // 查询无用户数据，释放内存直接返回
    if( resList .size() <= 0 )
    {
        delete info;
        return;
    }
    // 拷贝用户名
    strcpy ( info->m_UserName ,resList.front().c_str() );
    // 拼接视频本地完整存储路径 RootPath/用户名/文件名
    sprintf( info->m_szFilePath , "%sflv/%s/%s" , RootPath , info->m_UserName , info->m_szFileName );
    // 拼接RTMP播放地址
    sprintf ( info->m_szRtmp , "//%s/%s",RootPath , info->m_UserName , info->m_szFileName );
    // 判断当前上传文件是否为GIF动图，非GIF则填充GIF路径
    if( strcmp(rq->m_szFileType , "gif" ) != 0 )
    {
        strcpy( info->m_szGifName , rq->m_szGifName );
        sprintf( info->m_szGifPath , "%sflv/%s/%s" , RootPath , info->m_UserName , info->m_szGifName );

    // 以写模式打开本地文件，准备接收分片写入

    m_mapFileIDToFileInfo[ info->m_nFileID ] = info;

void TcpKernel::UploadBlockRq(int clientfd ,char* szbuf,int nlen)
{
    printf("clientfd:%d UploadBlockRq\n", clientfd);
    STRU_FILEBLOCK_RQ *rq =(STRU_FILEBLOCK_RQ *)szbuf;
    // 校验FileId是否存在上传任务，不存在直接返回丢弃分片
    if( m_mapFileIDToFileInfo .find ( rq->m_nFileId ) == m_mapFileIDToFileInfo.end() )
        return;
    // 根据文件ID取出缓存信息
    FileInfo* info = m_mapFileIDToFileInfo [ rq->m_nFileId ];

```
void OnlineDialog::UploadFile(QString filePath, Hobby hy , QString gifName )
{
    // 获取视频文件基础信息
    QFileInfo info(filePath);
    QString FileName = info.fileName();
    std::string strName = FileName.toStdString();
    const char* file_name = strName.c_str();

    // 实例化上传请求包头，构造自动清零所有字段
    STRU_UPLOAD_RQ rq;
    // 随机生成0~9999唯一文件ID
    rq.m_nFileId = qrand()%10000;
    // 填充视频总字节大小
    rq.m_nFileSize = info.size();
    // 拷贝视频文件名
    strcpy_s( rq.m_szFileName , MAX_PATH , file_name );

    // 截取文件后缀 mp4/flv/gif
    QByteArray bt = filePath.right( filePath.length() - filePath.lastIndexOf('.') -1 );
    memcpy(rq.m_szFileType , bt.data() , bt.length() );

    // 存在GIF动图路径则拷贝GIF文件名到包头
    if( !gifName.isEmpty() )
    {
        QFileInfo info(gifName);
        strcpy_s( rq.m_szGifName , MAX_PATH , info.fileName().toLocal8Bit().data() );
    }
    // 二进制拷贝8字节分类标签结构体
    memcpy( rq.m_szHobby , &hy , sizeof(hy) );
    // 填入当前登录用户ID
    rq.m_UserId = m_id;

    // TCP发送上传请求包头，告知服务端准备接收文件分片
    m_tcp->SendData( (char*)&rq , sizeof( rq ) );

    // 缓存文件信息，用于后续循环分片读取发送二进制
    FileInfo fi;
    fi.fileId = rq.m_nFileId;
    fi.fileName = FileName;
    fi.filePath = filePath;
}
```

# 四、你之前 BUG 根源：一个视频上传生成两个文件（视频 + GIF），服务端全部入库，前端列表双格子

## 问题定位

1. 客户端一次上传同时传递**视频文件 + GIF 缩略图**两个文件数据；
2. 服务端未区分主视频 / GIF 缩略图，收到任意文件都存入数据库视频表；
3. 前端刷新推荐列表时遍历服务端存储目录所有文件，GIF 也被当作独立视频渲染格子，一个视频占两个预览框。

## 修复方案

### 服务端 UploadRq 增加文件类型过滤判断

cpp

运行

```
// 在UploadRq函数末尾增加判断，仅非gif类型才入库、加入推荐列表
if( strcmp(rq->m_szFileType , "gif") == 0 )
{
    // GIF缩略图，仅保存到本地，不插入视频数据库、不加入推荐列表
    m_mapFileIDToFileInfo.erase(info->m_nFileID);
    delete info;
    return;
}
// 下方原有入库、缓存逻辑仅视频文件执行
```

### 前端刷新影视逻辑过滤后缀

前端请求推荐列表时，服务端筛选数据库中`filetype != "gif"`的数据返回，前端只渲染真实视频，过滤 GIF 缩略图条目。

### 传输区分主次

客户端上传时，**仅视频文件走完整分片上传入库流程**；GIF 缩略图仅附带路径信息同步存储，不单独生成一条视频数据记录。

// 上传文件分片处理
void TcpKernel::UploadBlockRq(int clientfd ,char* szbuf,int nlen)
{
    printf("clientfd:%d UploadBlockRq\n", clientfd);
    STRU_FILEBLOCK_RQ *rq =(STRU_FILEBLOCK_RQ *)szbuf;
    // 校验上传任务是否存在，不存在直接丢弃分片
    if( m_mapFileIDToFileInfo .find ( rq->m_nFileId ) == m_mapFileIDToFileInfo.end() )

## 六、配套业务 BUG 完整说明（截图内全部文字描述收录）

### 问题现象

客户端上传一个视频时，会同时上传**视频文件本体 + GIF 缩略图**两个文件；

服务端未做类型过滤，两个文件都会插入视频推荐数据库，前端推荐列表一个视频占用两个格子；

额外问题：.git 缓存、临时文件等杂文件也会被识别为视频展示。

### 根因

1. 客户端一次上传流程同时携带视频、GIF 两套文件数据；
2. 服务端 UploadRq、UploadBlockRq 无文件类型过滤逻辑，GIF 也执行完整入库流程；
3. 前端加载推荐列表时未过滤 gif 后缀数据。

### 修复逻辑

1. 服务端分片传输完成后增加判断：文件类型为 gif 时，仅本地保存文件，不执行 t_VideoInfo 入库；
2. 数据库查询推荐视频时，SQL 过滤`filetype != 'gif'`；
3. 客户端传输区分主次文件，仅视频文件走完整上传入库流程，GIF 仅作为附属缩略图存储。

## 七、全流程业务逻辑文字整理（所有截图业务说明汇总）

1. 用户注册：提交账号、MD5 密码、8 类视频喜好，服务端创建专属存储文件夹；
2. 客户端上传窗口选择本地视频，程序自动 FFmpeg 抽帧合成 GIF 缩略图；
3. 客户端封装`STRU_UPLOAD_RQ`上传首包，携带视频、GIF 名称、分类标签、文件总大小、随机 FileId；
4. 服务端`UploadRq`接收首包，查询用户名、拼接存储路径、创建本地空文件，存入`m_mapFileIDToFileInfo`；
5. 客户端循环拆分视频二进制，封装`STRU_FILEBLOCK_RQ`分片包持续发送；
6. 服务端`UploadBlockRq`循环写入本地文件，全部接收完成后关闭文件句柄；
7. 非 GIF 视频插入`t_VideoInfo`视频主表，返回上传成功响应给客户端；GIF 仅本地存储，不入库；
8. 前端刷新推荐列表，服务端查询`t_VideoInfo`返回所有视频数据，渲染视频预览格子；
9. 用户点击视频，`t_UserAction`新增播放记录，对应分类喜好分值 + 1，动态调整推荐权重；
10. `t_UserRecv`记录已推送视频，避免重复推荐，无新视频时清空重新推荐。

快速
