> **2. 创建总上下文**  
> 分配 AVFormatContext，它相当于视频文件的“大管家”，后面所有操作都围绕着它。

> **3. 打开文件**

> 
> **4. 获取流信息**

>

> 
> - 循环调用 av_read_frame 读出压缩包 AVPacket
>     
> - 送入解码器得到 YUV 帧
>     
> - 用 sws_scale 把 YUV 实时转成 RGB32
>     
> - 把 RGB 数据构造成 QImage，通过信号槽发给主线程 QLabel 显示

>     
>

> 
> 整个解码循环我放在了 QThread 子线程里，因为解码耗时严重，放主线程会阻塞 UI。主线程只负责接收图像、刷新界面和处理用户交互，这样完全符合 Qt 的线程模型。”

> “因为目前主流的视频编码标准，像 H.264、H.265，**内部压缩算法就是基于 YUV 设计的**。YUV 把亮度 Y 和色彩 UV 分离，人眼对亮度更敏感，所以编码时可以用更低的码率去压缩色彩信息，极大节省存储和传输带宽。解码器只是还原编码前的内容，所以解出来一定是 YUV。

### 四、面试官大概率追问的 3 个问题及答案

**1. AVPacket 和 AVFrame 有什么区别？**

> “AVPacket 存的是从文件读出来的压缩帧数据，体积小，还没解码；AVFrame 存的是解码后的原始像素数据，体积大，包含每个像素的完整信息。一个压缩、一个原始。”

**2. 为什么解码要放在子线程？**

> “Qt 的 UI 操作必须在主线程，而解码循环里全是 CPU 密集计算和 I/O 等待，放主线程会卡住事件循环，界面直接冻住。放到子线程后，主线程只管取图刷新，用户交互随时响应。”

**3. YUV 和 RGB 有什么区别？为什么解码器输出 YUV？**

> “YUV 是亮色分离格式，压缩率更高，适合传输存储；RGB 是物理三原色，适合直接显示。解码器输出 YUV 就是因为视频原始编码用的就是 YUV，我用 sws_scale 转成 RGB 再显示，转换上下文只创建一次循环复用。”

## 一、先搞懂：什么是 “流”？

你可以把**整个视频文件**想象成一个 **“混合快递大卡车”**：

- 这个卡车里不是只装一种货物，而是同时装了**3 种独立的 “货物流水线”**：
    
    1. **视频流**：装的是所有画面的压缩包（就是你要显示的图片）
    2. **音频流**：装的是所有声音的压缩包（说话、背景音乐）
    3. **字幕流**：装的是所有字幕的文字包（中文、英文、弹幕）
    
- 这 3 种 “流” 是**完全独立、分开打包、分开传输**的，就像卡车里的视频货物、音频货物、字幕货物，各装各的箱子，各走各的通道。
    

> 为什么要分开成 “流”？
> 
> 因为视频、音频、字幕的**播放速度、压缩方式、数据量完全不一样**：
> 
> - 视频是 25 帧 / 秒（每秒 25 张图）
> - 音频是 44100 次 / 秒（每秒采样 44100 次声音）
> - 字幕是几秒钟才出一行
>     
>     混在一起根本没法处理，分开成独立的 “流”，才能各自用最优的方式压缩和播放。

---

## 二、“流信息” 就是每个 “货物流水线” 的**身份证**

“流信息”，就是每个流的**详细说明书**，告诉 FFmpeg：

> “我这个流是什么类型的？用什么方式压缩的？有什么参数？”

就像每个快递箱上都贴了标签：“这是视频货物，用 H.264 压缩，尺寸 1920x1080，每秒 25 帧”。

---

## 三、你的代码里，“流信息” 到底存在哪里？有什么用？

### 1. 流信息存在哪里？

```
pFormatCtx->streams[i]
```

`pFormatCtx`是整个视频文件的 “大管家”，它有一个数组`streams`，**数组里的每一个元素，就是一个流的所有信息**（也就是一个`AVStream`结构体）。

比如一个普通的 MP4 文件，`streams`数组里可能有 3 个元素：

- `streams[0]`：视频流的信息
- `streams[1]`：音频流的信息
- `streams[2]`：中文字幕流的信息

### 2. 你代码里是怎么用流信息的？

你写的这段代码，就是在**从所有流里，找出 “视频流” 的身份证**：
```
// 遍历所有流
for ( i = 0; i < pFormatCtx->nb_streams; i++) {
    // 检查这个流的“类型”是不是视频
    if (pFormatCtx->streams[i]->codec->codec_type == AVMEDIA_TYPE_VIDEO)
    {
        videoStream = i; // 记录下视频流在数组里的位置
    }
}
```

- `pFormatCtx->nb_streams`：这个文件里总共有几个流（比如 3 个）
- `pFormatCtx->streams[i]->codec->codec_type`：这个流的 “类型”（是视频、音频还是字幕）

找到视频流之后，你就可以从它的 “身份证” 里，拿出解码需要的所有参数：
```
// 从视频流的身份证里，拿出解码器需要的信息
auto pCodecCtx = pFormatCtx->streams[videoStream]->codec;
```

这个`pCodecCtx`里，就包含了视频流的所有关键参数：

- `width`/`height`：视频的分辨率（比如 1920x1080）
- `pix_fmt`：视频的像素格式（比如 YUV420P）
- `r_frame_rate`：视频的帧率（比如 25fps）
- `codec_id`：视频的编码格式（比如 H.264、H.265）

FFmpeg 就是靠这些信息，才能找到对应的解码器，正确解码出图片。

- **流**：视频文件里独立的 “数据通道”（视频、音频、字幕）
- **流信息**：每个通道的 “身份证”，包含它的类型、编码、参数

## 一、先搞懂：I 帧 / P 帧 / B 帧 到底是什么？

### 核心问题：为什么视频不直接存一张张完整的图片？

你想一下：如果一个 1 分钟的视频，25 帧 / 秒，就是 1500 张 1920x1080 的图片，每张图片大概 5MB，那整个视频就是**7.5GB**，这显然太大了，根本没法传输和存储。

所以视频压缩的核心思路就是：**不要每张都存完整的图片，只存 “和上一张不一样的地方”**。I/P/B 帧就是实现这个思路的三种帧类型。

---

### 1. I 帧：完整的 “基准照片”（关键帧）

- **比喻**：你去旅游，每隔 10 张拍一张**完整的风景照**，这就是 I 帧。
- **特点**：
    
    - 包含一整幅完整的图像信息，不需要参考任何其他帧就能解码
    - 体积最大，但解码最快
    - 可以单独存在，所以视频的开头、快进的位置，一定是 I 帧
    
- **你的代码里的作用**：如果没有 I 帧，后面所有的 P 帧和 B 帧都解不出来，视频就会花屏。

---

### 2. P 帧：只存 “和上一张不一样的地方”（差异帧）

- **比喻**：你拍了一张完整的风景照（I 帧），下一张照片里，只有云动了，其他地方都没变。那你不用再拍一张完整的，只需要写一句 “云向右移动了 10 像素”，这就是 P 帧。
- **特点**：
    
    - 只参考**前面的 I 帧或 P 帧**进行编码
    - 体积比 I 帧小很多（大概是 I 帧的 1/10）
    - 不能单独存在，必须依赖前面的帧才能解码
    

---

### 3. B 帧：同时参考 “前面和后面的帧”（双向差异帧）

- **比喻**：你拍了第 1 张（I 帧）和第 4 张（P 帧），中间的第 2、3 张，你可以同时看第 1 张和第 4 张，然后写一句 “第 2 张是第 1 张和第 4 张的中间状态”，这就是 B 帧。
- **特点**：
    
    - 同时参考**前面的 I/P 帧**和**后面的 I/P 帧**
    - 体积最小（大概是 I 帧的 1/20），压缩率最高
    - **解码最慢**，因为要等后面的帧解码完才能解自己
    - **不能作为参考帧**（其他帧不能参考 B 帧）
    

---

### 为什么要有 B 帧？

因为 B 帧的压缩率最高，能让视频文件变得更小。但它有一个致命的副作用：**解码顺序和显示顺序不一样**。这就是所有同步问题的根源。

---

## 二、最关键的点：为什么解码顺序 ≠ 显示顺序？

看你图里的这个例子，我把它简化成最容易懂的 4 帧：

plaintext

```
显示顺序（我们眼睛看到的顺序）：I [0] → B [1] → B [2] → P [3]
```

按照正常逻辑，我们应该按 1→2→3→4 的顺序解码对吧？但不行！因为：

- B [1] 和 B [2] 都需要参考后面的 P [3] 才能解码
- 如果你先解 B [1]，这时候 P [3] 还没解出来，你根本不知道 B [1] 长什么样

所以**实际的解码顺序**必须是：

plaintext

```
解码顺序（解码器处理的顺序）：I [0] → P [3] → B [1] → B [2]
```

这就是你图里那些**交叉箭头**的意思！解码器先把后面的 P 帧解出来，再回头解中间的 B 帧，最后按显示顺序把帧排好，给屏幕显示。

---

## 三、PTS 和 DTS：两个时间戳，解决 “什么时候解，什么时候显示”

因为解码顺序和显示顺序不一样，所以每一帧都需要两个时间戳来标记：

表格

|缩写|全称|比喻|作用|
|:--|:--|:--|:--|
|**DTS**|解码时间戳|**“拆快递的时间”**|告诉解码器：“你应该在这个时间点，把这个包裹拆开”|
|**PTS**|显示时间戳|**“看礼物的时间”**|告诉播放器：“你应该在这个时间点，把这张图片显示在屏幕上”|

还是用上面的 4 帧例子：

表格

|帧|显示顺序|解码顺序|DTS（拆快递时间）|PTS（看礼物时间）|
|:--|:--|:--|:--|:--|
|I[0]|1|1|0ms|0ms|
|B[1]|2|3|60ms|20ms|
|B[2]|3|4|80ms|40ms|
|P[3]|4|2|20ms|60ms|

你看！B 帧的 DTS（解码时间）比 PTS（显示时间）大，因为它要等后面的 P 帧先解码完。

## 四、为什么你之前的代码会 “播放过快、不同步”？

你之前的代码里写了：

cpp

运行

```
msleep(5); // 固定等5毫秒再显示下一帧
```

这个写法**只在没有 B 帧、所有帧间隔都一样的理想情况下才有用**，但实际视频里：

1. 有 B 帧，解码顺序和显示顺序不一样
2. 每帧的间隔不是固定的（比如有的视频是 24fps，有的是 30fps，有的甚至是动态帧率）
3. 解码一帧的时间也不是固定的（I 帧解码慢，B 帧解码快）

所以固定延时会导致：

- 有时候解码太快，视频播放得比正常速度快很多
- 有时候解码太慢，视频卡顿、音画不同步

---

## 五、图里的 “简易同步思路” 到底是怎么解决问题的？

核心逻辑非常简单：**不要按解码速度显示，按 PTS（应该显示的时间）显示**。

我把它翻译成大白话：

1. 你有一个 **“手表”**（系统时钟，或者音频的时钟，因为音频是线性的，不会跳）
2. 每次解码出一帧，先看它的**PTS**（这帧应该在 “手表显示 10 秒的时候” 显示）
3. 然后看你的手表现在是几点：
    
    - 如果手表现在是**9.99 秒**：说明还没到显示时间，你就等一会儿（msleep (10)）
    - 如果手表现在是**10.01 秒**：说明已经晚了，别等了，直接显示这帧
    - 如果手表现在是**10.5 秒**：说明已经晚了半秒，直接把这帧丢了，显示下一帧（追赶上时间）
    

---


## 二、DTS 到底有什么用？（3 个核心作用，2 个和同步直接相关）

### 1. 最基础：解码器的 "指挥棒"，没有它视频根本解不出来

这是 DTS 最核心、最不可替代的作用。

解码器是个 "傻子"，它不知道哪帧该先解，哪帧该后解。它只会**严格按照 DTS 从小到大的顺序**来解码帧：

- 先解 DTS=0ms 的 I [0]
- 再解 DTS=20ms 的 P [3]
- 再解 DTS=40ms 的 B [1]
- 最后解 DTS=60ms 的 B [2]

如果没有 DTS，解码器面对一堆乱序的帧，根本不知道该先解哪一个。强行按接收顺序解码，会因为缺少参考帧（比如 B [1] 需要 P [3]）而解不出来，直接花屏。

> 注意：你现在写的代码里看不到 DTS 的处理，是因为 FFmpeg 内部已经帮你做了 —— 它会自动按 DTS 顺序把帧喂给解码器。但这不代表 DTS 不存在，只是它被封装在底层了。

---

### 2. 和同步直接相关：播放器的 "缓冲调度器"

一个合格的播放器，不能 "解一帧，显示一帧"，必须提前解码一些帧放在缓冲区里，这样播放的时候才不会卡顿。

DTS 就是用来告诉播放器：**你应该提前多久解码这一帧**。

- 比如一帧的 DTS 是 1000ms，PTS 是 1040ms
- 播放器就知道：我需要在 1000ms 的时候开始解码这一帧，这样才能在 1040ms 的时候，把解码好的帧显示出来

如果 DTS 错了，会直接导致同步问题：

- DTS 太早：播放器提前解码了太多帧，浪费内存
- DTS 太晚：到了 PTS 显示时间，帧还没解码完，导致卡顿、跳帧

---

### 3. 和同步直接相关：音视频同步的 "基准参考"

当你给播放器加上音频，做真正的 "音画同步" 时，DTS 就变得至关重要了。

因为**音频没有 B 帧**，音频的解码顺序和显示顺序是完全一样的，所以音频的 DTS 和 PTS 永远相等。这意味着：

- 音频的播放时钟，本质上就是基于 DTS 的
- 视频需要追赶的 "主时钟"，就是音频的 DTS 时钟

虽然最终音视频对齐是用 PTS 来比较的（"视频帧的 PTS" 和 "音频当前的 PTS"），但整个同步系统的时间基准，是由音频的 DTS 驱动的。

---

## 三、面试怎么说这个问题？（满分回答）

> 视频的显示同步本身确实主要使用 PTS，因为 PTS 标记的是帧应该被用户看到的时间，而 DTS 标记的是帧应该被解码的时间。
> 
> 但 DTS 绝对不是没用，它是整个解码流程的基础：
> 
> 1. 首先，因为 B 帧的存在，解码顺序和显示顺序不一样，解码器必须严格按照 DTS 的顺序来解码帧，否则会因为缺少参考帧而解不出来
> 2. 其次，播放器的缓冲管理依赖 DTS，它告诉播放器应该提前多久解码帧，保证在 PTS 到来时帧已经准备好
> 3. 最后，在音视频同步中，音频的 DTS 是整个同步系统的时间基准，视频需要追赶音频的 DTS 时钟

## 新增亮点：PTS 同步（你代码里的精华）

cpp

int64_t start_time = av_gettime();        // 播放开始时的系统微秒时间
int64_t frame_pts = 0;
AVRational time_base = pFormatCtx->streams[videoStream]->time_base;
// 每帧计算显示时刻
int64_t pts = pFrame->best_effort_timestamp;
frame_pts = pts * 1000000LL * av_q2d(time_base);  // 转为微秒
// 等待直到系统时间 >= 帧应显示的时间
while (frame_pts > current_time) {
    QThread::msleep(5);
    current_time = av_gettime() - start_time;
}

- 这是**视频同步播放的核心**：不用固定延时，而是让每一帧严格按照它的 PTS 展示。
    
- `best_effort_timestamp` 保证即使无 PTS 也能推算，兼容性强。
    
- 等待循环用 `msleep(5)` 小步等待，既不过于频繁，也不会延迟过大。
    
- 面试官如果问你“怎么保证播放速度正确”，你就能拿出这段逻辑来。

---

### 1. 你的代码用没用到 DTS？

**代码里确实没有显式处理 DTS，但这不意味着“没用上”。**

- 你用的是 `pFrame->best_effort_timestamp`，它和 `pFrame->pts` 的区别在于：  
    当视频帧**缺少 PTS** 时，`best_effort_timestamp` 会利用 DTS 和已知的帧间隔**推算出一个近似的 PTS**，还能**对 B 帧进行重排序**。
    
- 也就是说，FFmpeg 内部已经帮你用 DTS 做好了“解出正确显示顺序”的工作，最终给出一个可以直接用于显示的“虚拟 PTS”，你再用它去同步。
    
- 所以，不是你主动处理 DTS，而是你借了 `best_effort_timestamp` 这个封装好的接口，**间接用到了 DTS 的能力**，比自己手动处理更可靠。
    

面试时可以这么总结：

> “我没有直接操作 DTS，但使用的是 `best_effort_timestamp`，它内部会利用 DTS 对 B 帧重排序，给出正确的显示时刻，避免了手动处理 DTS/PTS 的复杂逻辑。”

---

### 2. 为什么是 `>=` 播放，而不是 `==` 才播放？

核心原因就一句话：**系统时间不可能精确踩准每一帧的理论显示时刻**。

详细解释：

- 视频帧率是 25fps，每帧间隔 40,000 微秒。理论上是 0、40000、80000... 播放。
    
- 但你的等待循环：
    
    cpp
    
    while (frame_pts > current_time)
        QThread::msleep(5);
    
    每次醒来检查，系统时间都是“离散跳跃”的，不是连续流。你很可能出现：
    
    - 当前时间 `frame_pts = 40000`，
        
    - 醒来一看系统时间已经是 `40015`，
        
    - 如果要求 `==`，就永远错过了，直接卡死在这一帧。
        

因此必须用 `>=`，含义是：

> “这帧应该显示的时刻已经到达或刚刚过去，现在立刻显示它，不要错过播放窗口。”

用 `>=` 还能带来两个附加好处：

- 即便解码/调度有微小波动，也能**自动追赶**：如果连续几帧处理稍慢，后续帧的 `frame_pts` 都会小于当前时间，就会连续快速显示，把延迟“追”回来。
    
- 避免了因等待过于精确导致的额外延迟抖动，播放更平滑。
    

面试时可以直接说：

> “因为系统时间的精度和调度延迟，不可能恰好等于每一帧的理论 PTS。所以用 `>=` 更稳健：一旦检测到这一帧的显示时间已到达或刚过，就立刻显示，既能保证不卡帧，也能利用后续帧自动追赶可能的延迟。”
