# 04 音视频混合播放器

## 一、整体架构

### 三线程模型

```
┌─────────────────────────────────────────────────┐
│                  主线程 (run)                     │
│                                                   │
│   av_read_frame() 读压缩包                        │
│        │                                          │
│        ├── 视频包 ──→ videoq 队列 ─────────────┐ │
│        │                                         │ │
│        └── 音频包 ──→ audioq 队列 ──────────------─┐  │ │
│                                               │  │ │
│   队列满了就 SDL_Delay 等待                     │  │ │
│   防止一口气读完占满内存                         │  │ │
└───────────────────────────────────────────────┼──┼─┘
                                                │  │
         ┌──────────────────────────────────────┘  │
         │                                         │
         ▼                                         ▼
┌─────────────────┐                    ┌─────────────────┐
│  video_thread    │                    │  SDL音频回调      │
│  (SDL线程)       │                    │  (SDL内部线程)    │
│                  │                    │                   │
│  从videoq取包    │                    │  从audioq取包     │
│  解码→YUV转RGB   │                    │  解码→重采样      │
│  emit信号→UI显示 │                    │  填充SDL缓冲区    │
└─────────────────┘                    └─────────────────┘
```

**三个线程各司其职：**

| 线程 | 职责 | 角色 |
|------|------|------|
| 主线程 | 读文件，把压缩包分发到队列 | 生产者 |
| 视频线程 | 从视频队列取包，解码，转RGB，通知UI显示 | 消费者 |
| 音频回调线程 | SDL需要声音数据时自动调用，从音频队列取包解码播放 | 消费者 |

**为什么需要队列？** 读文件速度 >> 解码速度 >> 播放速度。如果不用队列，读出来的包没地方放；队列设上限（`MAX_AUDIO_SIZE`/`MAX_VIDEO_SIZE`）防止内存爆炸。

## 二、核心数据结构

### VideoState — 全局播放状态

```cpp
typedef struct VideoState {
    AVFormatContext  *pFormatCtx;     // 文件上下文
    int              videoStream;     // 视频流索引
    int              audioStream;     // 音频流索引
    AVStream        *video_st;        // 视频流
    AVStream        *audio_st;        // 音频流
    AVCodecContext  *pCodecCtx;       // 视频解码器上下文
    AVCodecContext  *pAudioCodecCtx;  // 音频解码器上下文
    PacketQueue     *videoq;          // 视频包队列
    PacketQueue     *audioq;          // 音频包队列
    SDL_Thread      *video_tid;       // 视频解码线程
    SDL_AudioDeviceID audioID;        // SDL音频设备ID
    VideoPlayer     *m_player;        // 指向VideoPlayer，用于emit信号
    int              quit;            // 退出标志

    uint8_t          audio_buf[...];  // 音频解码缓冲区
    unsigned int     audio_buf_size;  // 缓冲区有效数据大小
    unsigned int     audio_buf_index; // 当前读取位置
    AVFrame         *audioFrame;      // 音频解码帧（复用）
    AVFrame          out_frame;       // 音频输出参数（格式/采样率/声道）
    int64_t          audio_clock;     // 音频时钟（用于同步）
} VideoState;
```

### PacketQueue — 生产者消费者队列

```cpp
typedef struct PacketQueue {
    AVPacketList *first_pkt;   // 队头

    SDL_mutex *mutex;          // 互斥锁
```

## 四、FFmpeg 3.x → 4.x API 变化详解

### 4.1 获取解码器上下文

```
旧版 (FFmpeg 3.x):                     新版 (FFmpeg 4.x):
streams[i]->codec                       streams[i]->codecpar
    ↑ 已废弃，指针为空！                    ↑ 正确的方式
```

**旧代码（崩溃）：**
```cpp
auto pCodecCtx = pFormatCtx->streams[videoStream]->codec;
```

**新代码（正确）：**
```cpp
auto pCodecCtx = avcodec_alloc_context3(NULL);
avcodec_parameters_to_context(pCodecCtx, pFormatCtx->streams[videoStream]->codecpar);
```

`codec` 成员在 FFmpeg 4.x 中不再自动分配内存，直接访问是**空指针**，程序立刻崩溃。

### 4.2 解码API

```
旧版 (一对多模式):                      新版 (分离模式):
avcodec_decode_video2()                 avcodec_send_packet() + avcodec_receive_frame()
avcodec_decode_audio4()                 avcodec_send_packet() + avcodec_receive_frame()
```

**旧代码：**
```cpp
int got_picture;
avcodec_decode_video2(ctx, frame, &got_picture, packet);
if (got_picture) { ... }
```

**新代码：**
```cpp
avcodec_send_packet(ctx, packet);           // 把压缩包送进解码器
int ret = avcodec_receive_frame(ctx, frame); // 从解码器取出解码后的帧
if (ret == 0) { ... }  // 成功拿到一帧
```

**为什么FFmpeg要这么改？** 因为一个packet可能产生多帧（B帧场景），旧API处理不了这种情况。新API可以连续调用 `receive_frame` 取出所有帧。

### 4.3 其他废弃API对照表

| 废弃API (3.x) | 替代API (4.x) | 说明 |
|---------------|---------------|------|
| `av_register_all()` | 不需要调用 | 4.x自动注册 |
| `av_new_packet()` | `av_packet_alloc()` | 分配packet |
| `av_free_packet()` | `av_packet_unref()` | 释放packet数据 |
| `av_dup_packet()` | `av_packet_ref()` | 引用packet |
| `avcodec_close()` | `avcodec_free_context()` | 关闭并释放解码器上下文 |
| `av_free(pFrame)` | `av_frame_free(&pFrame)` | 释放帧 |
| `avpicture_get_size()` | `av_image_get_buffer_size()` | 计算图像大小 |
| `avpicture_fill()` | `av_image_fill_arrays()` | 填充图像数据 |
| `memset(frame,0,sizeof(AVFrame))` | `av_frame_unref(frame)` | 重置帧（绝对不能用memset！） |

## 五、遇到的错误逐一详解

### 错误1：点击开始程序直接崩溃

**现象**：程序启动后点击"开始"按钮立即崩溃

**原因**：访问了废弃的 `streams[i]->codec` 指针

```cpp
// 这行代码在 FFmpeg 4.x 中 codec 指针为空，直接崩溃
auto pCodecCtx = pFormatCtx->streams[videoStream]->codec;
```

**修复**：用 `codecpar` + `avcodec_alloc_context3`

---

### 错误2：`Error in decoding audio frame` 循环打印后崩溃

**现象**：控制台不断输出音频解码错误，最终程序崩溃

**原因**：`avcodec_decode_video2`/`avcodec_decode_audio4` 是废弃API，与新版 `avcodec_alloc_context3` 创建的上下文不兼容，调用必然返回错误

**修复**：换成 `avcodec_send_packet` + `avcodec_receive_frame`

---

### 错误3：`Assertion ... private_ref ... failed at decode.c:655`

**现象**：程序断言失败崩溃

**原因**：
```cpp
memset(audioFrame, 0, sizeof(AVFrame));  // 致命错误！
```

`avcodec_receive_frame` 会给 `AVFrame` 设置内部的引用计数（`private_ref`），这是FFmpeg管理内存的核心机制：

```
avcodec_receive_frame(audioFrame)  →  audioFrame.private_ref = 0x12345678
memset(audioFrame, 0, ...)         →  audioFrame.private_ref = 0   ← 破坏了！
下次再操作audioFrame               →  FFmpeg断言失败，崩溃
```

**修复**：
```cpp
av_frame_unref(audioFrame);  // 正确释放，引用计数-1，而不是暴力清零
```

---

### 错误4：播放完一次后，再次点击开始只播放1秒就停

**现象**：第一次播放正常，第二次点击开始后只能播放约1秒

**原因**：`QThread` 的机制限制

```
m_player->start()  →  run()开始执行
run()执行完毕       →  线程结束，状态变为 Finished
再次 m_player->start()  →  无效！Finished的线程不能再start
```

**修复**：每次点击时销毁旧的，创建新的：
```cpp
delete m_player;
m_player = new VideoPlayer;
m_player->start();
```

---

### 错误5：`Timers cannot be stopped from another thread`

**现象**：再次点击开始时出现Qt警告

**原因**：旧的视频线程还在运行，主线程就 `delete m_player`：
- 旧线程还在用 `emit is->m_player->SIG_getOneImage()`
- 但 `m_player` 已经被 delete 了
- 访问已销毁的QObject → 崩溃/警告

**修复**：
```cpp
m_player->stop();   // 设置quit标志 + 中止队列
m_player->wait();   // 等待线程完全退出
delete m_player;    // 这时才安全删除
```

---

### 错误6：旧线程卡死在 `packet_queue_get` 退不出

**现象**：点击开始后程序卡死

**原因**：视频线程调用 `packet_queue_get(block=1)` 时：
```cpp
SDL_CondWait(queue->cond, queue->mutex);  // 队列空时，永远等在这里
```

`stop()` 只是设置了 `quit=1`，但视频线程**看不到**这个标志——它卡在 `SDL_CondWait` 上，根本没机会检查 `quit`。

**修复**：新增 `packet_queue_abort`：
```cpp
void packet_queue_abort(PacketQueue *queue) {
    queue->abort_request = 1;        // 设置中止标志
    SDL_CondBroadcast(queue->cond);  // 唤醒所有等待的线程
}

int packet_queue_get(...) {
    while(1) {
        if (queue->abort_request)    // 检查中止标志
            return -1;               // 立即退出
        ...
    }
}
```

---

### 错误7：`deprecated pixel format used` 警告

**现象**：控制台输出swscaler警告

**原因**：`AV_PIX_FMT_RGB32` 在新版中建议用带范围说明的格式

**说明**：这是**警告**不影响运行，要消除可改用 `AV_PIX_FMT_BGR0`

## 六、关键知识点总结

### 6.1 FFmpeg解码流程（4.x）

```
avformat_open_input()          打开文件
       │
avformat_find_stream_info()    获取流信息
       │
avcodec_alloc_context3()       分配解码器上下文
       │
avcodec_parameters_to_context() 从codecpar复制参数
       │
avcodec_find_decoder()         查找解码器
       │
avcodec_open2()                打开解码器
       │
循环 {
    av_read_frame()            读取压缩包
    avcodec_send_packet()      发送到解码器
    avcodec_receive_frame()    取出解码帧
}
```

### 6.2 音视频同步

音视频同步的核心思想：**以音频时钟为主时钟，视频帧追赶音频**

```
视频帧的PTS（显示时间戳）
    │
    ├── frame_pts > current_time → 等待（还没到显示时间）
    │
    └── frame_pts <= current_time → 立即显示（已经到时间了）
```

音频时钟计算：
```cpp
is->audio_clock = audioFrame->pts * av_q2d(is->audio_st->time_base) * 1000000;
```

### 6.3 队列大小限制

```cpp
#define MAX_AUDIO_SIZE (1024*16*25*10)  // 约4MB
#define MAX_VIDEO_SIZE (1024*255*25*2)  // 约12.5MB
```

当队列超过阈值时，读包线程暂停（`SDL_Delay`），等消费者消费后再继续。这样：
- 不会一口气读完整个文件导致内存溢出
- 保证队列中始终有数据供解码线程消费，不卡顿

### 6.4 安全退出流程

```
用户再次点击"开始"
    │
    ▼
stop()
    ├── quit = 1                通知所有线程要退出了
    ├── packet_queue_abort()    唤醒阻塞在队列上的线程
    │
    ▼
wait()                          等待run()线程完全退出
    │
    ▼
delete m_player                 线程已退出，安全删除
    │
    ▼
new VideoPlayer + start()       创建新实例开始播放
```

## 七、文件结构

| 文件 | 职责 |
|------|------|
| `videoplayer.h` | VideoState结构体、VideoPlayer类声明、全局函数声明 |
| `videoplayer.cpp` | 主线程run()、视频线程video_thread()、音频回调audio_callback()、音频解码audio_decode_frame() |
| `PacketQueue.h/cpp` | 生产者消费者队列（init/put/get/abort/flush） |
| `playerdialog.h/cpp` | Qt对话框，按钮事件、图像显示 |
| `main.cpp` | 程序入口 |


**视频同步的本质是“把视频帧的 PTS 和一个不断前进的参考时钟进行比较，决定等待还是丢弃”。**

- 那个不断前进的参考时钟，可以是**音频时钟**，也可以是**系统时钟**，但**绝对不是视频帧自己的 PTS**。
    
- 你之所以感到困惑，是因为 PTS 是“帧的标签”，而时钟是“现在几点了”。没有时钟，PTS 就只是一个数字，没法判断是超前还是落后。

## 一、先厘清两个概念的严格定义

|术语|定义|类比|
|---|---|---|
|**PTS**（显示时间戳）|视频或音频帧自己身上带的“理想显示时刻”，是一个绝对或相对的时间值，源于编码时的时间信息。|你电影票上印的“14:30 场”。|
|**时钟**（Clock）|播放过程中持续前进的“当前播放进度”，是实际时间的参照物。|电影院墙上挂的钟，显示现在真实时间 14:28。|

**同步逻辑**：你把电影票上的“14:30”和墙上钟的“14:28”做比较，知道还差 2 分钟才开场，于是去逛一圈。  
如果墙上没有钟，你只有一张写着“14:30”的票，你根本不知道什么时候该进场。

## 二、视频播放中，参考时钟从哪来？

取决于有没有音频流：

### 1. 有音频时 → 参考时钟 = **音频时钟**

- 因为声卡硬件会定时消费音频数据，音频的播放节奏极其精准。
    
- 我们在 `audio_decode_frame` 里，每解码出一帧音频，就用它的 PTS 更新 `is->audio_clock`，代表“当前音频已经播到哪了”。
    
- 视频线程在显示每一帧前，拿着自己的 `video_pts` 和 `audio_clock` 比较：
    
    - 如果 `video_pts > audio_clock`：视频超前，等待音频追上来。
        
    - 如果 `video_pts <= audio_clock`：视频正点或稍落后，立即显示。
        

这就是你之前看到的 `while (video_pts > audio_clock) { SDL_Delay(5); }` 那一段，它出现在 **`timer_callback` 里的同步逻辑**，也应该是任何音视频播放器中视频线程的标准做法。

### 2. 无音频时 → 参考时钟 = **系统时钟**

- 没有声卡驱动的音频时钟，就必须自己造一个。
    
- 典型的做法：记录播放开始的系统时间 `start_time`，然后每一帧的目标显示时刻 = `start_time + video_pts`。当前系统时间 `av_gettime()` 就是这个“自制时钟”。
    
- 比较：`target_time` 与 `current_time`，等待直到 `current_time >= target_time`。
    

你给出的 `video_thread` 代码，恰恰就是用的**系统时钟同步**：

cpp

int64_t start_time = av_gettime();   // 自制时钟的起点
...
int64_t pts = pFrame->best_effort_timestamp;
frame_pts = pts * 1000000LL * av_q2d(time_base);
int64_t current_time = av_gettime() - start_time;
while (frame_pts > current_time) {
    SDL_Delay(5);
    current_time = av_gettime() - start_time;
}

这里完全没有用到 `audio_clock`，是一个**纯视频播放（无音频）的同步策略**。它把系统时间作为钟，视频帧的 PTS 作为显示时刻，两者对比实现同步。

## 三、那你的代码里为什么 `video_thread` 没使用音频时钟？

这正是这段代码的一个**设计瑕疵/简化**。  
你的主线程 `run()` 里有判断：只要存在音频流，就会创建视频线程，但 `video_thread` 内部却完全无视音频时钟，依然用系统时钟做同步。这在实际播放带音频的文件时，**无法实现音画同步**（嘴唇对不上声音），因为视频和音频各自用各自的时钟，会逐渐漂移。

**正确的做法应该是**：视频线程的同步参考时钟需要动态选择——有音频时用 `is->audio_clock`，无音频时用系统时钟。  
你之前在 `timer_callback` 片段里看到的那个 `if(is->audioStream != -1) { while... }` 循环，才是音画同步的意图。`video_thread` 里缺失了这段逻辑。

你可以这么改进 `video_thread` 里的同步等待：

cpp

if (is->audioStream != -1) {  // 有音频，以音频时钟为准
    while (video_pts > is->audio_clock) {
        SDL_Delay(5);
    }
} else {  // 无音频，以系统时钟为准
    int64_t target = start_time + video_pts;
    while (av_gettime() < target) {
        SDL_Delay(5);
    }
}

## 四、回到你的困惑：“视频到底是咋同步的？”

一句话总结：**视频的同步就是“PTS + 主时钟”的持续比较游戏**。

- PTS 告诉你这一帧应该在哪个时间点显示。
    
- 主时钟告诉你现在时间进行到哪了。
    
- 比较结果决定了这一帧是立刻显示、等待，还是已经太晚需要丢弃。
    
- 主时钟在有音频时取自音频时钟，无音频时取自系统时钟。
    

你之前说的“PTS同步”和“时钟同步”其实是一回事：

- “PTS同步”是口语化的说法，强调用 PTS 来控制显示时机。
    
- “时钟同步”更严谨，指出了必须有一个外部时钟（音频或系统）作为基准。
    

现在你应该能清晰地区分：**PTS 是帧自己的属性，时钟是外部裁判**。视频同步就是让帧的 PTS 追上裁判的计时。


## 一、回调函数和解码函数是“消费者”里的两层

在生产者-消费者模型里，**音频的消费者由SDL回调线程担当**，它包含两层逻辑：

SDL 回调线程（消费者）
    │
    ├─ audio_callback          ← 外层：响应声卡中断，必须立刻返回
    │     │
    │     └─ audio_decode_frame ← 内层：从队列取包、解码、重采样，填满内部缓冲区
    │
    └─ 数据写入 stream，声卡播放

- **`audio_callback`** 是 SDL 直接调用的入口，是“取餐铃”响时要你做的事。
    
- 它为了保证自己能在极短时间内返回，不会直接去读队列、解码，而是先检查一个内部缓冲区 `is->audio_buf`。如果缓冲区的数据够用，就直接拷贝给 SDL；如果不够，才会调用 `audio_decode_frame` 去补充缓冲区。
    
- **`audio_decode_frame`** 是真正执行“从队列取包→送入解码器→重采样→填充音频缓冲区”的地方。它虽然被回调调用，但逻辑上属于消费者的核心工作。
    

所以你说得对：**它们都在消费者线程里执行**，只不过 `audio_callback` 是调度层，`audio_decode_frame` 是执行层。

---

## 二、为什么音频不像视频那样“解码→转换→发信号”？

视频和音频在这里走上了两条完全不同的路，根本原因是**数据最终的去向不同**：

|视频|音频|
|---|---|---|
|**数据最终要去的对象**|GUI 控件（如 QLabel）|声卡硬件|
|**消费方式**|主线程主动取图显示|SDL 线程被动要数据|
|**传递方式**|通过 Qt 信号槽把 `QImage` 发给主线程|直接在回调里把 PCM 数据填入 SDL 提供的 `stream` 缓冲区|
|**同步信号**|需要，因为跨线程操作 UI|不需要，回调已经在正确线程里直接写数据|

视频之所以要“发信号”，是因为解码在子线程，而 Qt 规定只能在主线程更新 UI。  
音频之所以“不发信号”，是因为 SDL 的回调本身就运行在它自己创建的高优先级音频线程里，声卡驱动直接从回调写入的 `stream` 缓冲区读取数据，**不存在跨线程 UI 操作的需求**。

你可以把 `stream` 看成 SDL 给你准备的一个“直达声卡的快递箱”，你填进去，它就直接播放，不需要再经过主线程或 UI 框架。

---

## 三、回调函数的作用（再帮你钉死记忆）

回调函数的核心作用就一句话：  
**SDL 定时向你索要数据，你必须立刻把指定长度的 PCM 音频数据交出来，不能有任何耽搁。**

- SDL 通过 `wanted_spec.callback = audio_callback` 注册了这个函数指针。
    
- 声卡每播完 1024 个样本（约 23ms），SDL 就在自己的音频线程里调用一次 `audio_callback`，传给你一个空的 `stream` 和长度 `len`。
    
- 你在这个函数里必须把 `len` 字节的数据填进去，然后函数返回。
    
- 如果在这个函数里做耗时操作（如读文件、网络请求），导致返回超时，声卡就会因为没有数据而输出噪声或静音（爆音）。
    

它就是那个“声卡的取餐铃”，不能让它等你。

---

## 四、一个完整的音频数据流，从文件到耳朵

帮你串一遍，就知道每个函数在哪一步：

1. **主线程**（生产者）：  
    `av_read_frame` → 得到音频 `AVPacket` → `packet_queue_put` 放入 `audioq`
    
2. **SDL 回调线程**（消费者）：
    
    - SDL 触发 `audio_callback(stream, len)`
        
    - 检查内部缓冲区 `is->audio_buf`，如果空了：
        
        - 调用 `audio_decode_frame`
            
            - `packet_queue_get(audioq)` 取出一个压缩包
                
            - `avcodec_send_packet` + `avcodec_receive_frame` 解码成原始 PCM
                
            - 用 `swr_convert` 重采样成 SDL 需要的格式
                
            - 将数据放入 `is->audio_buf`，并更新 `is->audio_clock`（音视频同步的基准！）
                
    - 把 `is->audio_buf` 里已有的数据用 `SDL_MixAudioFormat` 拷贝到 `stream`
        
    - 返回
        

整个过程没有信号槽，没有 UI 操作，一切为了“快”和“准时”。

## 五、一句话总结

**音频消费者** = `audio_callback`（调度）+ `audio_decode_frame`（解码填充）。  
它不发射信号，因为数据直接交给声卡；它存在的唯一目的，就是在 SDL 按铃时立刻交出足额、正确格式的音频数据。

## 一、`stream` 不是你定义的，是 SDL 传给你的

在回调函数的参数里：

c

void audio_callback(void *userdata, Uint8 *stream, int len)

- `stream` 的类型是 `Uint8*`，即 `unsigned char*`，是个**指针**。
    
- 这个指针是 **SDL 在调用你的回调函数时传进来的**，不是你创建的，也不是你分配的内存。
    
- 它指向一块 **SDL 内部的音频缓冲区**，这块缓冲区声卡能够直接访问。
    

所以：

- `stream` 是 SDL 交给你的一个**已经分配好的内存地址**。
    
- 你只需要把 PCM 数据拷贝到这个地址里，SDL 就能把它播放出来。
    

---

## 二、`stream` 就是 SDL 的音频缓冲区吗？关系是这样的

**SDL 的音频缓冲区** 是 SDL 自己管理的一块或多块内存，用于和声卡交换数据。  
`stream` 就是这块缓冲区的**当前可用写入地址**。

流程如下：

1. SDL 初始化时，根据你的 `wanted_spec`（如采样率、声道数、缓冲区样本数）在内部创建音频缓冲区。
    
2. 声卡播放音频时，会不断消耗缓冲区里的数据。
    
3. 当缓冲区里的数据快播完了，SDL 会在它的音频线程里调用你的回调函数，并传给你：
    
    - `stream`：缓冲区当前空闲部分的起始地址。
        
    - `len`：需要你填写的字节数。
        
4. 你必须在回调里把 `len` 字节的 PCM 数据写入 `stream`。
    
5. 回调返回后，SDL 会告诉声卡：“这块数据已经准备好了，你可以播了。”
    

**类比**：

- SDL 的音频缓冲区 = 一张白纸，声卡是打印机，会逐行打印（播放）纸上的内容。
    
- 当白纸快用完时，SDL 给你一支笔和一个空白的续页位置（`stream`），说：“请在这里写下 1024 个字（`len` 字节），我马上交给打印机。”
    
- 你写完后，SDL 就把续页接上，打印机无缝继续打。
    

---

## 三、为什么不直接操作 SDL 的缓冲区？

因为 SDL 已经把缓冲区的当前写入位置封装成了 `stream` 传给你，**你不需要（也不应该）直接去操作 SDL 的内部缓冲区数据结构**。原因有三：

1. **安全隔离**：SDL 管理着缓冲区的读写指针、环形缓冲区等细节，如果你自己乱写，可能会破坏 SDL 的内部状态。
    
2. **跨平台兼容**：不同操作系统和声卡驱动的缓冲区实现不同，SDL 通过回调模式统一了接口。
    
3. **简单清晰**：你只需要关注“把 `len` 字节数据拷贝到 `stream`”这一件事，不用管缓冲区有几个块、当前写到哪了。
    

所以，**`stream` 就是 SDL 开放给你的、可以安全写入的缓冲区入口**。  
你往 `stream` 里写数据，就等于往 SDL 的音频缓冲区里写数据，声卡随后就会播放它。

---

## 四、`stream` 是声卡吗？是播放声音的吗？

- `stream` **不是声卡**，它只是一块内存。声卡是硬件，`stream` 是内存地址。
    
- 但 `stream` **直接关联声音播放**：写入 `stream` 的数据会被声卡以设定的采样率数模转换，变成模拟电信号，推动耳机/音箱发声。
    
- 你在回调里不写 `stream`，声卡就会播之前残留的数据或全0，结果就是没声音或噪声。
    

---

## 五、完整串联：从你的代码到声音出来

1. 你调用 `SDL_OpenAudioDevice` 时传入了 `wanted_spec`，SDL 根据这些参数创建音频设备，并在后台启动音频线程。
    
2. 音频线程里，SDL 维护着一个循环，不断检查声卡的播放进度。
    
3. 每次需要新数据时，SDL 就调用 `audio_callback`，传给你：
    
    - `stream`：指向内部缓冲区的可用位置
        
    - `len`：需要你填充的字节数
        
4. 你的 `audio_callback` 从自己的 `is->audio_buf` 里取出数据，用 `SDL_MixAudioFormat` 拷贝到 `stream`。
    
5. 回调返回后，SDL 标记这块数据已就绪，声卡就会在适当时刻播放它。
    
6. 声音就出来了。
    

---

## 六、一句话总结

`stream` 是 SDL 为你准备的、与声卡直接相连的内存地址。你把音频数据拷进去，就是交给声卡播放。它不是声卡硬件本身，但它是声卡能直接读取的内存缓冲区的当前入口。

## 实战排错：点击"打开视频"后程序异常退出

### 先说结论：两个 bug，一个比一个隐蔽

这次遇到的问题是：**程序启动后，点击"打开视频"按钮，程序直接异常结束了**。

经过排查，发现是两个 bug 叠在一起导致的。下面我用最通俗的方式，一个一个讲清楚。

---

### Bug 1：stop() 函数里的死循环（这是主犯）

#### 用大白话理解这个问题

想象一个场景：

> 你去餐厅吃饭，服务员说"请稍等，我们先把上一桌的菜收完再给您点单"。
> 但是！上一桌根本没来过人（桌子一直是空的），所以"收完"这个条件**永远不会满足**。
> 于是服务员就一直在那等啊等，等啊等……整个餐厅都卡住了。

我们的代码就犯了完全一样的错误。

#### 代码是怎么写的

当你点击"打开视频"按钮时，代码会先调用 `stop(true)` 想把旧的播放器停下来：

```cpp
// playerdialog.cpp 第83行
m_player->stop(true);
```

然后 `stop()` 函数里面有这样一段：

```cpp
// videoplayer.cpp 第38-44行
if( isWait )
{
    while(!m_videoState.readThreadFinished)  // ← 问题就在这！
    {
        SDL_Delay(10);  // 等10毫秒再检查
    }
}
```

这段代码的意思是："只要读线程还没结束，我就一直等。"

#### 为什么会死循环？

关键在于 `readThreadFinished` 这个标志：

- `readThreadFinished` 的初始值是 `false`（还没结束）
- 只有当 `run()` 函数执行到最后，才会把它设为 `true`（已经结束了）

**但是！** 第一次点击"打开视频"时，播放器是刚创建的，`run()` 函数**根本还没开始执行**，所以 `readThreadFinished` 永远是 `false`。

```
第一次点击"打开视频"的执行流程：

1. m_player->stop(true)        ← 想停止播放器
2. 进入 while 循环
3. 检查 readThreadFinished     ← 是 false
4. SDL_Delay(10)               ← 等10毫秒
5. 检查 readThreadFinished     ← 还是 false（因为线程根本没跑过！）
6. SDL_Delay(10)               ← 继续等
7. 检查 readThreadFinished     ← 还是 false
   ... 无限循环 ...
   → 程序界面冻结，最终被系统杀掉
```

#### 怎么修？

很简单：**在等待之前，先看看线程到底有没有在运行**。如果线程根本没启动，那就不需要等。

```cpp
// 修改前：
if( isWait )
{
    while(!m_videoState.readThreadFinished)

// 修改后：
if( isWait && this->isRunning() )   // ← 加了这个条件
{
    while(!m_videoState.readThreadFinished)
```

`this->isRunning()` 是 Qt 的 QThread 类自带的方法，它会告诉我们这个线程**当前是否正在运行**。

- 线程在跑 → `isRunning()` 返回 true → 进入 while 循环等它结束 ✅
- 线程没跑 → `isRunning()` 返回 false → 跳过 while 循环 ✅

---

### Bug 2：空指针导致的崩溃（这是从犯）

#### 用大白话理解"空指针"

> 你拿着一张纸条去找人，纸条上写着"去3号楼201室找张三"。
> 但如果纸条上什么都没写（地址是空的），你按照这个空地址去找人，结果就是——你不知道去哪，迷路了，甚至可能闯进了不该去的地方。

在编程里，"空指针"就是那个**什么都没写的纸条**。指针变量本应该存一个内存地址，但如果它是 `NULL`（空），就意味着"没有指向任何有效的内存"。你强行去读它指向的内容，程序就会崩溃。

#### 代码里的问题

我们的视频线程 `video_thread` 里有两处代码，直接使用了 `audioq`（音频队列）：

**位置1** — 判断视频是否播放结束：
```cpp
// videoplayer.cpp 第421行
if( is->readFinished && is->audioq->nb_packets == 0)
//                              ^^^^^^^
//                              如果 audioq 是空的(NULL)，这里就崩了！
```

**位置2** — 音视频同步等待：
```cpp
// videoplayer.cpp 第473行
if( is->audioq->size == 0 ) break;
//  ^^^^^^^^
//  同样的问题，audioq 可能是 NULL
```

#### 为什么 audioq 会是 NULL？

不是所有视频文件都有音频！比如：

- 无声视频（监控录像、延时摄影等）
- GIF 动画转的 MP4
- 某些屏幕录制文件

当视频文件**没有音频流**时，代码里的这段就不会执行：

```cpp
// videoplayer.cpp 第177行
if( audioStream != -1 )    // ← 音频流不存在，这个条件为 false
{
    // 这里面的代码不会执行
    m_videoState.audioq = new PacketQueue;   // ← 所以 audioq 不会被创建
    ...
}
```

结果就是 `audioq` 一直是 `NULL`，后面视频线程一访问它就崩了。

#### 怎么修？

**位置1** — 在访问之前先检查 audioq 是不是 NULL：

```cpp
// 修改前：
if( is->readFinished && is->audioq->nb_packets == 0)

// 修改后：
if( is->readFinished && (is->audioq == NULL || is->audioq->nb_packets == 0))
//                       ^^^^^^^^^^^^^^^^^^^^
//                       先检查 audioq 是否存在，存在才看里面有没有包
```

逻辑解读：
- `audioq == NULL` → 根本没有音频队列，说明没有音频流，视频播完就可以结束 ✅
- `audioq->nb_packets == 0` → 有音频队列但里面没包了，也说明播完了 ✅

**位置2** — 用 if 把整个音视频同步逻辑包起来：

```cpp
// 修改前：
double audio_pts = is->audio_clock;
while(1)
{
    if( is->quit ) break;
    if( is->audioq->size == 0 ) break;    // ← 崩溃点
    ...
}

// 修改后：
if (is->audioq != NULL) {                  // ← 有音频才做同步
    double audio_pts = is->audio_clock;
    while(1)
    {
        if( is->quit ) break;
        if( is->audioq->size == 0 ) break; // ← 安全了，因为外面已经确认不为NULL
        ...
    }
}
// 没有音频的话，直接跳过同步，视频帧按自己的节奏显示
```

---

### 两个 bug 的关系

```
点击"打开视频"
    │
    ▼
调用 stop(true)
    │
    ├── Bug 1: 线程没启动，while死循环 → 程序卡死/崩溃
    │         （这是你看到"程序异常结束"的直接原因）
    │
    └── 如果 Bug 1 没发生（比如第二次打开）
         │
         ▼
      启动新线程播放视频
         │
         ├── 如果视频有音频 → 正常播放 ✅
         │
         └── 如果视频没有音频 → Bug 2: 空指针崩溃 ❌
```

Bug 1 是**每次第一次打开都会触发**的，所以你看到的现象就是"点击打开视频后程序异常结束"。

Bug 2 是**潜在隐患**，只有播放无音频的视频时才会触发。但修了 Bug 1 之后，如果你恰好打开了一个没有音频的视频，就会碰到 Bug 2。所以两个都要修。

---

### 初学者避坑指南：从这两个 bug 中学到什么

#### 1. 永远不要假设"之前的步骤一定执行了"

Bug 1 的教训：`stop()` 假设了线程一定在运行，但第一次调用时线程根本没启动。

**写代码时多问自己**：如果这个函数被调用时，它依赖的前置条件还没满足，会怎样？

#### 2. 永远不要假设"指针一定不是空的"

Bug 2 的教训：代码假设 `audioq` 一定有值，但有些视频文件根本没有音频流。

**写代码时的好习惯**：使用指针之前，先检查它是不是 NULL：

```cpp
// 好习惯 ✅
if (ptr != NULL) {
    ptr->doSomething();
}

// 危险 ❌
ptr->doSomething();   // 如果 ptr 是 NULL，直接崩溃
```

#### 3. while 循环一定要有"退出保障"

Bug 1 的 `while(!readThreadFinished)` 看似没问题——线程跑完了自然就退出了。但它没考虑到"线程根本没跑"的情况。

**写代码时的好习惯**：while 循环的退出条件，要覆盖所有可能的场景，不能只考虑"正常流程"。
