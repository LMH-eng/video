# 06 推荐算法（协同过滤 + 热度排序）

## 模块概述

推荐系统基于**8维兴趣向量 + 余弦相似度协同过滤 + 热度排序**，实现个性化视频推荐。用户兴趣向量在Redis中存储和增量更新，注册时初始化→观看/点赞时增量→登录时衰减，形成"冷启动→热收敛"的闭环。

## 技术选型与理由

> **面试官追问**：为什么不直接用热门排序？为什么不用深度学习推荐？

| 方案 | 优势 | 劣势 | 选择理由 |
|------|------|------|----------|
| **协同过滤+热度混合** | 可解释性强、实现简单、冷启动有兜底 | 8维向量维度低、精度有限 | **面试项目需要可解释+可实现的推荐**，深度学习需要大量训练数据和GPU资源 |
| 纯热门排序 | 最简单、无需用户画像 | 所有用户看到相同内容，无个性化 | 无个性化=无粘性，用户很快流失 |
| 深度学习推荐 | 精度高、自动特征工程 | 需大量训练数据+GPU+复杂部署 | 项目数据量小（百级用户），深度学习无法有效训练 |

> **面试官追问**：8维向量够用吗？抖音用多少维？
> - 8维是注册UI的8个复选框决定的，是**用户显式偏好**
> - 抖音用数百维隐式特征（观看时长、滑动速度、重复观看、分享行为等）
> - 8维是起点，如果要做更精准推荐，需要增加隐式行为特征维度

## 核心实现流程

### 注册时：初始化兴趣向量到Redis

```cpp
void CLogic::initUserHobbyToRedis(int userId, char* hobby) {
    // 将8维Hobby结构体转为double数组存入Redis
    // hobby[0]=food: 如果选中=1.0，未选中=0.0
    double vec[DEF_HOBBY_COUNT];
    for(int i = 0; i < DEF_HOBBY_COUNT; i++) {
        vec[i] = (hobby[i] != 0) ? 1.0 : 0.0;
    }
    
    // Redis存储：key="user:hobby:{userId}", value=8个double
    char key[64];
    sprintf(key, "user:hobby:%d", userId);
    m_redis->setVector(key, vec, DEF_HOBBY_COUNT);
}
```

### 观看/点赞时：增量更新兴趣向量

```cpp
void CLogic::updateUserHobby(int userId, char* hobby, int delta) {
    // delta=+1（点赞/观看）或-1（取消点赞）
    double vec[DEF_HOBBY_COUNT];
    char key[64];
    sprintf(key, "user:hobby:%d", userId);
    m_redis->getVector(key, vec, DEF_HOBBY_COUNT);
    
    // 增量更新：对应分类维度 += delta * 0.3
    for(int i = 0; i < DEF_HOBBY_COUNT; i++) {
        if(hobby[i] != 0) {
            vec[i] += delta * 0.3;  // 0.3是增量系数，防止单次行为过度影响
        }
    }
    
    m_redis->setVector(key, vec, DEF_HOBBY_COUNT);
}
```

> **面试官追问**：增量系数0.3怎么确定的？
> - 如果系数=1.0，一次点赞就能让某维度翻倍，太敏感
> - 如果系数=0.1，需要10次点赞才能显著变化，太迟钝
> - 0.3是经验值：3~4次相同分类的行为就能让兴趣向量明显偏向该分类
> - **生产级需要A/B测试确定最优系数**

### 登录时：衰减注册兴趣

```cpp
void CLogic::decayUserHobby(int userId) {
    double vec[DEF_HOBBY_COUNT];
    char key[64];
    sprintf(key, "user:hobby:%d", userId);
    m_redis->getVector(key, vec, DEF_HOBBY_COUNT);
    
    // ★ 衰减公式：new = old * 0.8
    // 注册时的兴趣权重较高（初始值1.0），每次登录衰减20%
    // 实际观看行为会增量更新，衰减+增量让向量收敛到真实偏好
    for(int i = 0; i < DEF_HOBBY_COUNT; i++) {
        vec[i] *= 0.8;
    }
    
    m_redis->setVector(key, vec, DEF_HOBBY_COUNT);
}
```

> **面试官追问**：为什么衰减0.8而不是0.5？
> - 0.5衰减太快：3次登录后初始兴趣只剩0.125，如果用户还没产生足够行为数据，推荐会失去方向
> - 0.8衰减温和：5次登录后初始兴趣剩0.33，同时增量更新已经积累了足够的真实偏好
> - **0.8是"温和收敛"策略**，0.5是"激进遗忘"策略

### 余弦相似度计算

```cpp
double CLogic::cosineSimilarity(double* vecA, double* vecB, int n) {
    double dotProduct = 0.0;   // 点积
    double normA = 0.0;        // 向量A的模
    double normB = 0.0;        // 向量B的模
    
    for(int i = 0; i < n; i++) {
        dotProduct += vecA[i] * vecB[i];
        normA += vecA[i] * vecA[i];
        normB += vecB[i] * vecB[i];
    }
    
    if(normA == 0 || normB == 0) return 0.0;  // 防止除零
    
    return dotProduct / (sqrt(normA) * sqrt(normB));
    // 返回值范围[-1, 1]，1=完全相似，0=无关联，-1=完全相反
}
```

> **面试官追问**：余弦相似度和欧氏距离的区别？
> - **余弦相似度**：衡量方向相似性，不考虑向量大小（用户A兴趣强度=3.0，用户B=1.0，但偏好方向相同→相似度高）
> - **欧氏距离**：衡量绝对距离，考虑向量大小（强度差异大的用户被认为不相似）
> - **推荐场景选余弦**：因为兴趣强度不重要（新用户和老用户可能偏好相同），偏好方向才重要

### 协同过滤：找相似用户


> **面试官追问**：如果流量增加10倍，协同过滤哪里会先出问题？
> - **计算量**：10万用户→每个推荐请求需要计算10万次余弦相似度→O(10万×8)→约0.8M次乘法→耗时约800ms→不可接受
> - **解决方案**：
>   1. 预计算：定期（每小时）计算所有用户相似度矩阵，存Redis，推荐时直接查→O(1)
>   2. 近似计算：LSH（局部敏感哈希）快速找相似用户，O(1)近似
>   3. 分簇：将用户分为K个簇，只在同簇内计算相似度→O(K×平均簇大小)

## 遇到的难点

### 难点1：冷启动问题

- **问题**：新注册用户只有8维初始兴趣（0/1二值），没有观看/点赞行为数据，推荐精度极低
- **解决**：**混合策略**——协同过滤结果不足时，用热度排序兜底
  - CF推荐5个视频 + 热度推荐10个视频 = 15个
  - 随着用户观看/点赞行为积累，CF比例逐渐增大

> **面试官追问**：抖音怎么解决冷启动？
> - 抖音用"探索策略"：新用户先推送各类热门视频，观察观看时长/完播率
> - 几次交互后就能建立初步兴趣画像，然后逐步从热门→个性化过渡
> - 本项目的衰减+增量机制也是类似的思路

### 难点2：Redis挂了怎么办？

- **问题**：Redis连接失败时，兴趣向量无法读取，推荐功能完全瘫痪
- **解决**：**降级策略**
  ```cpp
  if(!m_redisConnected) {
      cout << "Redis连接失败，推荐功能降级到MySQL" << endl;
      // 降级：直接用热度排序，不做协同过滤
      hotVideos = getRecommendByHot(15);
  }
  ```
- **代价**：降级后推荐无个性化，所有用户看到相同热门视频

> **面试官追问**：Redis和MySQL的一致性怎么保证？
> - 兴趣向量只在Redis中存储，MySQL只存初始Hobby（注册时）
> - 如果Redis重启，兴趣向量丢失→从MySQL读初始Hobby重建→但增量更新数据丢失
> - **生产级方案**：定期将Redis兴趣向量同步到MySQL（每小时一次），Redis重启后从MySQL恢复

## 模拟面试题

### Q1：你的推荐算法和抖音的差距在哪？
**回答要点**：
1. **特征维度**：8维显式兴趣 vs 数百维隐式特征（观看时长、滑动速度、重复观看率等）
2. **算法模型**：余弦相似度CF vs 深度学习（Wide&Deep、DIN等）
3. **实时性**：每次请求实时计算 vs 预计算+实时特征更新
4. **但核心思想一致**：用户画像→相似度计算→个性化排序→冷启动兜底

### Q2：如果用户兴趣随时间变化（比如从喜欢搞笑变成喜欢教育），你的算法能跟上吗？
**回答要点**：
- **能跟上**，因为增量更新机制：每次观看/点赞教育类视频，教育维度+0.3
- 衰减机制也会让旧的搞笑兴趣逐渐减弱（每次登录衰减20%）
- **但收敛速度慢**：需要约5~10次教育类行为才能让教育维度超过搞笑维度
- **优化方案**：增加时间衰减（最近的兴趣权重更高），或用指数移动平均（EMA）

### Q3：余弦相似度在什么情况下会失效？
**回答要点**：
- **全零向量**：用户没有选择任何兴趣标签→余弦相似度分母为0→返回0→无法计算相似度→降级到热度排序
- **维度灾难**：8维不会遇到，但如果是1000维，大部分维度为0→稀疏向量→余弦相似度不稳定→需要降维（PCA/矩阵分解）
- **反直觉情况**：用户A只喜欢美食(3.0)，用户B只喜欢教育(3.0)，余弦相似度=0→完全不相似→但两人可能都喜欢"美食教育"类视频→需要多标签交叉推荐
