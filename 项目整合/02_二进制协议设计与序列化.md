# 02 二进制协议设计与序列化

## 模块概述

`packdef.h` 是全项目的"宪法级"文件，定义了所有常量、28个协议号、状态码、传输结构体。采用**C结构体 + `#pragma pack(push,1)` 紧凑字节对齐**，实现零拷贝直接网络传输，无JSON/XML解析开销。

## 技术选型与理由

> **面试官追问**：为什么不直接用JSON？Protobuf呢？

| 方案 | 优势 | 劣势 | 选择理由 |
|------|------|------|----------|
| **二进制结构体** | 零解析开销、固定长度易粘包处理、带宽极省 | 不可读、协议升级需版本管理 | 服务端纯C++，无跨语言需求；高频小包（点赞仅userId+videoId），JSON头部比数据大 |
| JSON | 可读、灵活、跨语言 | 解析开销、字段长度不固定→粘包处理复杂 | C++服务端JSON解析每次需分配内存，高并发GC压力大 |
| Protobuf | 高效、跨语言、有版本管理 | 需.proto+编译链、引入第三方依赖 | 28个协议规模不大，Protobuf编译链投入产出比不高 |

**核心决策逻辑**：协议设计优先考虑**服务端性能**——epoll+线程池每秒处理万级请求，JSON的序列化/反序列化开销在高并发场景不可忽视。

## 核心实现流程

### 协议号设计：偏移数组 → O(1)路由

```cpp
// 协议号从10000开始，连续编号
#define _DEF_PACK_BASE      (10000)
#define _DEF_PACK_REGISTER_RQ  (_DEF_PACK_BASE + 0)
#define _DEF_PACK_REGISTER_RS  (_DEF_PACK_BASE + 1)
#define _DEF_PACK_LOGIN_RQ     (_DEF_PACK_BASE + 2)
// ... 共28个协议号，RQ/RS配对

// 服务端路由：type - _DEF_PACK_BASE = 数组索引，直接跳转函数指针
void CLogic::setNetPackMap() {
    m_NetPackMap[_DEF_PACK_REGISTER_RQ - _DEF_PACK_BASE] = &CLogic::RegisterRq;
    m_NetPackMap[_DEF_PACK_LOGIN_RQ    - _DEF_PACK_BASE] = &CLogic::LoginRq;
    // O(1)路由，无需switch-case遍历
}
```

> **面试官追问**：为什么从10000开始而不是0？
> - 防止与系统协议号冲突（0~9999是系统保留区）
> - 偏移量直接做数组索引，10000有明确的"自定义协议"语义

### 紧凑字节对齐

```cpp
#pragma pack(push, 1)  // 1字节对齐，消除内存空洞
typedef struct STRU_UPLOAD_RQ {
    PackType m_nType;          // 4字节
    int m_UserId;              // 4字节
    int m_nFileId;             // 4字节
    int64_t m_nFileSize;       // 8字节
    char m_szHobby[8];         // 8字节兴趣向量
    char m_szFileName[260];    // 视频文件名
    char m_szGifName[260];     // GIF文件名
    char m_szFileType[40];     // 文件后缀
    char m_szFileHash[33];     // MD5哈希
    char m_szTitle[100];       // 视频标题
} STRU_UPLOAD_RQ;
#pragma pack(pop)  // 恢复默认对齐
```

> **面试官追问**：如果不加`#pragma pack(push,1)`会怎样？
> - 默认8字节对齐，`int64_t`前会插入4字节padding，结构体大小变大
> - **更严重的问题**：客户端和服务端如果编译器/平台不同，对齐规则不同，同一结构体大小不一致 → `sizeof`不匹配 → 解析错位 → 数据错乱甚至崩溃
> - 1字节对齐保证**跨平台、跨编译器**结构体大小一致

### RQ/RS配对模式

```cpp
// 每个请求都有对应的响应，协议号相邻（RQ奇数/RS偶数）
#define _DEF_PACK_REGISTER_RQ  (10000)  // 客户端→服务端
#define _DEF_PACK_REGISTER_RS  (10001)  // 服务端→客户端
#define _DEF_PACK_LOGIN_RQ     (10002)
#define _DEF_PACK_LOGIN_RS     (10003)
// ... 14对RQ/RS
```

> **面试官追问**：为什么不用一个协议号+字段区分请求/响应？
> - 分离协议号可以让路由表直接索引，不需要额外判断字段
> - 请求和响应的处理函数完全不同（RegisterRq写数据库 vs RegisterRs弹窗提示），分开更清晰

### Hobby 8维兴趣向量

```cpp
typedef struct Hobby {
    char food;     // 美食
    char funny;    // 搞笑
    char ennergy;  // 正能量
    char dance;    // 舞蹈
    char music;    // 歌曲
    char video;    // 影视
    char outside;  // 户外
    char edu;      // 教育
} Hobby;
```

**8字节贯穿全链路**：注册时用户选择 → 上传时视频标注 → 推荐时用户兴趣向量匹配 → Redis存储增量更新。这是推荐算法的基石。

> **面试官追问**：为什么用char而不是float表示兴趣程度？
> - 注册时用户只能"选/不选"，是0/1的二值逻辑，char足够
> - 运行时在Redis中转为double向量做余弦相似度计算，char是存储格式，double是计算格式
> - 如果要支持兴趣程度（很感兴趣/一般/不感兴趣），可以改用int权重，但注册UI复杂度增加

## 遇到的难点

### 难点1：协议结构体大小不一致导致解析错位

- **问题**：早期开发时，客户端Windows MSVC和服务端Linux GCC对同一结构体的`sizeof`不同（MSVC默认8字节对齐，GCC也是但padding位置可能不同）
- **发现**：服务端收到注册包后，解析出的tel字段是乱码
- **排查**：在两端分别打印`sizeof(STRU_REGISTER_RQ)`，发现差了4字节
- **解决**：全部加上`#pragma pack(push,1)`，保证1字节对齐，两端大小一致

### 难点2：协议升级时向前兼容

- **面试官追问**：如果你要给STRU_LOGIN_RQ加一个"设备类型"字段，老客户端怎么办？

> **回答要点**：
> 1. **方案A**：新协议号（_DEF_PACK_LOGIN_V2_RQ = 10050），老客户端继续用10002，服务端根据协议号选择不同处理函数 → **最安全，但协议号膨胀**
> 2. **方案B**：在结构体末尾追加字段，老客户端的包size较小，服务端根据`nlen`判断版本 → **简单，但需要每个处理函数加版本判断**
> 3. **本项目选择**：28个协议号足够，如果需要升级用方案A

## 性能优化

- **零拷贝传输**：`SendData(fd, (char*)&rq, sizeof(rq))`，结构体直接内存映射到网络缓冲区，无序列化/反序列化步骤
- **固定长度**：所有结构体字段都是固定大小数组（`char tel[40]`而非`string`），`sizeof`即可确定包长度，粘包处理极简
- **O(1)路由**：协议号偏移 → 函数指针数组直接索引，无switch-case遍历

> **面试官追问**：零拷贝有什么隐患？
> - 结构体中不能有指针（指针地址在另一进程无效），所有字符串必须是固定大小数组
> - 不能有虚函数表（vtable指针跨进程无效）
> - 本项目严格遵守这两条规则

## 模拟面试题

### Q1：你的二进制协议和Protobuf比，性能差多少？
**回答要点**：Protobuf用varint编码压缩整数，小整数更省带宽；但本项目协议包大部分是固定大小字符串（tel[40]、name[40]），varint对字符串没有压缩效果。实测28个协议中，只有LIKE_RQ（4+4+4+1=13字节）能被Protobuf压缩，其他包Protobuf和二进制结构体大小差不多。**性能差距主要在解析开销**——Protobuf需要解码每字段的tag+length+value，二进制结构体直接memcpy。

### Q2：如果服务端收到一个非法协议号（不在10000~10027范围），你怎么处理？
**回答要点**：路由数组大小是`_DEF_PACK_COUNT(100)`，协议号偏移超出范围时函数指针为NULL，调用前必须检查。当前代码在`Buffer_Deal`中先检查`m_NetPackMap[index] != NULL`才调用，非法协议号直接丢弃包+打印警告日志。**这是防御性编程的基本要求**。

### Q3：你的Hobby结构体只有8维，如果要做100维的兴趣向量（类似抖音），协议怎么改？
**回答要点**：8维是注册UI的8个复选框决定的，实际推荐算法在Redis中用double数组存储。如果要100维：
1. 注册UI改为"选择3~5个最感兴趣的标签"
2. 协议中Hobby改为`char hobby[100]`或用位图`uint64_t hobby[2]`（128位够用）
3. 推荐算法从余弦相似度改为矩阵分解（MF）或深度学习嵌入向量
4. **但100维的余弦相似度计算量是O(100n)，10万用户需要优化**——Redis预计算相似用户列表，定期更新
