# 01 项目总览与架构

## 模块概述

短视频平台是一个类抖音的短视频分享系统，包含**客户端（Qt/C++）+ 服务端（Linux C++）+ 流媒体服务器（nginx-rtmp）**三层架构，实现了从注册登录、视频上传、推荐浏览、RTMP播放、点赞评论的完整闭环。

## 技术栈全景

| 层级 | 技术 | 作用 |
|------|------|------|
| **客户端** | Qt 5 + FFmpeg 4.2.2 + SDL 2.0.10 | UI框架 + 音视频解码 + 音频输出 |
| **服务端** | Linux epoll + 线程池 + MySQL + Redis | 高并发网络 + 业务逻辑 + 数据存储 + 缓存 |
| **流媒体** | nginx-rtmp-module + ffmpeg | 视频转码 + RTMP流分发 |
| **通信协议** | 自定义二进制结构体（`#pragma pack(push,1)`） | 零拷贝网络传输 |

## 核心数据流：从用户点击到视频播放

```
用户点击缩略图
  → MainDialog.SIG_PlayRq(videoId, userId)
  → CKernel.slot_playRq → TCP发送STRU_PLAY_RQ
  → 服务端CLogic.PlayRq → MySQL查视频信息 → 返回RTMP URL + 点赞状态
  → CKernel.slot_dealClientData → PlayerDialog.playUrl(rtmpUrl)
  → VideoPlayer.run() → avformat_open_input(rtmpUrl)
  → 读线程av_read_frame → PacketQueue入队
  → 视频线程解码+同步等待 → QImage信号 → UI显示
  → 音频回调SDL → 扬声器输出
```

## 技术选型与理由

### 为什么选 Qt 而不是 Electron/Web？

> **面试官追问**：现在短视频平台基本都是Web/React Native，你为什么用Qt？

| 方案 | 优势 | 劣势 | 选择理由 |
|------|------|------|----------|
| **Qt/C++** | 直接调用FFmpeg C API、零中间层损耗、内存可控 | UI开发效率低于Web、跨平台部署复杂 | **音视频处理需要极致性能**，Qt信号槽天然适合多线程事件分发，C++直接操作FFmpeg/SDL无语言边界损耗 |
| Electron | UI开发快、生态丰富 | Node.js调FFmpeg需child_process、内存占用高、延迟大 | 音视频实时性要求高，Electron的IPC开销不可接受 |
| React Native | 移动端友好 | 调原生FFmpeg需JNI桥接、调试链路长 | 项目定位是桌面端学习项目，非移动端 |

> **如果流量增加10倍**：Qt客户端本身不是瓶颈，瓶颈在服务端epoll单进程。解决方案：多进程部署 + nginx负载均衡。

### 为什么选 epoll 而不是 libevent/Boost.Asio？

> **面试官追问**：为什么不直接用现成的高性能网络库？

| 方案 | 优势 | 劣势 | 选择理由 |
|------|------|------|----------|
| **epoll原生** | 完全可控、无第三方依赖、学习底层原理 | 需自己处理粘包/线程安全/连接管理 | **面试项目核心目的是展示对Linux网络编程的深度理解**，用原生epoll能证明你真正懂内核级I/O多路复用 |
| libevent | 成熟稳定、跨平台 | 抽象层厚、回调链复杂、难以定制 | 面试官想看你能不能从零写出一个可用的网络框架 |
| Boost.Asio | C++标准提案、proactor模式 | 编译慢、依赖重、学习曲线陡 | 项目是C+少量C++风格，Boost太重 |

### 为什么选 nginx-rtmp 而不是自建流媒体？

> **面试官追问**：为什么不自己写RTMP服务？

| 方案 | 优势 | 选择理由 |
|------|------|----------|
| **nginx-rtmp-module** | 成熟稳定、ffmpeg自动转码、运维简单 | 项目重点是客户端和服务端逻辑，流媒体是基础设施，没必要重复造轮子 |
| 自建RTMP服务 | 完全可控 | RTMP协议实现复杂（握手+chunk+消息），投入产出比极低 |

### 为什么选二进制协议而不是JSON？

> **面试官追问**：现在RESTful+JSON是主流，你为什么用二进制？

| 方案 | 优势 | 劣势 | 选择理由 |
|------|------|------|----------|
| **二进制结构体** | 零解析开销、固定长度易粘包处理、带宽极省 | 不可读、协议升级需版本管理 | **视频平台高频小包（点赞/评论/推荐），JSON解析开销在C++服务端不可忽视**；结构体直接内存映射，send/recv零拷贝 |
| JSON | 可读、灵活、跨语言 | 解析开销（序列化/反序列化）、字段长度不固定→粘包处理复杂 | 服务端是纯C++，没有跨语言需求 |

## 遇到的难点

### 1. 倍速播放电音问题（详见09章节）
- **问题**：1.5x/2.0x时出现正常声音叠加电音
- **排查**：从SDL_MixAudioFormat混音叠加 → rubberband滤镜相位问题 → aformat格式转换
- **解决**：memcpy替换MixAudioFormat + 去掉rubberband + aformat强制S16输出

### 2. 上传乱序Bug（详见05/11章节）
- **问题**：UploadRq和FileBlockRq乱序导致服务端找不到上传任务
- **排查**：异步投递线程池时，两个包可能被不同线程处理
- **解决**：改为同步处理（Buffer_Deal直接调用回调，不再投递新任务）

### 3. 音视频同步（详见08章节）
- **问题**：视频快于音频或反之
- **排查**：audio_clock和video_clock必须用原始PTS在同一时间域比较
- **解决**：音频为主时钟，视频线程同步等待audio_clock

## 模拟面试题

### Q1：你这个项目最大的技术挑战是什么？
**回答要点**：倍速播放的电音问题。排查过程体现了系统性调试能力——从SDL回调层（MixAudioFormat混音叠加）→ FFmpeg滤镜层（rubberband相位声码器失真）→ 数据格式层（FLT/S16格式不匹配），三层逐级排查，最终定位到三个独立原因叠加导致问题。修复后0.5x~4.0x全程无电音。

### Q2：如果用户量从100增长到10万，你的架构哪里会先出问题？
**回答要点**：
1. **epoll单进程**：单进程epoll+线程池（最大200线程）在10万并发时CPU和内存会成为瓶颈 → 解决：多进程部署，每个进程绑定CPU核心，nginx负载均衡分发
2. **MySQL单实例**：10万用户的推荐查询（协同过滤cosine计算）会很慢 → 解决：读写分离 + Redis缓存热门推荐结果
3. **RTMP流带宽**：10万用户同时播放，nginx-rtmp单机带宽不够 → 解决：CDN分发 + HLS多码率自适应

### Q3：你的项目和抖音/B站的真实架构差距在哪？
**回答要点**：
1. **推荐算法**：我用8维兴趣向量+余弦相似度，抖音用深度学习+实时特征工程，差距巨大
2. **流媒体**：我用RTMP单码率，B站用HLS+ABR多码率自适应+P2P加速
3. **存储**：我用单MySQL，生产级需要分库分表+对象存储（OSS/S3）
4. **但核心网络框架（epoll+线程池+二进制协议）的设计思路和生产级一致**
