# 07 客户端核心调度器（CKernel单例）

## 模块概述

`ckernel.h/cpp` 是客户端的**中枢调度器**，采用单例模式，负责：加载配置、创建TCP连接、管理所有UI对话框、将UI信号转发为TCP网络请求、将网络响应分发到对应UI更新。它是Qt信号槽跨模块通信的"总线"。

## 技术选型与理由

> **面试官追问**：为什么用单例而不是依赖注入？为什么用Qt信号槽而不是直接函数调用？

| 方案 | 优势 | 劣势 | 选择理由 |
|------|------|------|----------|
| **单例+信号槽** | 全局唯一入口、UI和网络完全解耦、跨线程安全 | 单例无法多实例、信号槽有轻微开销 | Qt信号槽天然支持跨线程通信（VideoPlayer在子线程emit→UI主线程slot），这是**最关键的选型理由** |
| 全局变量+函数调用 | 简单直接 | UI和网络紧耦合、跨线程不安全 | VideoPlayer在QThread中运行，直接调用UI方法会跨线程崩溃 |
| 依赖注入 | 更灵活、可测试 | Qt项目没有DI框架、增加复杂度 | 项目规模不大（4个Dialog+1个Player），单例足够 |

> **面试官追问**：信号槽和回调函数的优劣？
> - **信号槽**：支持多接收者（一个信号可连接多个槽）、跨线程安全（Qt自动队列化）、编译期类型检查
> - **回调函数**：更快（无Qt元对象系统开销）、C风格、但不支持多接收者、跨线程需自己加锁
> - 本项目选信号槽：**VideoPlayer子线程→UI主线程的跨线程通信是刚需**，回调函数无法安全跨线程

## 核心实现流程

### 单例模式

```cpp
class CKernel : public QObject {
    Q_OBJECT
private:
    explicit CKernel(QObject *parent = nullptr);       // 构造私有
    explicit CKernel(const CKernel &kernel){}           // 拷贝构造私有
    ~CKernel(){}                                         // 析构私有
public:
    static CKernel* GetInstance() {
        static CKernel kernel;  // C++11保证线程安全的静态局部变量
        return &kernel;
    }
};
```

> **面试官追问**：为什么用静态局部变量而不是new？
> - C++11标准保证静态局部变量的初始化是线程安全的（编译器自动加锁）
> - 不需要手动delete，程序结束时自动析构
> - 比`new CKernel()`更安全（不会忘记释放）

### 初始化流程：配置→TCP→UI→信号连接

```cpp
CKernel::CKernel(QObject *parent) : QObject(parent), m_userid(0) {
    // 1. 加载config.ini（IP和端口）
    loadIniFile();
    
    // 2. 创建TCP客户端
    m_tcpClient = new TcpClientMediator;
    connect(m_tcpClient, SIGNAL(SIG_ReadyData(uint,char*,int)),
            this, SLOT(slot_dealClientData(uint,char*,int)));
    m_tcpClient->OpenNet(m_ip.toUtf8().data());
    
    // 3. 创建PlayerDialog（主界面，最先显示）
    m_playerDialog = new PlayerDialog;
    connect(m_playerDialog, SIGNAL(SIG_openLoginDialog()),
            this, SLOT(slot_openLoginDialog()));
    connect(m_playerDialog, SIGNAL(SIG_playRtmpUrl(QString)),
            this, SLOT(slot_playRtmpUrl(QString)));
    connect(m_playerDialog, SIGNAL(SIG_LikeRq(int)),
            this, SLOT(slot_likeRq(int)));
    m_playerDialog->show();
}
```

### 配置文件加载

```cpp
void CKernel::loadIniFile() {
    m_ip = "192.168.18.129";  // 默认值
    m_port = "8000";
    
    QString path = QCoreApplication::applicationDirPath() + "/config.ini";
    QFileInfo info(path);
    if(info.exists()) {
        QSettings setting(path, QSettings::IniFormat);
        setting.beginGroup("net");
        QVariant strIP = setting.value("ip", "");
        QVariant strPort = setting.value("port", "");
        if(!strIP.toString().isEmpty()) m_ip = strIP.toString();
        if(!strPort.toString().isEmpty()) m_port = strPort.toString();
        setting.endGroup();
    } else {
        // 不存在则创建默认配置
        QSettings setting(path, QSettings::IniFormat);
        setting.beginGroup("net");
        setting.setValue("ip", m_ip);
        setting.setValue("port", m_port);
        setting.endGroup();
    }
}
```

> **面试官追问**：为什么用INI而不是JSON配置？
> - QSettings原生支持INI格式，读写一行代码搞定
> - JSON需要额外库（QJsonDocument），INI更简单
> - 配置只有ip和port两个字段，INI足够

### UI信号 → CKernel槽 → TCP发送

```cpp
// 登录信号转发
void CKernel::slot_loginRq(QString tel, QString password) {
    STRU_LOGIN_RQ rq;
    strcpy(rq.tel, tel.toUtf8().data());
    strcpy(rq.password, password.toUtf8().data());
    m_tcpClient->SendData(0, (char*)&rq, sizeof(rq));  // 结构体直接发送
}

// 推荐信号转发
void CKernel::slot_recommendRq(int userId, int page) {
    STRU_RECOMMEND_RQ rq;
    rq.m_UserId = userId;
    rq.m_nPage = page;
    m_tcpClient->SendData(0, (char*)&rq, sizeof(rq));
}
```

### 网络响应 → switch分发 → UI更新

```cpp
void CKernel::slot_dealClientData(unsigned int lSendIP, char *buf, int nlen) {
    int type = *(int*)buf;  // 取协议号
    
    switch(type) {
    case _DEF_PACK_LOGIN_RS: {
        STRU_LOGIN_RS* rs = (STRU_LOGIN_RS*)buf;
        if(rs->result == login_success) {
            m_userid = rs->userid;
            m_loginDialog->hide();
            m_mainDialog = new MainDialog;
            // 连接MainDialog的所有信号...
            m_mainDialog->setUserId(m_userid);
            m_mainDialog->show();
            slot_recommendRq(m_userid, 0);  // 登录后自动刷新推荐
        }
        break;
    }
    case _DEF_PACK_RECOMMEND_RS: {
        STRU_RECOMMEND_RS* rs = (STRU_RECOMMEND_RS*)buf;
        m_mainDialog->slot_dealRecommendRs(rs);
        break;
    }
    case _DEF_PACK_PLAY_RS: {
        STRU_PLAY_RS* rs = (STRU_PLAY_RS*)buf;
        if(rs->result == play_success) {
            m_playerDialog->playUrl(QString::fromUtf8(rs->url));
            m_playerDialog->setCurrentVideo(rs->videoId, rs->likeCount, rs->isLiked, ...);
        }
        break;
    }
    // ... 14个协议响应处理
    }
    delete[] buf;  // 回收内存
}
```

> **面试官追问**：为什么用switch而不是函数指针映射（和服务端不同）？
> - 客户端只有1个线程处理网络响应（Qt事件循环），不需要O(1)路由的极致性能
> - switch-case更直观，14个case可读性尚可
> - 如果协议数超过50个，客户端也应改为函数指针映射

## 遇到的难点

### 难点1：VideoPlayer子线程→UI主线程的跨线程通信

- **问题**：VideoPlayer在QThread中运行，直接调用UI方法（如`m_playerDialog->updateImage()`）会跨线程崩溃
- **发现**：运行时Qt警告"QImage cannot be used outside the GUI thread"
- **解决**：VideoPlayer通过Qt信号`SIG_getOneimage(QImage)`发送图像→PlayerDialog的`slot_setImage(QImage)`在主线程接收→安全更新UI
  ```cpp
  // VideoPlayer子线程中：
  emit is->m_player->SIG_getOneimage(tmpImg.copy());  // emit信号
  
  // PlayerDialog主线程中：
  connect(m_player, SIGNAL(SIG_getOneimage(QImage)),
          this, SLOT(slot_setImage(QImage)));  // Qt自动队列化跨线程
  ```

> **面试官追问**：Qt信号槽跨线程的原理？
> - Qt元对象系统（MOC）在编译时生成元对象代码
> - 跨线程时，emit将信号参数打包成事件→投递到目标线程的事件循环→目标线程在事件循环中调用slot
> - 这是**队列连接（Qt::QueuedConnection）**，比直接连接（Qt::DirectConnection）多一步事件投递，但保证线程安全

### 难点2：登录后动态创建MainDialog

- **问题**：MainDialog在登录成功后才创建，需要在创建时连接所有信号
- **解决**：在slot_dealClientData的LOGIN_RS分支中创建MainDialog并连接信号
  ```cpp
  if(!m_mainDialog) {
      m_mainDialog = new MainDialog;
      connect(m_mainDialog, SIGNAL(SIG_RecommendRq(int,int)),
              this, SLOT(slot_recommendRq(int,int)));
      // ... 连接所有信号
  }
  m_mainDialog->setUserId(m_userid);
  m_mainDialog->show();
  ```

> **面试官追问**：为什么不在CKernel构造时创建MainDialog？
> - 未登录用户不需要MainDialog（只显示PlayerDialog）
> - 延迟创建节省资源，登录后才分配

## 模拟面试题

### Q1：你的CKernel单例和MVC架构有什么关系？
**回答要点**：
- CKernel是**Controller**角色：接收UI信号（View）→转发为网络请求（Model）→接收网络响应→分发到UI更新
- View：LoginDialog、MainDialog、PlayerDialog、CommentDialog
- Model：TcpClientMediator（网络）、VideoPlayer（音视频）
- **但不是严格MVC**：CKernel同时承担了Controller和Mediator的角色，UI之间不直接通信，全部通过CKernel中转

### Q2：如果用户切换视频时上一个视频还在播放，怎么处理？
**回答要点**：
- PlayerDialog.playUrl()中先调用`m_player->stop(true)`停止上一个视频
- stop(true)会等待VideoPlayer线程自然结束（while(!readThreadFinished)循环等待）
- 然后调用`m_player->setFileName(newUrl)`启动新视频
- **潜在问题**：如果网络流卡住，stop等待可能超时→需要加超时保护（当前代码没有）

### Q3：你的配置文件如果被恶意篡改（IP改为攻击者服务器），客户端会怎样？
**回答要点**：
- 客户端会连接到攻击者服务器→所有数据（密码、视频等）发送给攻击者→**严重安全风险**
- **修复方案**：
  1. 配置文件加密存储（AES加密）
  2. 程序内置默认IP，配置文件只是覆盖选项
  3. 连接时验证服务器证书（TLS）
  4. **最简单的方案**：config.ini设为只读权限，普通用户无法修改
