# 14 客户端网络通信层（netapi）

## 模块概述

`netapi/` 目录封装了客户端的网络通信抽象层，采用**中介者模式**（INetMediator接口 → TcpClientMediator/TcpServerMediator实现），基于Qt QTcpSocket异步通信，通过SIG_ReadyData信号回调将网络数据传递给CKernel，粘包处理在中介者层完成（4字节长度头+循环读取完整包）。

## 技术选型与理由

> **面试官追问**：为什么用中介者模式？为什么用QTcpSocket而不是原生socket？

| 方案 | 优势 | 劣势 | 选择理由 |
|------|------|------|----------|
| **中介者模式+QTcpSocket** | 抽象层可替换（TCP/UDP/WebSocket）、Qt异步信号槽跨线程安全 | Qt依赖 | **中介者模式让CKernel不关心底层通信方式**，QTcpSocket的异步信号槽天然适合Qt事件循环 |
| 原生socket+阻塞I/O | 无Qt依赖、可控 | 需自己处理异步/跨线程/粘包 | Qt项目用原生socket需要在QThread中手动管理，不如QTcpSocket方便 |
| 原生socket+非阻塞I/O | 性能极致 | epoll/select在Windows不可用（需WSAAsyncSelect） | 客户端是Windows平台，原生非阻塞I/O需要用Winsock API→跨平台困难 |

> **面试官追问**：中介者模式和直接调用TcpClient有什么区别？
> - 直接调用：CKernel依赖TcpClient的具体实现→如果要换成UDP或WebSocket→需要修改CKernel
> - 中介者模式：CKernel只依赖INetMediator接口→换通信方式只需创建不同的Mediator→CKernel无需修改
> - **这是"依赖倒置原则"（DIP）的典型应用**

## 核心实现流程

### INetMediator抽象接口

```cpp
class INetMediator {
public:
    virtual bool OpenNet(const char* ip = NULL) = 0;  // 打开网络连接
    virtual void CloseNet() = 0;                       // 关闭连接
    virtual int SendData(unsigned int lSendIP, char* szbuf, int nlen) = 0;  // 发送数据
signals:
    virtual void SIG_ReadyData(unsigned int lSendIP, char* buf, int nlen) = 0;  // 数据到达信号
};
```

### TcpClientMediator：QTcpSocket异步通信

```cpp
class TcpClientMediator : public INetMediator {
    QTcpSocket *m_tcpSocket;
    
    bool OpenNet(const char* ip) {
        m_tcpSocket = new QTcpSocket;
        m_tcpSocket->connectToHost(ip, 8000);  // 连接服务端
        connect(m_tcpSocket, SIGNAL(readyRead()),
                this, SLOT(slot_readyRead()));  // 数据到达→Qt信号
        connect(m_tcpSocket, SIGNAL(SIG_ReadyData(uint,char*,int)),
                this, SIGNAL(SIG_ReadyData(uint,char*,int)));  // 传递给CKernel
    }
    
    int SendData(unsigned int lSendIP, char* szbuf, int nlen) {
        // ★ 粘包协议：先发4字节包长度，再发数据包内容
        int packLen = nlen;
        m_tcpSocket->write((char*)&packLen, 4);  // 4字节长度头
        m_tcpSocket->write(szbuf, nlen);           // 数据包
        return nlen;
    }
};
```

### 粘包处理：4字节长度头+循环读取

```cpp
void TcpClientMediator::slot_readyRead() {
    while(m_tcpSocket->bytesAvailable() > 0) {
        // 1. 读4字节包长度
        if(m_tcpSocket->bytesAvailable() < 4) return;  // 长度头还没到
        
        int packLen = 0;
        m_tcpSocket->read((char*)&packLen, 4);  // 读长度
        
        // 2. 等待完整数据包到达
        while(m_tcpSocket->bytesAvailable() < packLen) {
            m_tcpSocket->waitForReadyRead(100);  // 等待100ms
        }
        
        // 3. 读取完整数据包
        char* buf = new char[packLen];
        m_tcpSocket->read(buf, packLen);
        
        // 4. ★ 发射信号→CKernel处理
        Q_EMIT SIG_ReadyData(0, buf, packLen);
    }
}
```

> **面试官追问**：waitForReadyRead(100)会不会阻塞UI？
> - 不会！slot_readyRead在Qt事件循环中调用，waitForReadyRead只等待100ms
> - 如果100ms内数据没到→返回→下次readyRead信号时继续读取
> - **但100ms可能导致延迟**：如果网络慢，每100ms才检查一次→优化：改为事件驱动（用QDataStream缓冲区累积读取）

> **面试官追问**：为什么new char[packLen]而不是用固定缓冲区？
> - 不同协议包大小不同：LIKE_RQ=13字节，UPLOAD_RQ=680字节，FILEBLOCK_RQ=4KB+
> - 固定缓冲区需要最大包大小（4KB+）→浪费内存
> - 动态分配按需分配→内存效率更高
> - **注意**：buf由CKernel的slot_dealClientData在处理完后delete[]释放

## 遇到的难点

### 难点1：QTcpSocket粘包处理

- **问题**：TCP是流式协议，多个包可能合并成一个TCP段到达→需要区分包边界
- **解决**：4字节长度头协议→先读长度→再读对应长度的数据→保证包边界清晰

> **面试官追问**：粘包和半包的区别？
> - **粘包**：两个包合并成一个TCP段→需要拆分→4字节长度头解决
> - **半包**：一个包被拆成两个TCP段→需要拼接→waitForReadyRead等待完整包
> - 本项目的粘包处理同时解决了粘包和半包问题

### 难点2：断线重连

- **问题**：网络断开时QTcpSocket自动断开→用户需要重新登录
- **当前方案**：没有自动重连！断开后用户需要重启程序
- **优化方案**：QTcpSocket的disconnected信号→尝试重连（3次，间隔5秒）→重连成功后重新登录

## 模拟面试题

### Q1：你的QTcpSocket和原生socket在Windows上的性能差距？
**回答要点**：
- QTcpSocket底层用Winsock API（WSAAsyncSelect），和原生socket本质相同
- 性能差距：QTcpSocket多一层Qt事件循环→每次读写多约10μs的开销→万级请求/秒场景下差距约10%
- **但客户端只有1个TCP连接**→万级请求/秒不存在→性能差距可忽略
- QTcpSocket的优势：异步信号槽+跨线程安全+自动事件循环→开发效率远高于原生socket

### Q2：如果要支持UDP通信（如实时弹幕），中介者模式怎么扩展？
**回答要点**：
- 创建UdpMediator继承INetMediator
- OpenNet改为QUdpSocket::bind
- SendData改为QUdpSocket::writeDatagram（无粘包问题，UDP每个包独立）
- CKernel只需创建UdpMediator替代TcpClientMediator→其他代码不变
- **这就是中介者模式的价值**：通信方式变更不影响业务逻辑

### Q3：你的粘包处理和服务端的有什么区别？
**回答要点**：
- 服务端：recv_task中先recv 4字节长度→循环recv完整包→阻塞模式
- 客户端：QTcpSocket readyRead信号→先read 4字节→waitForReadyRead等待完整包→异步模式
- **核心协议相同**（4字节长度头+数据包），但实现方式不同（阻塞recv vs 异步信号）
- 服务端阻塞模式适合线程池处理，客户端异步模式适合Qt事件循环
