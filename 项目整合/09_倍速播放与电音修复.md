# 09 倍速播放与电音修复（atempo滤镜 + 自适应降速）

## 模块概述

倍速播放通过FFmpeg **atempo音频滤镜**实现变速不变调，配合**aformat滤镜**强制输出S16格式，以及**自适应降速机制**应对网络流队列耗空。电音问题经过三层排查（SDL混音叠加→rubberband相位失真→FLT/S16格式不匹配）最终修复。

## 技术选型与理由

> **面试官追问**：atempo的原理是什么？为什么不直接改采样率？

| 方案 | 原理 | 优势 | 劣势 | 选择理由 |
|------|------|------|------|----------|
| **atempo滤镜** | WSOLA（波形相似度叠加）算法：在音频波形中找相似片段→重叠拼接→变速但保持音调 | 变速不变调、音质好 | 0.5~2.0范围限制、有内部缓冲延迟 | **业界标准方案**，ffplay也用atempo |
| 改采样率 | 降低采样率→播放变快但音调升高（男声变女声） | 最简单 | **变调**→不可接受 | 短视频平台必须变速不变调 |
| rubberband | 相位声码器：FFT→相位调整→IFFT | 理论音质更好 | **实测部分素材有相位叠加电音感**、依赖librubberband | 尝试后发现电音，已弃用 |
| 简单丢帧 | 每2帧丢1帧→2倍速 | 最快 | 音频断裂、严重失真 | 不可接受 |

> **面试官追问**：atempo为什么限制0.5~2.0？超过2.0怎么办？
> - WSOLA算法的重叠窗口在极端倍率下（>2.0或<0.5）无法找到足够相似的片段→拼接质量严重下降
> - **解决方案**：串联多个atempo，每个不超过2.0
>   - 3.0x = atempo(tempo=2.0) + atempo(tempo=1.5) → 2.0×1.5=3.0
>   - 4.0x = atempo(tempo=2.0) + atempo(tempo=2.0) → 2.0×2.0=4.0

## 核心实现流程

### 滤镜图构建：abuffer → atempo → aformat → abuffersink

```cpp
bool init_audio_filter(VideoState *is) {
    // 销毁旧滤镜图
    if(is->audio_filter_graph) avfilter_graph_free(&is->audio_filter_graph);
    
    is->audio_filter_graph = avfilter_graph_alloc();
    
    // 1. abuffer源滤镜（输入：解码器格式）
    char args[512];
    snprintf(args, sizeof(args),
             "time_base=1/%d:sample_rate=%d:sample_fmt=%s:channel_layout=0x%llX",
             is->pAudioCodecCtx->sample_rate,
             is->pAudioCodecCtx->sample_rate,
             av_get_sample_fmt_name((AVSampleFormat)is->audio_decoder_fmt),
             (long long)is->pAudioCodecCtx->channel_layout);
    avfilter_graph_create_filter(&is->audio_filter_src, abuffersrc, "in", args, NULL, ...);
    
    // 2. abuffersink输出滤镜
    avfilter_graph_create_filter(&is->audio_filter_sink, abuffersink, "out", NULL, NULL, ...);
    
    // 3. ★ atempo滤镜（变速不变调）
    AVFilterContext *last = is->audio_filter_src;
    double remaining = is->playbackSpeed;
    
    // 倍速>2.0时串联多个atempo
    while(remaining > 2.0) {
        AVFilterContext *atempo;
        avfilter_graph_create_filter(&atempo, avfilter_get_by_name("atempo"),
                                     "atempo1", "tempo=2.0", NULL, ...);
        avfilter_link(last, 0, atempo, 0);
        last = atempo;
        remaining /= 2.0;
    }
    
    // 最后一个atempo
    if(fabs(remaining - 1.0) > 0.001) {
        char tmp[64];
        snprintf(tmp, sizeof(tmp), "tempo=%.4f", remaining);
        AVFilterContext *atempo;
        avfilter_graph_create_filter(&atempo, avfilter_get_by_name("atempo"),
                                     "atempo_last", tmp, NULL, ...);
        avfilter_link(last, 0, atempo, 0);
        last = atempo;
    }
    
    // 4. ★ aformat滤镜（强制输出S16，和SDL输出格式一致）
    char aformat_args[256];
    snprintf(aformat_args, sizeof(aformat_args),
             "sample_fmts=s16:sample_rates=%d:channel_layouts=0x%llX",
             is->pAudioCodecCtx->sample_rate,
             (long long)is->pAudioCodecCtx->channel_layout);
    AVFilterContext *aformat_ctx;
    avfilter_graph_create_filter(&aformat_ctx, aformat_filter, "fmt", aformat_args, NULL, ...);
    avfilter_link(last, 0, aformat_ctx, 0);
    last = aformat_ctx;
    
    // 5. 链接到输出
    avfilter_link(last, 0, is->audio_filter_sink, 0);
    avfilter_graph_config(is->audio_filter_graph, NULL);
}
```

### 倍速切换：setSpeed → pending_speed → 滤镜重建

```cpp
void VideoPlayer::setSpeed(double speed) {
    if(speed < 0.25) speed = 0.25;
    if(speed > 4.0) speed = 4.0;
    if(fabs(m_videoState.playbackSpeed - speed) < 0.001) return;  // 速度没变
    
    // ★ 只标记请求，不立即重建（避免线程安全问题）
    m_videoState.target_speed = speed;
    m_videoState.pending_speed = speed;
    m_videoState.audio_filter_rebuild = 1;  // 音频滤镜重建标志
    m_videoState.video_filter_rebuild = 1;  // 视频滤镜重建标志
}
```

> **面试官追问**：为什么不立即重建滤镜？
> - setSpeed由UI主线程调用，但滤镜在audio_decode_frame（SDL音频回调线程）中使用
> - **跨线程直接重建→竞态条件**：可能正在用旧滤镜处理帧时，新滤镜被创建→崩溃
> - 只标记pending_speed和rebuild标志，由audio_decode_frame在下一次调用时安全重建

### 滤镜重建：drain旧缓冲 + 重建新滤镜图

```cpp
// audio_decode_frame中的滤镜重建逻辑
if(is->audio_filter_rebuild) {
    // 1. ★ drain旧atempo中残留的缓冲数据（否则重建时丢失→音频断裂）
    if(is->audio_filter_graph && is->playbackSpeed != 1.0) {
        for(int flush_i = 0; flush_i < 3; flush_i++)  // 多次flush确保彻底
            av_buffersrc_add_frame(is->audio_filter_src, NULL);
        AVFrame *drain_frame = av_frame_alloc();
        while(av_buffersink_get_frame(is->audio_filter_sink, drain_frame) >= 0)
            av_frame_unref(drain_frame);
        av_frame_free(&drain_frame);
    }
    
    // 2. 更新倍速值
    is->playbackSpeed = is->pending_speed;
    
    // 3. 重建swr_ctx（1.0x时需要，非1.0x时由aformat处理）
    if(is->swr_ctx) { swr_free(&is->swr_ctx); is->swr_ctx = NULL; }
    if(fabs(is->playbackSpeed - 1.0) < 0.01 && is->audio_decoder_fmt != AV_SAMPLE_FMT_S16) {
        // 1.0x：需要swr_ctx做解码器格式→S16转换
        is->swr_ctx = swr_alloc_set_opts(...);
        swr_init(is->swr_ctx);
    }
    
    // 4. 重建滤镜图
    init_audio_filter(is);
    is->audio_filter_rebuild = 0;
}
```

### 自适应降速：网络流队列耗空时自动降速

```cpp
// audio_decode_frame中的自适应逻辑
if(is->target_speed > 1.0 + 0.01 && audioq) {
    int q = audioq->nb_packets;
    Uint32 now = SDL_GetTicks();
    Uint32 since_last = now - is->last_adaptive_change;
    
    // ★ 队列极低（<3包）且冷却已过（>5秒）→ 降速到1.0x
    if(q < 3 && is->playbackSpeed > 1.0 + 0.01 && since_last > 5000) {
        is->pending_speed = 1.0;
        is->audio_filter_rebuild = 1;
        is->last_adaptive_change = now;
    }
    // ★ 队列恢复（>30包）→ 回到目标倍速
    else if(is->playbackSpeed < is->target_speed - 0.01 && since_last > 5000) {
        if(q > 30) {
            is->pending_speed = is->target_speed;
            is->audio_filter_rebuild = 1;
            is->last_adaptive_change = now;
        }
        // 时间兜底：1.0x超过10秒→强制尝试恢复
        else if(elapsed > 10000) {
            is->pending_speed = is->target_speed;
            is->audio_filter_rebuild = 1;
            is->last_adaptive_change = now;
        }
    }
}
```

> **面试官追问**：5秒冷却时间怎么确定的？
> - 每次切换倍速都需要重建滤镜→重建过程中有短暂音频断裂（约50ms）
> - 如果频繁切换（如每秒切换一次），音频会频繁断裂→听起来像卡顿
> - 5秒冷却确保：降速后至少5秒不会再切换→用户感知不到断裂
> - **如果流量增加10倍**：网络流数据到达更不稳定→冷却时间可能需要延长到10秒

## 遇到的难点：电音问题的三层排查

### 第一层：SDL_MixAudioFormat混音叠加

- **现象**：1.5x/2.0x时出现"正常声音+叠加电音"
- **排查**：
  1. `SDL_MixAudioFormat`是**混音函数**，把源数据**加到**目标stream已有数据上
  2. `memset(stream,0,len)`只在循环开始前清零一次
  3. 倍速时atempo输出样本数变少→需要多次循环填满SDL buffer→每次循环`stream`指针前进→新数据叠加到**前一次循环已写入的区域**
- **修复**：改为`memcpy`直接拷贝
  ```cpp
  // 修改前（混音叠加→双重声音）：
  SDL_MixAudioFormat(stream, ..., AUDIO_S16SYS, len1, 100);
  
  // 修改后（直接拷贝→无叠加）：
  memcpy(stream, (uint8_t*)is->audio_buf + is->audio_buf_index, len1);
  ```

### 第二层：rubberband滤镜相位叠加

- **现象**：修复第一层后仍有轻微电音感（"一层叠加的金属质感"）
- **排查**：
  1. 代码中优先尝试rubberband滤镜（相位声码器）
  2. rubberband用FFT→相位调整→IFFT，在某些素材上会产生**相位叠加失真**
  3. 人声素材尤其明显（人声的谐波结构在相位调整后产生金属感）
- **修复**：去掉rubberband优先路径，强制只用atempo
  ```cpp
  // 修改前（优先rubberband）：
  const AVFilter *rb_filter = avfilter_get_by_name("rubberband");
  if(rb_filter) { ... use rubberband ... }
  
  // 修改后（只用atempo）：
  // 直接串联atempo，不再尝试rubberband
  ```

### 第三层：FLT/S16格式不匹配

- **现象**：修复前两层后仍有轻微失真
- **排查**：
  1. atempo输出FLT格式（float），但swr_ctx配置为FLTP→S16转换
  2. FLT和FLTP是不同格式（FLT=packed float, FLTP=planar float），swr_ctx无法正确处理FLT→S16
  3. 格式转换错误→音频数据被错误解读→失真
- **修复**：添加aformat滤镜在滤镜图内部完成FLT→S16转换
  ```cpp
  // aformat强制输出S16，彻底绕过swr_ctx
  snprintf(aformat_args, sizeof(aformat_args),
           "sample_fmts=s16:sample_rates=%d:channel_layouts=0x%llX", ...);
  avfilter_graph_create_filter(&aformat_ctx, aformat_filter, "fmt", aformat_args, ...);
  ```

> **面试官追问**：三层排查的思路是什么？体现了什么能力？
> - **系统性调试能力**：不是"改一个地方试试"，而是从**数据流链路**逐层排查
> - 链路：SDL输出层 → FFmpeg滤镜层 → 数据格式层
> - 每层排查：先确认该层是否有问题→修复→验证→如果还有问题→继续排查下一层
> - **体现了"不猜测、按链路排查"的工程思维**

## 性能优化

- **滤镜重建开销**：每次切换倍速需要销毁旧滤镜图+创建新滤镜图→约50ms音频断裂
  - 优化思路：atempo支持动态调整tempo参数（`avfilter_graph_send_command`），无需重建整个滤镜图
  - 但本项目FFmpeg 4.2.2版本不支持该API，只能重建
- **自适应降速冷却**：5秒冷却避免频繁重建→减少音频断裂次数

> **面试官追问**：如果流量增加10倍，自适应降速哪里会先出问题？
> - **冷却时间过长**：网络流数据到达不稳定时，5秒冷却可能导致降速后迟迟不能恢复→用户长时间停留在1.0x
> - 优化：根据网络质量动态调整冷却时间（网络好→2秒冷却，网络差→10秒冷却）

## 模拟面试题

### Q1：atempo和rubberband的区别？为什么rubberband理论更好但实测更差？
**回答要点**：
- atempo=WSOLA（波形相似度叠加）：在时域找相似片段→重叠拼接→简单但有效
- rubberband=相位声码器：FFT→频域相位调整→IFFT→理论上更精确
- **rubberband实测更差的原因**：
  1. 相位声码器在频域操作，FFT窗口大小固定→某些频率的相位调整不精确→金属感
  2. 人声的谐波结构复杂（基频+多次谐波），相位调整后谐波间的关系被破坏→失真
  3. WSOLA在时域操作，直接找相似波形片段→不破坏谐波结构→人声更自然
- **结论**：理论最优≠实测最优，必须用实际素材验证

### Q2：倍速播放时视频怎么同步加速？
**回答要点**：
- atempo让音频少输出样本→SDL以原采样率播放→音频实际播放变快
- audio_clock用原始PTS更新→但音频实际播放速率×speed→audio_clock前进速率×speed
- 视频线程比较video_clock和audio_clock→video_clock也需要×speed才能跟上
- **关键**：video_clock和audio_clock都用原始PTS，在同一时间域可直接比较
- 倍速效果：atempo压缩音频→SDL快速播放→audio_clock快速前进→视频同步等待时间缩短→视频也快速渲染→自然实现倍速

### Q3：你的自适应降速和YouTube的自适应码率（ABR）有什么区别？
**回答要点**：
- YouTube ABR：根据网络带宽动态切换视频分辨率（1080p→720p→480p），保证流畅播放
- 本项目自适应降速：根据音频队列深度动态切换播放速度（2.0x→1.0x），保证音频不中断
- **核心思想相同**：牺牲质量（分辨率/速度）换取流畅性
- **区别**：ABR切换码率是离散的（几个预设分辨率），降速是连续的（0.5~4.0任意值）
- YouTube还用HLS多码率流，本项目用RTMP单码率流→无法做ABR
