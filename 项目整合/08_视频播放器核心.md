# 08 视频播放器核心（FFmpeg + SDL）

## 模块概述

`videoplayer.h/cpp` 是客户端音视频播放的核心引擎，采用**三线程模型**（读线程+视频线程+音频回调），基于FFmpeg解码+SDL音频输出，通过PacketQueue生产者-消费者队列实现线程间数据传递，以音频时钟为主时钟驱动音视频同步。

## 技术选型与理由

> **面试官追问**：为什么用三线程而不是两线程？为什么选SDL而不是Qt音频？

| 方案 | 优势 | 劣势 | 选择理由 |
|------|------|------|----------|
| **三线程（读+视频+音频回调）** | 读线程不阻塞解码、音视频独立处理、同步逻辑清晰 | 线程间通信需队列+锁 | **读线程必须独立**——av_read_frame可能阻塞（网络流），如果和解码在同一线程，阻塞时音视频都卡住 |
| 两线程（读+解码合并） | 线程少 | av_read_frame阻塞时解码也停→音频中断 | 网络流场景下两线程不可行 |
| Qt QMediaPlayer | 一行代码搞定 | 无法自定义倍速/滤镜/同步逻辑 | 项目核心是学习FFmpeg底层原理，QMediaPlayer是黑盒 |

> **面试官追问**：为什么音频用SDL回调而不是独立线程？
> - SDL音频回调由SDL内部线程驱动，回调频率由SDL_AUDIO_BUFFER_SIZE（1024样本）决定
> - 回调模式保证**音频输出节奏由硬件采样率驱动**，不会因为CPU调度导致音频断续
> - 如果用独立线程+定时推送，线程调度不精确→音频抖动

## 核心实现流程

### 三线程模型

```
┌─────────────────────────────────────────────┐
│              读线程 (VideoPlayer::run)        │
│  av_read_frame → PacketQueue入队              │
│  音频包→audioq  视频包→videoq                  │
└──────────────────┬──────────────────────────┘
                   │ PacketQueue
┌──────────────────┴──────────┬────────────────┐
│     视频线程 (video_thread)  │  音频回调(audio_callback) │
│  取包→解码→PTS同步等待→      │  取包→解码→滤镜→PCM→       │
│  YUV转RGB→emit QImage信号   │  memcpy到SDL stream        │
│                              │  更新audio_clock            │
└──────────────────────────────┴────────────────────────────┘
```

### PacketQueue：生产者-消费者队列

```cpp
typedef struct PacketQueue {
    AVPacketList *first_pkt, *last_pkt;  // 链表头尾
    int nb_packets;                       // 包数量
    int size;                             // 总字节大小
    SDL_mutex *mutex;                     // 互斥锁
    SDL_cond  *cond;                      // 条件变量
    int abort_request;                    // 中止标志（quit时唤醒阻塞的消费者）
} PacketQueue;

// 生产者（读线程）：入队
int packet_queue_put(PacketQueue *queue, AVPacket *packet) {
    SDL_LockMutex(queue->mutex);
    AVPacketList *pkt_list = new AVPacketList;
    pkt_list->pkt = *packet;
    // 链表操作...
    queue->nb_packets++;
    queue->size += packet->size;
    SDL_CondSignal(queue->cond);  // 通知消费者
    SDL_UnlockMutex(queue->mutex);
}

// 消费者（视频/音频线程）：出队
int packet_queue_get(PacketQueue *queue, AVPacket *packet, int block) {
    SDL_LockMutex(queue->mutex);
    while(queue->nb_packets == 0 && !queue->abort_request) {
        if(!block) { SDL_UnlockMutex(queue->mutex); return 0; }
        SDL_CondWait(queue->cond, queue->mutex);  // 阻塞等待
    }
    // 取出第一个包...
    SDL_UnlockMutex(queue->mutex);
}
```

> **面试官追问**：为什么用SDL_mutex而不是pthread_mutex？
> - SDL_mutex跨平台（Windows/Linux/Mac），pthread_mutex只在Linux可用
> - 客户端是Windows平台（MSVC编译），pthread不可用
> - SDL_mutex底层在Linux用pthread_mutex，在Windows用CRITICAL_SECTION

### 读线程：av_read_frame入队

```cpp
void VideoPlayer::run() {
    // 初始化FFmpeg：打开文件→查找流→打开编解码器→SDL音频
    // ...
    
    while(1) {
        if(m_videoState.quit) break;
        if(m_videoState.isPause) { SDL_Delay(10); continue; }
        
        // seek处理...
        
        // ★ 读取数据包
        if(av_read_frame(pFormatCtx, packet) < 0) {
            DelayCount++;
            if(DelayCount >= 300) { m_videoState.readFinished = true; break; }
            SDL_Delay(10); continue;
        }
        
        // 视频包：队列满时丢包（避免阻塞读线程饿死音频队列）
        if(packet->stream_index == m_videoState.videoStream) {
            if(m_videoState.videoq->size > MAX_VIDEO_SIZE)
                av_packet_unref(packet);  // 丢包
            else
                packet_queue_put(m_videoState.videoq, packet);
        }
        // 音频包：永远入队（不限大小，保证倍速时音频不中断）
        else if(packet->stream_index == m_videoState.audioStream) {
            packet_queue_put(m_videoState.audioq, packet);
        }
    }
}
```

> **面试官追问**：为什么视频队列满时丢包而音频不限大小？
> - 视频帧大（每帧几十KB），队列积压会消耗大量内存→丢包不影响体验（下一帧会覆盖）
> - 音频包小（每帧几KB），且音频中断比视频卡顿更明显→音频队列不能丢包
> - 倍速播放时音频消耗更快（1.5x时音频包消耗速率×1.5），队列需要更多缓冲

### 音视频同步：音频为主时钟

```cpp
// 视频线程中的同步等待逻辑
if(is->audioq != NULL) {
    while(1) {
        if(is->quit) break;
        double audio_pts = is->audio_clock;  // 音频当前PTS（微秒）
        double v_pts = is->video_clock;       // 视频当前PTS（微秒）
        if(v_pts <= audio_pts) break;         // 视频不超前→可以渲染
        
        // 视频超前音频→等待（每5ms检查一次）
        if(++syncWaitCount > 400) break;      // 超时保护（2秒）
        SDL_Delay(5);
    }
}
```

> **面试官追问**：为什么选音频为主时钟而不是视频为主时钟？
> - **音频更敏感**：人耳对音频断续的感知远强于人眼对视频卡顿的感知（50ms音频断裂就很明显，100ms视频卡顿才明显）
> - **音频节奏固定**：SDL以固定采样率播放，audio_clock前进速度稳定
> - **视频可以等**：视频帧超前时等待audio_clock追赶，用户感知不到2~3帧的延迟
> - 如果选视频为主时钟：音频需要跳帧或变速来追赶视频→音频质量受损

> **面试官追问**：倍速时同步逻辑怎么工作？
> - atempo让音频少输出样本→SDL以原采样率播放→音频实际播放变快
> - audio_clock用原始PTS更新（不变），但音频实际播放速度变快→audio_clock前进速率×speed
> - 视频线程比较video_clock和audio_clock→video_clock也需要×speed才能跟上
> - **关键**：video_clock和audio_clock都用原始PTS，在同一时间域可直接比较
> - 倍速效果：atempo压缩音频→SDL快速播放→audio_clock快速前进→视频同步等待时间缩短→视频也快速渲染→自然实现倍速

### 音频回调：SDL驱动填充

```cpp
void audio_callback(void *userdata, Uint8 *stream, int len) {
    VideoState *is = (VideoState*)userdata;
    memset(stream, 0, len);  // ★ 先清零（重要！）
    if(is->isPause) return;
    
    while(len > 0) {
        if(is->audio_buf_index >= is->audio_buf_size) {
            // 缓冲区耗尽→解码新帧填充
            audio_data_size = audio_decode_frame(is, is->audio_buf, sizeof(is->audio_buf));
            if(audio_data_size < 0) {
                // 解码失败→插入静音
                is->audio_buf_size = 1024;
                memset(is->audio_buf, 0, is->audio_buf_size);
            } else {
                is->audio_buf_size = audio_data_size;
            }
            is->audio_buf_index = 0;
        }
        
        // ★ memcpy而非SDL_MixAudioFormat（修复电音Bug的关键）
        int len1 = is->audio_buf_size - is->audio_buf_index;
        if(len1 > len) len1 = len;
        memcpy(stream, (uint8_t*)is->audio_buf + is->audio_buf_index, len1);
        
        len -= len1;
        stream += len1;
        is->audio_buf_index += len1;
    }
}
```

## 遇到的难点

### 难点1：SDL_MixAudioFormat导致电音（详见09章节）

- **问题**：1.5x/2.0x时出现正常声音叠加电音
- **根因**：`SDL_MixAudioFormat`是混音叠加函数，把新音频数据加到stream已有数据上→多次循环时叠加→双重声音
- **解决**：改为`memcpy`直接拷贝，`memset(stream,0,len)`在循环开始前清零一次即可

### 难点2：seek后音频时钟回退导致视频卡住

- **问题**：seek后audio_clock停留在旧位置，video_clock跳到新位置→video_clock > audio_clock→视频线程一直等待→画面卡住
- **解决**：seek成功后重置audio_clock到seek位置
  ```cpp
  m_videoState.audio_clock = (double)m_videoState.seek_pos;
  ```

### 难点3：网络流av_read_frame阻塞

- **问题**：RTMP流av_read_frame可能阻塞数秒（网络波动），阻塞期间读线程无法入队→音视频队列耗空→播放中断
- **解决**：设置中断回调`decode_interrupt_cb`，quit时返回1中断阻塞操作
  ```cpp
  pFormatCtx->interrupt_callback.callback = decode_interrupt_cb;
  pFormatCtx->interrupt_callback.opaque = &m_videoState;
  // quit时回调返回1→av_read_frame立即返回→线程可退出
  ```

> **面试官追问**：如果流量增加10倍（更多用户同时播放RTMP流），哪里会先出问题？
> - **nginx-rtmp带宽**：10万用户同时播放，单机带宽不够→需要CDN分发
> - **客户端缓冲**：RTMP流数据到达慢→队列耗空→播放中断→增大rtmp_buffer（当前30秒）或自适应降速
> - **服务端转码**：10万个ffmpeg转码进程→CPU爆炸→需要排队转码

## 模拟面试题

### Q1：你的音视频同步和ffplay的实现有什么区别？
**回答要点**：
- ffplay用**帧率同步**：计算每帧应显示的时间，用av_usleep精确等待
- 本项目用**音频主时钟同步**：视频帧等待audio_clock追赶，SDL_Delay(5)粗略等待
- ffplay更精确（微秒级），本项目更简单（5ms级）
- **但本项目更适合倍速播放**：atempo改变音频输出速率→audio_clock自然加速→视频同步等待缩短→倍速无需额外逻辑

### Q2：PacketQueue的abort_request机制是什么？为什么需要？
**回答要点**：
- 当用户停止播放（quit=true），读线程和音视频线程都需要退出
- 但消费者可能正在SDL_CondWait阻塞等待新包→如果没有abort_request，消费者永远等不到signal→线程无法退出→程序卡住
- abort_request=1时，packet_queue_get立即返回0→消费者检测到0→退出循环→线程安全结束
- packet_queue_abort同时SDL_CondSignal唤醒所有等待的消费者

### Q3：为什么视频帧用SWS_BICUBIC转换而不是SWS_FAST_BILINEAR？
**回答要点**：
- SWS_BICUBIC：双三次插值，画质好但慢（约5ms/帧）
- SWS_FAST_BILINEAR：快速双线性插值，快但画质差（约1ms/帧，有明显锯齿）
- 本项目选BICUBIC：视频播放对画质要求高，5ms/帧在30fps下可接受（每帧33ms预算）
- **如果需要更低延迟**（如直播），可改用FAST_BILINEAR或硬件加速（VA-API/D3D11VA）
