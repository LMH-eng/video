# 05 业务逻辑层（注册/登录/上传/播放/点赞/评论/搜索）

## 模块概述

`clogic.h/cpp` 是服务端业务逻辑核心，通过**协议号→函数指针映射实现O(1)路由**，处理注册、登录、上传、推荐、播放、点赞、评论、搜索等14类业务请求，与MySQL和Redis交互。

## 技术选型与理由

> **面试官追问**：为什么用函数指针映射而不是switch-case？

| 方案 | 优势 | 劣势 | 选择理由 |
|------|------|------|----------|
| **函数指针数组** | O(1)路由、添加新协议只需注册映射、代码结构清晰 | 数组大小固定、协议号必须连续偏移 | 28个协议号从10000连续编号，偏移直接做索引，**添加新业务只需一行映射注册** |
| switch-case | 直观、无需数组 | 添加新协议需修改switch、28个case代码膨胀 | 28个case的switch可读性差，每个case至少5行，总共140+行 |

> **面试官追问**：为什么不用std::unordered_map<int, function>？
- 函数指针数组比hash_map更快（直接索引 vs hash计算+碰撞处理）
- 协议号连续偏移，数组天然适合；hash_map更适合协议号不连续的场景

### 注册流程：手机号+密码+昵称+兴趣标签

```cpp
void CLogic::RegisterRq(sock_fd clientfd, char* szbuf, int nlen) {
    STRU_REGISTER_RQ *rq = (STRU_REGISTER_RQ*)szbuf;
    
    // 1. MySQL查询手机号是否已存在
    char sql[256];
    sprintf(sql, "select tel from user where tel='%s';", rq->tel);
    list<string> res;
    m_sql->Select(sql, res);
    
    if(!res.empty()) {
        // 用户已存在，返回失败
        STRU_REGISTER_RS rs;
        rs.result = user_is_exist;
        SendData(clientfd, (char*)&rs, sizeof(rs));
        return;
    }
    
    // 2. MySQL插入新用户
    sprintf(sql, "insert into user(tel,name,password,hobby) values('%s','%s','%s','%s');",
            rq->tel, rq->name, rq->password, rq->hobby);
    m_sql->Update(sql);
    
    // 3. ★ Redis初始化用户兴趣向量（用于推荐算法）
    initUserHobbyToRedis(userId, rq->hobby);
    
    // 4. 返回注册成功
    STRU_REGISTER_RS rs;
    rs.result = register_success;
    SendData(clientfd, (char*)&rs, sizeof(rs));
}
```

> **面试官追问**：注册时为什么不先查Redis而是查MySQL？
> - Redis只存兴趣向量，不存用户基本信息（tel/name/password）
> - 用户注册是低频操作（一天可能几十次），MySQL查询开销可接受
> - **高频操作（推荐/点赞）才需要Redis加速**

```cpp
    SendData(clientfd, (char*)&rs, sizeof(rs));

```

> **面试官追问**：为什么登录时要衰减注册兴趣？
> - 注册时用户选择的兴趣标签是"初始偏好"，可能不准确（用户随便选的）
> - 衰减公式：`new_weight = old_weight * 0.8`，每次登录衰减20%
> - 实际观看行为会增量更新兴趣（`updateUserHobby`），衰减+增量让兴趣向量逐渐收敛到真实偏好
> - **这是推荐算法的"冷启动→热收敛"策略**

### 上传流程：MD5去重 + 分块传输 + nginx-rtmp转码

```cpp
void CLogic::UploadRq(sock_fd clientfd, char* szbuf, int nlen) {
    STRU_UPLOAD_RQ *rq = (STRU_UPLOAD_RQ*)szbuf;
    
    // 1. ★ MD5去重：查询数据库是否有相同hash的视频
    char sql[256];
    sprintf(sql, "select id from video where filehash='%s';", rq->m_szFileHash);
    list<string> res;
    m_sql->Select(sql, res);
    
    if(!res.empty()) {
        // 视频已存在，秒传！直接返回RTMP URL
        STRU_UPLOAD_RS rs;
        rs.result = video_already_exist;
        SendData(clientfd, (char*)&rs, sizeof(rs));
        return;
    }
    
    // 2. 新视频：创建上传任务映射
    UploadInfo info;
    info.fileId = rq->m_nFileId;
    info.clientfd = clientfd;
    info.userId = rq->m_UserId;
    info.fileSize = rq->m_nFileSize;
    strcpy(info.fileName, rq->m_szFileName);
    strcpy(info.fileHash, rq->m_szFileHash);
    strcpy(info.hobby, rq->m_szHobby);
    strcpy(info.title, rq->m_szTitle);
    
    // 打开临时文件用于写入
    info.fd = open(tempPath, O_WRONLY | O_CREAT, 0666);
    
    // ★ 加锁保护m_mapUpload（多线程可能同时上传）
    pthread_mutex_lock(&m_mapMutex);
    m_mapUpload[info.fileId] = info;
    pthread_mutex_unlock(&m_mapMutex);
    
    // 3. 返回同意上传
    STRU_UPLOAD_RS rs;
    rs.result = upload_accept;
    SendData(clientfd, (char*)&rs, sizeof(rs));
}
```

### 文件块上传：逐块写入临时文件

```cpp
void CLogic::FileBlockRq(sock_fd clientfd, char* szbuf, int nlen) {
    STRU_FILEBLOCK_RQ *rq = (STRU_FILEBLOCK_RQ*)szbuf;
    
    // 1. 从m_mapUpload找到上传任务
    pthread_mutex_lock(&m_mapMutex);
    auto it = m_mapUpload.find(rq->m_nFileId);
    if(it == m_mapUpload.end()) {
        pthread_mutex_unlock(&m_mapMutex);
        return;  // 找不到任务，丢弃
    }
    UploadInfo &info = it->second;
    pthread_mutex_unlock(&m_mapMutex);
    
    // 2. 写入临时文件
    write(info.fd, rq->m_szBlockData, rq->m_nBlockSize);
    info.uploadedSize += rq->m_nBlockSize;
    info.blockIndex++;
    
    // 3. 上传完成？
    if(info.uploadedSize >= info.fileSize) {
        close(info.fd);
        
        // ★ 调用nginx-rtmp的ffmpeg自动转码
        // 上传的视频文件被nginx-rtmp监控目录检测到 → ffmpeg转码 → 生成RTMP流地址
        
        // MySQL插入视频记录
        sprintf(sql, "insert into video(userid,url,gif,hobby,name,filehash,title) values(...);");
        m_sql->Update(sql);
        
        // Redis更新视频分类信息
        // ...
        
        // 从上传映射中移除
        pthread_mutex_lock(&m_mapMutex);
        m_mapUpload.erase(rq->m_nFileId);
        pthread_mutex_unlock(&m_mapMutex);
    }
    
    // 4. 返回文件块确认
    STRU_FILEBLOCK_RS rs;
    rs.result = fileblock_success;
    rs.m_nFileId = rq->m_nFileId;
    rs.m_nBlockIndex = rq->m_nBlockIndex;
    SendData(clientfd, (char*)&rs, sizeof(rs));
}
```

## 遇到的难点

### 难点1：上传乱序Bug（最关键）

- **问题**：客户端先发UPLOAD_RQ紧接着发FILEBLOCK_RQ，但服务端有时先处理FILEBLOCK_RQ
- **发现**：日志显示"FileBlockRq: fileId不存在"，重试又成功
- **排查**：原设计recv_task接收完数据后异步投递Buffer_Deal，两个包被不同线程处理→顺序不确定
- **解决**：recv_task中同步调用m_recv_callback，保证同一fd的包按顺序处理
- **代价**：业务处理在recv线程中执行，但MySQL查询<1ms，影响可忽略

> **面试官追问**：同步处理会不会降低吞吐量？
> - 单个fd的业务处理阻塞不影响其他fd（EPOLLONESHOT保证每个fd独立）
> - 线程池200线程，200个fd可并行处理业务逻辑
> - 瓶颈分析：MySQL查询耗时1ms，200线程×1ms=每秒20万次查询，足够

### 难点2：m_mapUpload的线程安全

- **问题**：多个客户端同时上传，m_mapUpload被多线程并发读写
- **解决**：pthread_mutex_t m_mapMutex保护所有m_mapUpload操作
- **优化**：锁粒度在map级别（不是全局锁），不同fileId的上传可以并发处理

> **面试官追问**：为什么不用std::map + std::mutex？
> - 项目是C风格+少量C++，pthread_mutex_t更符合整体风格
> - std::mutex需要C++11，项目编译环境可能不支持完整C++11

## 性能优化与安全策略

- **MD5秒传**：相同视频不上传第二次，节省带宽和存储
- **SQL注入防护**：当前用sprintf拼接SQL，**存在注入风险**！生产级应改用参数化查询
- **pthread_mutex保护共享数据**：m_mapUpload、Redis操作都加锁

> **面试官追问**：你的SQL拼接有注入风险，怎么改？
> - 当前：`sprintf(sql, "select tel from user where tel='%s';", rq->tel)` → 如果tel是`' OR '1'='1`，查询返回所有用户
> - 修复：改用MySQL参数化查询（prepared statement），`mysql_stmt_bind_param`绑定参数
> - 或在业务层做输入校验：tel必须是纯数字、password长度限制、name不含特殊字符

## 模拟面试题

### Q1：MD5秒传的原理？碰撞概率怎么办？
**回答要点**：
- 客户端计算文件MD5（32字符hex），上传请求中携带m_szFileHash
- 服务端查video表filehash字段，如果已存在则直接返回RTMP URL，跳过上传
- MD5碰撞概率：理论约2^64次运算才可能碰撞，实际视频文件（>1MB）碰撞概率极低
- **但不是零**：生产级可用SHA-256替代MD5，碰撞概率降到2^128
- **更安全的方案**：MD5 + 文件大小双重校验，碰撞概率进一步降低

### Q2：如果10万用户同时上传视频，你的上传流程哪里会先出问题？
**回答要点**：
- **m_mapUpload内存**：10万个UploadInfo×约700字节=70MB，内存可承受
- **临时文件fd**：10万个open文件描述符→Linux默认ulimit 1024，需要调大ulimit或改为异步写入
- **nginx-rtmp转码**：10万个ffmpeg转码进程→CPU和内存爆炸→需要排队转码（消息队列+Worker）
- **磁盘IO**：10万个文件同时写入→磁盘IO瓶颈→改为对象存储（OSS/S3）

### Q3：上传过程中客户端断网了，临时文件怎么清理？
**回答要点**：
- 当前代码没有清理机制！断网后临时文件永远留在磁盘
- **修复方案**：
  1. 定时扫描临时目录，删除超过1小时的未完成文件
  2. 客户端重连后重新上传（从断点续传或从头开始）
  3. 更好的方案：上传完成后才写数据库，临时文件定期清理
