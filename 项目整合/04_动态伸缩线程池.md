# 04 动态伸缩线程池

## 模块概述

`Thread_pool.h/cpp` 实现了生产者-消费者模型的动态伸缩线程池，支持**自动扩容（busy≥80%）和缩容（空闲超60秒退出）**，环形任务队列容量50000，最大200线程。

## 技术选型与理由

> **面试官追问**：为什么不直接用C++11的std::thread？为什么不固定线程数？

| 方案 | 优势 | 劣势 | 选择理由 |
|------|------|------|----------|
| **自实现动态线程池** | 完全可控、展示底层编程能力、自动伸缩 | 需自己处理条件变量/互斥锁/虚假唤醒 | **面试项目核心目的是展示对多线程同步机制的深度理解** |
| C++11 std::thread + 固定数 | 简单、标准库 | 无法动态伸缩、突发流量时线程不够 | 固定线程数在突发流量时吞吐下降 |
| 半同步/半异步模式 | 更高并发度 | 实现复杂、调试困难 | 项目规模不需要这么复杂的模式 |

> **面试官追问**：线程数为什么最大200？怎么确定的？
> - 经验公式：CPU密集型=CPU核心数，I/O密集型=CPU核心数×(1+等待时间/计算时间)
> - 本项目是I/O密集型（MySQL查询+网络recv等待），等待时间远大于计算时间
> - 200是上限，实际运行时Manager会根据busy率动态调整，通常维持在10~50线程

## 核心实现流程

### 线程池创建

```cpp
// 创建线程池：最大200线程，最小10线程，环形队列50000
pool_t *Pool_create(int max_threads, int min_threads, int queue_capacity) {
    pool->max_threads = max_threads;    // 200
    pool->min_threads = min_threads;    // 10
    pool->alive_threads = min_threads;  // 初始10个工作线程
    pool->busy_threads = 0;
    
    // 环形任务队列
    pool->task_queue = new task_t[queue_capacity];  // 50000
    
    // 同步原语
    pthread_mutex_init(&pool->lock, NULL);
    pthread_cond_init(&pool->not_empty, NULL);   // 消费者等待
    pthread_cond_init(&pool->not_full, NULL);    // 生产者等待
    
    // 创建初始10个工作线程 + 1个管理线程
    for(int i = 0; i < min_threads; i++)
        pthread_create(&pool->threads[i], NULL, Custom, pool);
    pthread_create(&pool->manager_tid, NULL, Manager, pool);
}
```

### 生产者：加锁 → 入队 → 信号通知

```cpp
int Producer_add(task_t task, void *arg) {
    pthread_mutex_lock(&pool->lock);
    // 队列满时等待
    while(pool->queue_size == pool->queue_capacity)
        pthread_cond_wait(&pool->not_full, &pool->lock);
    
    // 环形队列入队
    pool->task_queue[pool->queue_rear].function = task;
    pool->task_queue[pool->queue_rear].arg = arg;
    pool->queue_rear = (pool->queue_rear + 1) % pool->queue_capacity;
    pool->queue_size++;
    
    // 通知消费者
    pthread_cond_signal(&pool->not_empty);
    pthread_mutex_unlock(&pool->lock);
}
```

### 消费者：加锁 → 取任务 → 执行 → busy--

```cpp
void *Custom(void *arg) {
    pool_t *pool = (pool_t*)arg;
    while(1) {
        pthread_mutex_lock(&pool->lock);
        // ★ while而非if：防止条件变量虚假唤醒
        while(pool->queue_size == 0 && !pool->shutdown) {
            pthread_cond_wait(&pool->not_empty, &pool->lock);
            // 空闲超时60秒 → 线程自毁（alive > min时才退出）
            if(pool->alive_threads > pool->min_threads) {
                pool->alive_threads--;
                pthread_mutex_unlock(&pool->lock);
                pthread_exit(NULL);
            }
        }
        
        // 取任务（环形队列出队）
        task_t task = pool->task_queue[pool->queue_front];
        pool->queue_front = (pool->queue_front + 1) % pool->queue_capacity;
        pool->queue_size--;
        pool->busy_threads++;
        
        pthread_cond_signal(&pool->not_full);
        pthread_mutex_unlock(&pool->lock);
        
        // ★ 执行任务不加锁，允许并发
        task.function(task.arg);
        
        pthread_mutex_lock(&pool->lock);
        pool->busy_threads--;
        pthread_mutex_unlock(&pool->lock);
    }
}
```

### 管理线程：每5秒检测 → 扩容

```cpp
void *Manager(void *arg) {
    pool_t *pool = (pool_t*)arg;
    while(!pool->shutdown) {
        sleep(5);
        int busy = pool->busy_threads;
        int alive = pool->alive_threads;
        
        // ★ 扩容策略：busy >= 80% * alive → 每次新增min_threads个
        if(busy * 100 >= alive * 80) {
            for(int i = 0; i < pool->min_threads && pool->alive_threads < pool->max_threads; i++) {
                pthread_create(&pool->threads[pool->alive_threads++], NULL, Custom, pool);
            }
        }
        // 缩容由Custom线程自毁实现（空闲60秒退出），不由Manager主动kill
    }
}
```

> **面试官追问**：为什么扩容每次加10个而不是1个？
> - busy≥80%说明压力较大，每次只加1个需要5秒×多次扩容才能缓解
> - 加10个一次性缓解，减少扩容轮次
> - 不能加太多（如50个），可能只是短暂峰值，浪费资源

## 遇到的难点

### 难点1：条件变量虚假唤醒

- **问题**：`pthread_cond_wait`可能在没有signal的情况下返回，导致从空队列取任务→崩溃
- **发现**：偶发性崩溃，日志显示"取任务时queue_size=0"
- **解决**：用`while`而非`if`检查条件
  ```cpp
  // 错误（虚假唤醒时直接取→崩溃）：
  if(pool->queue_size == 0) pthread_cond_wait(...);
  
  // 正确（虚假唤醒后重新检查）：
  while(pool->queue_size == 0 && !pool->shutdown) pthread_cond_wait(...);
  ```

### 难点2：线程退出安全性

- **问题**：早期用`pthread_cancel`杀空闲线程，cancel可能在持锁时被杀→死锁
- **排查**：服务端偶发性无响应，gdb发现有线程持锁后被cancel
- **解决**：改为**线程自毁**——Custom线程cond_wait超时后自己`pthread_exit`，确保退出前已释放锁

> **面试官追问**：pthread_cancel为什么不安全？
> - cancel是异步终止，线程可能在任何代码点被杀
> - 如果正持有pthread_mutex_lock，被杀后锁永远不释放→其他线程死锁
> - pthread_cleanup_push/pop可以注册清理函数，但需要在每个锁操作处注册，极易遗漏
> - **线程自毁是最安全的退出方式**

## 性能优化

- **环形队列** vs 链式队列：环形队列预分配内存，入队/出队O(1)，零分配开销；链式队列每次入队需new/delete，高频场景开销不可忽视
- **锁粒度优化**：取任务加锁，执行任务不加锁，最小化持锁时间

> **面试官追问**：如果流量增加10倍，线程池哪里会先出问题？
> - **环形队列容量50000**：10倍流量时队列可能满→生产者阻塞→epoll投递任务被阻塞→整个服务卡住
> - 解决：增大queue_capacity或改为无界队列（但有内存风险）
> - 更好的方案：**多进程部署**，每个进程有自己的线程池，nginx负载均衡分发

## 模拟面试题

### Q1：你的线程池和C++11 std::thread有什么区别？
**回答要点**：
- std::thread每次创建新线程，任务完成后销毁，创建/销毁开销约1ms/次
- 线程池复用线程，任务完成后线程等待新任务，**避免反复创建/销毁**
- 本项目每秒可能处理万级任务，如果每次都new thread，仅线程创建就消耗10秒/万次
- 线程池的10个初始线程复用后，万级任务只需任务切换开销（约1μs/次）

### Q2：环形队列和链式队列的取舍？
**回答要点**：
- 环形队列：预分配内存、零分配开销、但容量固定
- 链式队列：动态扩容、但每次入队需malloc（约100ns/次）
- 本项目选环形队列：任务频率高（万级/秒），malloc开销×万级=1ms/秒，看似不多但加上内存碎片和cache miss，实际影响更大
- **如果流量增加10倍**：环形队列50000可能不够→需要增大容量或改为动态扩容的环形队列（满时realloc翻倍）

### Q3：busy_threads的统计有没有竞态问题？
**回答要点**：
- busy_threads++在取任务时（持锁状态），busy_threads--在任务完成后（单独持锁）
- 两个操作都在pthread_mutex_lock保护下，不存在竞态
- 但**busy_threads的值在Manager读取时可能不精确**（Manager读到的值可能比实际旧1~2个操作），但这不影响扩缩容决策——80%阈值本身是模糊的，±2%误差不影响判断
