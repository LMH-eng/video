# 13 评论系统（发表/拉取/删除 + 敏感词过滤 + 频率限制）

## 模块概述

评论系统实现了**发表/拉取/删除**的完整闭环，服务端包含**敏感词过滤**和**10秒频率限制**（防刷评论），客户端CommentDialog支持分页拉取、实时发表、删除自己的评论。点赞数通过LIKE_COUNT_RS全服广播同步。

## 技术选型与理由

> **面试官追问**：敏感词过滤用什么算法？为什么不用AI审核？

| 方案 | 优势 | 劣势 | 选择理由 |
|------|------|------|----------|
| **字符串匹配敏感词库** | 简单、可控、无依赖 | 无法识别变体、无法理解语义 | 项目规模小，几十个敏感词足够 |
| DFA自动机 | O(m)只遍历评论长度 | 实现稍复杂 | 几十个词不需要DFA |
| AI审核（NLP模型） | 精度高、能理解语义 | 需GPU+训练数据、延迟高 | 投入产出比低 |

> **面试官追问**：如果流量增加10倍，敏感词过滤哪里会先出问题？
> - 词库增长到几千个→字符串匹配O(n*m)→耗时增加
> - **优化**：改用DFA或AC自动机→O(m)一次扫描所有敏感词

## 核心实现流程

### 服务端：敏感词过滤 + 频率限制

```cpp
void CLogic::CommentSubmitRq(sock_fd clientfd, char* szbuf, int nlen) {
    STRU_COMMENT_SUBMIT_RQ *rq = (STRU_COMMENT_SUBMIT_RQ*)szbuf;
    
    // 1. ★ 敏感词检测（遍历敏感词库）
    if(containsSensitiveWord(rq->content)) {
        STRU_COMMENT_SUBMIT_RS rs;
        rs.result = comment_sensitive;
        SendData(clientfd, (char*)&rs, sizeof(rs));
        return;
    }
    
    // 2. ★ 频率限制：Redis记录上次评论时间，10秒内不允许再次发表
    // Redis key="comment:rate:{userId}", TTL=10秒
    // 10秒内再次发表→Redis key存在→返回comment_rate_limit
    // 10秒后→key自动过期→可以再次发表
    
    // 3. MySQL插入评论
    sprintf(sql, "insert into comment(userId,videoId,content,time) values(...);");
    m_sql->Update(sql);
    
    // 4. 返回成功
    STRU_COMMENT_SUBMIT_RS rs;
    rs.result = comment_success;
    rs.commentId = newCommentId;
    SendData(clientfd, (char*)&rs, sizeof(rs));
}
```

> **面试官追问**：频率限制用Redis而不是MySQL的原因？
> - Redis读写O(1)微秒级，MySQL查询毫秒级，差100倍
> - Redis TTL自动过期，不需要手动清理；MySQL需要定时清理过期记录
> - 高并发下Redis更稳定（内存操作），MySQL可能成为瓶颈


> **面试官追问**：Broadcast遍历所有fd逐一SendData→10万用户时怎么办？
> - 10万fd逐一SendData→耗时约1秒（每次SendData约10μs）→延迟不可接受
> - **优化方案**：
>   1. 多线程广播：将10万fd分成10组，每组1万fd，10个线程并行广播→耗时降至100ms
>   2. 发布-订阅模式：客户端订阅Redis频道→服务端publish→Redis推送给所有订阅者→O(1)
>   3. WebSocket广播：更适合Web端，本项目是TCP二进制协议→不适用

## 遇到的难点

### 难点1：评论数一致性

- **问题**：用户A点赞→服务端广播→用户B的缩略图点赞数更新→但用户B正在播放的视频点赞数没更新
- **解决**：LIKE_COUNT_RS广播同时更新两个UI（MainDialog缩略图 + PlayerDialog播放器）

### 难点2：评论删除权限

- **问题**：用户只能删除自己的评论，不能删除别人的评论
- **解决**：CommentDialog中只对自己的评论显示删除按钮
  ```cpp
  // 只有userId == m_userId的评论才显示删除按钮
  if(comment.userId == m_userId) {
      QPushButton *deleteBtn = new QPushButton("删除");
      // ...
  }
  ```

> **面试官追问**：如果用户删除评论后，其他用户的评论列表怎么同步？
> - 当前没有实时同步！其他用户需要手动刷新（翻页）才能看到评论被删除
> - **优化方案**：类似点赞广播，评论删除也广播→所有在线用户实时更新评论列表
> - 但评论删除是低频操作，手动刷新可接受

## 模拟面试题

### Q1：你的敏感词过滤和微信/QQ的有什么区别？
**回答要点**：
- 微信/QQ：AI审核+人工审核+多层过滤（关键词+语义+图片OCR）
- 本项目：简单字符串匹配
- **差距**：无法识别变体（"傻X"→"傻叉"→"sha_x"），无法理解语境（"这个电影太垃圾了"是评论还是辱骂）
- **优化方向**：先加DFA匹配变体，再加简单的语义规则（否定词+敏感词=辱骂）

### Q2：频率限制10秒够吗？如果用户写了一条很长的评论需要2分钟？
**回答要点**：
- 10秒限制的是"发表频率"，不是"编写时间"
- 用户可以花2分钟写评论，但发表后10秒内不能再发表第二条
- **这是合理的**：正常人不可能10秒内写两条有意义的评论
- **极端情况**：用户快速回复多条评论→10秒限制可能影响体验→改为3秒或5秒

### Q3：评论数据如果量很大（一个视频10万条评论），分页查询怎么优化？
**回答要点**：
- 当前：`LIMIT offset, perPage`→offset=100000时MySQL需要扫描100000+10行→慢查询
- **优化1**：用游标分页（cursor-based pagination）→`WHERE id > lastId LIMIT 10`→只扫描10行
- **优化2**：Redis缓存热门视频的评论列表→避免每次查MySQL
- **优化3**：只显示最近100条评论+总数→不需要加载10万条历史评论
