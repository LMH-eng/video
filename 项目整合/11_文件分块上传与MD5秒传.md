# 11 文件分块上传与MD5秒传

## 模块概述

客户端通过`UploadWorker`（QThread子线程）实现文件分块上传：先发STRU_UPLOAD_RQ包头（含MD5哈希）→服务端MD5去重→分块发送STRU_FILEBLOCK_RQ（4KB/块）→逐块确认。支持MD5秒传和QAtomicInt取消标志。

## 技术选型与理由

> **面试官追问**：为什么用4KB分块？为什么在子线程上传？

| 方案 | 优势 | 劣势 | 选择理由 |
|------|------|------|----------|
| **4KB分块+子线程** | 不阻塞UI、逐块确认可靠、内存占用小 | 大文件需很多次网络传输 | 4KB接近TCP MSS；子线程保证UI流畅 |
| 整文件上传 | 一次传输 | 内存占用大、不可中断、失败需从头重传 | 大文件不可行 |
| HTTP multipart | Web标准 | 需HTTP服务器，与TCP二进制协议体系不一致 | 协议体系不一致 |

> **面试官追问**：4KB是不是太小了？
> - 1GB文件需要250000次网络传输，每次都有协议开销
> - **优化方案**：改为64KB分块+断点续传（记录已上传块序号）
> - 但4KB的优势：中途断开只丢失4KB，重传代价小

## 核心实现流程

### 上传流程：包头 → 分块 → 确认

```
客户端                          服务端
  │── UPLOAD_RQ(MD5+大小) ───→│  MD5去重→秒传或创建映射
  │←── UPLOAD_RS(accept/exist)│
  │── FILEBLOCK_RQ(块0) ────→│  写入临时文件→确认
  │←── FILEBLOCK_RS(块0OK) ─│
  │── FILEBLOCK_RQ(块1) ────→│  ...
  │  ... (循环直到文件结束)    │
  │── FILEBLOCK_RQ(块N) ────→│  最后一块→完成→转码→入库
```

### UploadWorker子线程上传

```cpp
void UploadWorker::slot_doUpload(QString filePath, QString imgPath, Hobby hy, int userId, QString title) {
    // 1. 计算文件MD5
    QString fileHash = computeMD5(filePath);
    
    // 2. 发送UPLOAD_RQ包头
    STRU_UPLOAD_RQ rq;
    rq.m_UserId = userId;
    rq.m_nFileSize = fileSize;
    strcpy(rq.m_szFileHash, fileHash.toUtf8().data());
    memcpy(rq.m_szHobby, &hy, sizeof(Hobby));
    strcpy(rq.m_szTitle, title.toUtf8().data());
    m_tcpClient->SendData(0, (char*)&rq, sizeof(rq));
    
    // 3. 循环分块发送
    QFile file(filePath);
    file.open(QIODevice::ReadOnly);
    while(!file.atEnd() && !m_bCancelled) {
        STRU_FILEBLOCK_RQ blockRq;
        blockRq.m_nBlockSize = file.read(blockRq.m_szBlockData, _DEF_BUFFER);  // 4KB
        m_tcpClient->SendData(0, (char*)&blockRq, sizeof(blockRq));
        Q_EMIT SIG_progress(uploadedSize, fileSize);  // 进度信号→UI更新
    }
    
    Q_EMIT SIG_uploadFinish(1);  // 完成信号
}
```

### MD5秒传：服务端去重

```cpp
void CLogic::UploadRq(sock_fd clientfd, char* szbuf, int nlen) {
    STRU_UPLOAD_RQ *rq = (STRU_UPLOAD_RQ*)szbuf;
    
    // ★ MD5去重：查video表filehash字段
    sprintf(sql, "select id,url from video where filehash='%s';", rq->m_szFileHash);
    m_sql->Select(sql, res);
    
    if(!res.empty()) {
        // 视频已存在→秒传！零传输
        STRU_UPLOAD_RS rs;
        rs.result = video_already_exist;
        SendData(clientfd, (char*)&rs, sizeof(rs));
        return;
    }
    // 新视频→创建上传任务→返回upload_accept
}
```

> **面试官追问**：秒传和断点续传的区别？
> - **秒传**：MD5相同→文件完全相同→跳过上传→零传输
> - **断点续传**：上传中断后从断点继续→只传剩余部分→部分传输
> - 本项目只实现了秒传，没有断点续传→中断后需从头重传

### 取消上传：QAtomicInt线程安全标志

```cpp
// MainDialog中取消上传
void MainDialog::cancelUpload() {
    if(m_uploadWorker) {
        m_uploadWorker->cancel();  // 设置m_bCancelled=1
    }
}

// UploadWorker中检查取消标志
void UploadWorker::cancel() {
    m_bCancelled = 1;  // QAtomicInt，线程安全
}

// 上传循环中检查
while(!file.atEnd() && !m_bCancelled) {  // ★ m_bCancelled是原子变量，跨线程安全读取
    // ... 发送文件块
}
```

> **面试官追问**：为什么用QAtomicInt而不是bool+mutex？
> - QAtomicInt是原子操作（CPU级CAS指令），无需加锁→零开销
> - bool+mutex每次读取需要加锁→开销约100ns/次→250000次读取=25ms
> - 取消标志只需要"设置+读取"两个操作，QAtomicInt足够

## 遇到的难点

### 难点1：上传乱序Bug（详见03/05章节）

- UPLOAD_RQ和FILEBLOCK_RQ乱序→服务端找不到上传任务
- 解决：recv_task中同步调用业务回调

### 难点2：秒传后客户端仍继续发送文件块

- **问题**：服务端返回video_already_exist，但客户端上传线程还在循环发送
- **解决**：CKernel收到video_already_exist后调用`m_mainDialog->cancelUpload()`→设置m_bCancelled=1→上传循环退出

## 模拟面试题

### Q1：MD5碰撞概率？怎么进一步降低？
**回答要点**：
- MD5碰撞概率：理论约2^64次运算才可能碰撞，实际视频文件（>1MB）碰撞极低
- **但不是零**：2004年王小云团队证明了MD5碰撞可以在1小时内找到
- **降低方案**：MD5+文件大小双重校验；或改用SHA-256（碰撞概率2^128）
- **生产级**：SHA-256 + 文件大小 + 前1KB内容指纹，三重校验

### Q2：如果10万用户同时上传，哪里会先出问题？
**回答要点**：
- **m_mapUpload内存**：10万×700字节=70MB，可承受
- **临时文件fd**：10万open文件描述符→超过ulimit→需调大或改为异步写入
- **nginx-rtmp转码**：10万ffmpeg进程→CPU爆炸→需排队转码（消息队列+Worker）
- **磁盘IO**：10万文件同时写入→IO瓶颈→改为对象存储

### Q3：你的上传和HTTP Range断点续传有什么区别？
**回答要点**：
- HTTP Range：客户端在HTTP请求头指定Range(bytes=1000-2000)→服务端返回指定范围
- 本项目：自定义TCP协议STRU_FILEBLOCK_RQ，每块4KB，逐块确认
- **区别**：HTTP Range是"拉"模式（客户端请求特定范围），本项目是"推"模式（客户端主动推送每块）
- HTTP Range更适合断点续传（客户端知道断点位置），本项目更适合实时确认（每块都有ACK）
