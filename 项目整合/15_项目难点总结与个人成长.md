## 难点1：倍速播放电音问题（最核心的面试亮点）

### 问题现象
1.5x和2.0x倍速播放时，正常声音叠加一层"电音/金属质感"，听起来像双重声音。

### 排查过程（三层逐级排查）

**第一层：SDL输出层——SDL_MixAudioFormat混音叠加**

```cpp
// 问题代码：
SDL_MixAudioFormat(stream, (uint8_t*)is->audio_buf + is->audio_buf_index,
                   AUDIO_S16SYS, len1, 100);
```

- `SDL_MixAudioFormat`是**混音叠加函数**，把源数据**加到**目标stream已有数据上
- `memset(stream,0,len)`只在循环开始前清零一次
- 倍速时atempo输出样本数变少→需要多次循环填满SDL buffer→每次循环stream指针前进→新数据叠加到前一次循环已写入的区域→**双重声音**

**修复**：改为`memcpy`直接拷贝
```cpp
memcpy(stream, (uint8_t*)is->audio_buf + is->audio_buf_index, len1);
```

**第二层：FFmpeg滤镜层——rubberband相位叠加**

- 代码中优先尝试rubberband滤镜（相位声码器）
- rubberband用FFT→相位调整→IFFT，在人声素材上产生**相位叠加失真**（金属质感）
- WSOLA（atempo）在时域找相似片段→不破坏谐波结构→人声更自然

**修复**：去掉rubberband优先路径，强制只用atempo

**第三层：数据格式层——FLT/S16格式不匹配**

- atempo输出FLT格式（float），但swr_ctx配置为FLTP→S16转换
- FLT和FLTP是不同格式（packed float vs planar float），swr_ctx无法正确处理FLT→S16
- 格式转换错误→音频数据被错误解读→失真

**修复**：添加aformat滤镜在滤镜图内部完成FLT→S16转换
```cpp
// aformat强制输出S16，彻底绕过swr_ctx
snprintf(aformat_args, "sample_fmts=s16:sample_rates=%d:channel_layouts=0x%llX", ...);
avfilter_graph_create_filter(&aformat_ctx, aformat_filter, "fmt", aformat_args, ...);
```

### 面试回答要点

> **面试官追问**：这个排查过程体现了什么能力？
> - **系统性调试**：不是"改一个地方试试"，而是从数据流链路逐层排查
> - 链路：SDL输出层 → FFmpeg滤镜层 → 数据格式层
> - 每层排查：确认该层是否有问题→修复→验证→如果还有问题→继续排查下一层
> - **体现了"不猜测、按链路排查"的工程思维**

> **面试官追问**：如果还有第四层问题怎么办？
> - 继续按链路排查：解码器层（解码器输出格式）→ 编码层（视频编码格式）→ 网络层（RTMP流数据完整性）
> - 每层都有明确的排查方法：打印format/rate/channels、对比原始帧和滤镜输出帧、检查swr_convert参数

## 难点2：上传乱序Bug（服务端架构级问题）

### 问题现象
客户端先发STRU_UPLOAD_RQ紧接着发STRU_FILEBLOCK_RQ，但服务端有时先处理FILEBLOCK_RQ→找不到上传任务映射→日志显示"fileId不存在"。

### 排查过程

1. **发现**：上传时服务端日志显示"FileBlockRq: fileId不存在"，重试又成功
2. **分析**：原设计recv_task接收完数据后，再Producer_add(Buffer_Deal, pData)异步投递业务处理
3. **根因**：两个连续的包被线程池中不同线程处理→线程调度顺序不确定→后发的包可能先被处理
4. **验证**：在Buffer_Deal中打印线程ID和包序号→确认乱序

### 解决方案

```cpp
// 修改前（异步，会乱序）：
Producer_add(Buffer_Deal, pData);  // 投递到线程池，顺序不确定

// 修改后（同步，保证顺序）：
m_recv_callback(sockfd, buf, nPackSize);  // 在recv线程中直接调用
```

### 面试回答要点

> **面试官追问**：同步处理会不会降低吞吐量？
> - 单个fd的业务处理阻塞不影响其他fd（EPOLLONESHOT保证每个fd独立）
> - 线程池200线程，200个fd可并行处理业务逻辑
> - MySQL查询<1ms，影响可忽略
> - **如果业务逻辑耗时>100ms**：需要改为异步+顺序保证（如fd专属队列）

> **面试官追问**：为什么不在FileBlockRq中等待UploadRq处理完成？
> - 这是一种方案（加条件变量等待），但增加了代码复杂度
> - 同步处理更简单：recv_task直接调用回调→同一fd的包按recv顺序处理→天然有序
> - **选择同步的原因：简单可靠，性能足够**

## 难点3：音视频同步调试

### 问题现象
视频快于音频或反之→画面和声音不同步→体验很差。

### 排查过程

1. **分析**：audio_clock和video_clock必须用**原始PTS**在同一时间域比较
2. **验证**：打印audio_clock和video_clock→发现倍速时video_clock没乘speed→视频落后
3. **关键发现**：atempo让音频少输出样本→SDL以原采样率播放→audio_clock前进速率×speed→video_clock也需要×speed才能跟上
4. **但实际实现更简单**：video_clock和audio_clock都用原始PTS→在同一时间域可直接比较→倍速效果自然实现

### 面试回答要点

> **面试官追问**：为什么不用setpts滤镜调整视频PTS？
> - 早期版本用了setpts滤镜（`setpts=PTS*1/speed`）→但setpts改变了PTS值→video_clock和audio_clock不在同一时间域→同步逻辑出错
> - 最终方案：不使用setpts，视频线程同步等待逻辑直接比较原始PTS→倍速时audio_clock自然加速→视频同步等待缩短→自然实现倍速
> - **教训**：滤镜改变PTS会破坏同步逻辑→应该用原始PTS做同步

## 终极模拟面试题

### Q1：用一句话总结你这个项目？
**回答要点**：
> 这是一个类抖音的短视频平台，客户端Qt+FFmpeg+SDL实现音视频播放和倍速变速不变调，服务端Linux epoll+线程池+MySQL+Redis实现高并发业务处理和协同过滤推荐，核心亮点是倍速电音的三层排查修复和上传乱序Bug的同步处理方案。

### Q2：你在这个项目中最大的收获是什么？
**回答要点**：
> 最大的收获是**系统性调试能力**。倍速电音问题不是靠猜测解决的，而是从SDL输出层→FFmpeg滤镜层→数据格式层逐层排查，每层都有明确的验证方法。这种"按链路排查"的思维方式，比任何具体技术知识都更有价值——因为技术会过时，但调试方法论不会。

### Q3：如果让你重新做这个项目，你会改进什么？
**回答要点**：
1. **协议升级机制**：当前28个协议号没有版本管理→加新字段需要新协议号→改为Protobuf或加版本号字段
2. **断点续传**：当前上传中断需从头重传→改为64KB分块+记录已上传块序号
3. **SQL注入防护**：当前sprintf拼接SQL→改为参数化查询
4. **自动重连**：当前断线需重启→改为3次重连+自动重新登录
5. **HLS多码率**：当前RTMP单码率→改为HLS+ABR自适应码率
