## 二、推荐算法核心设计

### 2.1 算法架构：混合推荐（Hybrid Recommendation）

```
                    ┌─────────────────────┐
                    │   RecommendRq       │
                    │   用户ID → Top15视频  │
                    └──────────┬──────────┘
                               │
              ┌────────────────┼────────────────┐
              ▼                ▼                ▼
      ┌──────────────┐ ┌──────────────┐ ┌──────────────┐
      │ 内容匹配推荐  │ │ 协同过滤推荐  │ │ 热门探索推荐  │
      │ (Content-Based│ │(Collaborative│ │ (Hot Explore)│
      │  Filtering)  │ │  Filtering)   │ │              │
      ├──────────────┤ ├──────────────┤ ├──────────────┤
      │ 目标: 10个    │ │ 目标: 3个     │ │ 目标: 2个     │
      │ 标签点积×100  │ │ 相似用户偏好  │ │ 跨标签热门    │
      │ +热度×时间衰减│ │ 余弦相似度    │ │ 打破信息茧房  │
      └──────┬───────┘ └──────┬───────┘ └──────┬───────┘
             │                │                │
             └────────────────┼────────────────┘
                              ▼
                   ┌────────────────────┐
                   │   去重合并 → Top15  │
                   │   不足15→兜底补齐    │
                   └────────────────────┘
```

### 2.2 评分公式

**内容匹配评分**:
```
score = (用户爱好向量 · 视频标签向量) × 100 + hotdegree × POW(0.95, 天数差)
```

展开为SQL:
```sql
score = (
    IFNULL(food,0)*food权重 + IFNULL(funny,0)*funny权重 +
    IFNULL(ennergy,0)*ennergy权重 + IFNULL(dance,0)*dance权重 +
    IFNULL(music,0)*music权重 + IFNULL(video,0)*video权重 +
    IFNULL(outside,0)*outside权重 + IFNULL(edu,0)*edu权重
) * 100
+ IFNULL(hotdegree,0) * POW(0.95, DATEDIFF(NOW(), time))
ORDER BY score DESC, id DESC LIMIT 20
```

**三个优先级**:
1. **标签匹配分 × 100**：用户兴趣越匹配分数越高（主因素）
2. **hotdegree × 时间衰减**：同匹配度下热门优先，但老视频自然降温
3. **id DESC**：时间越新越前

## 三、用户兴趣模型

### 3.1 八维兴趣向量

| 维度 | 含义 | Redis member | 对应分类 |
|------|------|-------------|---------|
| myVec[0] | food | "food" | 美食 |
| myVec[1] | funny | "funny" | 搞笑 |
| myVec[2] | ennergy | "ennergy" | 正能量 |
| myVec[3] | dance | "dance" | 舞蹈 |
| myVec[4] | music | "music" | 歌曲 |
| myVec[5] | video | "video" | 影视 |
| myVec[6] | outside | "outside" | 户外 |
| myVec[7] | edu | "edu" | 教育 |

**存储结构**: Redis ZSet，key = `user:hobby:{userId}`
- member = 维度名称（如 "video"）
- score = 该维度的兴趣权重值

### 3.2 权重的来源与演化

```
注册时(initUserHobbyToRedis):
  勾选的标签 → 初始权重 = 10
  未勾选的标签 → 权重 = 0

行为驱动(updateUserHobby):
  播放视频   → 匹配标签 +5
  播放完成   → 匹配标签 +10
  点赞视频   → 匹配标签 +20
  取消点赞   → 匹配标签 -20

登录衰减(decayUserHobby):
  所有维度 × 0.9（向下取floor）
  score < 1 时清零
  作用: 让行为驱动的兴趣逐渐超越注册时的选择
```

**权重变化示例**（影视用户）:

```
时间线          video权重    说明
─────────────────────────────
注册(勾选影视)       10      initUserHobbyToRedis
首次登录            9        10 × 0.9 = 9 (decay)
播放一个影视视频    14       9 + 5 = 14 (PlayRq)
点赞该视频          34       14 + 20 = 34 (LikeRq)
再次登录           30        34 × 0.9 ≈ 30 (decay)
播放完成另一个      40       30 + 10 = 40 (FinishRq)
```

### 3.3 行为信号的完整链路

```
客户端操作          发送包类型        服务端处理函数        效果
─────────          ─────────        ──────────────        ────
点击播放视频        STRU_PLAY_RQ     PlayRq()              hotdegree+1, 爱好+5
播放完成(进度>90%)   STRU_FINISH_RQ   FinishRq()            爱好+10
点击点赞            STRU_LIKE_RQ     LikeRq(like=1)        hotdegree+1, 爱好+20
取消点赞            STRU_LIKE_RQ     LikeRq(like=0)        hotdegree-1, 爱好-20
请求推荐列表        STRU_RECOMMEND_RQ RecommendRq()        读取最新向量→计算→返回Top15
```

## 四、三路推荐通道详解

### 4.1 通道一：内容匹配推荐（目标10个）

**原理**: 基于内容的过滤(Content-Based Filtering)

**实现**: SQL内联加权评分查询
- 从Redis读取用户8维爱好向量 myVec[0..7]
- 构造带权重的SQL，每个维度乘以对应权重
- 加上时间衰减的热度值
- 取Top 10（实际查20条预留去重空间）

**降级策略**: 如果加权SQL失败（如列不存在），降级为简单的 `ORDER BY id DESC LIMIT 10`

```cpp
// 核心代码片段
sprintf(sqlBuf,
    "SELECT id, ..., "
    "((IFNULL(food,0)*%f + ... + IFNULL(edu,0)*%f) * 100 "
    "+ IFNULL(hotdegree,0) * POW(0.95, DATEDIFF(NOW(), time))) as score "
    "FROM t_VideoData ORDER BY score DESC, id DESC LIMIT 20;",
    myVec[0], myVec[1], ..., myVec[7]);
```

### 4.2 通道二：协同过滤推荐（目标3个）

**原理**: 基于用户的协同过滤(User-Based Collaborative Filtering)

**实现**: getRecommendByCF()
1. 遍历所有其他用户的Redis爱好向量
2. 计算当前用户与每个用户的**余弦相似度**
3. 取相似度 > 0.3 的用户作为"相似用户"
4. 收集这些相似用户上传/点赞过的视频（排除当前用户已有视频）
3. 按相似度加权排序，取Top 3

**余弦相似度公式**:
```
sim(A,B) = (A·B) / (||A|| × ||B||)
```

**适用场景**: 新用户注册后无行为数据时效果有限；活跃用户越多效果越好。

### 4.3 通道三：热门探索推荐（目标2个）

**原理**: 全站热门排行 + 随机探索

**实现**: getRecommendByHot()
1. 查询全站按 hotdegree 降序的视频
2. 如果提供了用户向量参数，对结果做标签匹配二次排序
3. 取Top N条返回

**作用**: 打破信息茧房——即使用户只喜欢影视，也能看到美食/搞笑等其他分类的热门内容。保证推荐的多样性。

### 4.4 兜底补齐机制

当三路去重后总数不足15时：
1. 先用更多热门视频补充
2. 热门不够则用最新视频(`ORDER BY id DESC`)填充
3. 保证始终返回15个视频，客户端格子全部填满

## 五、数据流完整链路

### 5.1 推荐请求处理流程

```
RecommendRq(userId)
│
├─ 1. 从Redis获取用户爱好向量
│   ├─ 成功 → 得到 myVec[8]
│   └─ 失败 → 从MySQL t_UserData读取原始值 × 10 作为初始向量
│
├─ 2. 三路并行收集候选视频ID（用set<int>去重）
│   ├─ 2a. 内容匹配: SQL加权查询 → 最多10个
│   ├─ 2b. 协同过滤: 余弦相似度 → 最多3个
│   ├─ 2c. 热门探索: 全站热门 → 最多2个
│   └─ 2d. 兜底补齐: 热门+最新 → 补足到15个
│
├─ 3. 批量查询视频详情
│   ├─ 查t_VideoData获取 name/url/gif/hobby
│   ├─ 处理GIF路径拼接HTTP地址
│   └─ 查t_UserLike获取实时点赞数
│
├─ 4. 组装STRU_RECOMMEND_RS响应包
│   └─ count = 实际视频数(≤15)
│
└─ 5. SendData发送给客户端
```

### 5.2 客户端接收与展示

```
ckernel.cpp 收到 STRU_RECOMMEND_RS
  → m_mainDialog->slot_dealRecommendRs(rs)
    → 清除旧缩略图
    → 遍历 rs->count 个视频
      → 填充视频信息到UI控件
      → 异步下载GIF并显示为封面图
    → rs->count == 0 时显示空列表
```

## 七、协议定义（关键结构体）

### 7.1 STRU_VIDEO_INFO（单个视频信息，800字节）

```cpp
struct STRU_VIDEO_INFO {
    int id;                  // 4字节 - 视频ID
    int userid;              // 4字节 - 上传者ID
    int likeCount;           // 4字节 - 点赞数
    char name[_MAX_FILE_PATH];  // 260字节 - 标题
    char url[_MAX_FILE_PATH];   // 260字节 - RTMP地址
    char gif[_MAX_FILE_PATH];   // 260字节 - GIF封面URL
    char hobby[8];           // 8字节 - 8维标签(0/1)
};  // 共计 800 字节
```

### 7.2 STRU_RECOMMEND_RS（推荐响应包，12008字节）

```cpp
struct STRU_RECOMMEND_RS {
    int type;                           // 4字节 - 包类型=10021
    int count;                          // 4字节 - 视频数量(≤15)
    STRU_VIDEO_INFO videoList[15];      // 15×800=12000字节
};  // 共计 12008 字节
```

### 7.3 注意事项：32位/64位对齐问题

`STRU_UPLOAD_RQ` 中包含 `int64_t m_nFileSize`，在32位客户端和64位服务端上的对齐方式不同：

- 32位(MinGW): int64_t 按4字节对齐 → 结构体 624 字节
- 64位(Linux): int64_t 按8字节对齐 → 结构体 632 字节

**解决方案**: 使用 `#pragma pack(push, 1)` 强制1字节对齐，确保两端一致。

## 九、测试验证方法

### 9.1 准备测试数据

在MySQL中执行test_data.sql，插入30条覆盖8个分类的测试视频：
- 每个分类3~5条视频
- hotdegree从3到50不等
- 时间跨度1~30天（验证时间衰减效果）

### 9.2 验证步骤

**Step 1: 不同标签用户的推荐差异**
- 注册用户A（只勾"美食") → 登录 → 前10个应为Food类
- 注册用户B（只勾"影视") → 登录 → 前10个应为Video类
- 服务端日志确认爱好向量正确

**Step 2: 行为驱动验证**
- 用影视账号登录 → 记录 video 权重（如34）
- 播放+点赞一个影视视频 → 再次请求推荐
- video 权重应变为 34+5+20=59
- 影视类视频排名进一步提升

**Step 3: 登录衰减验证**
- 当前 video=59
- 退出重新登录 → video 应变为 59×0.9≈53
- 多次登录后逐渐下降，但行为积累的兴趣保持领先

**Step 4: 信息茧房打破验证**
- 无论什么标签用户，最后2个位置应有跨标签热门视频
- 这就是"热门探索"通道的作用

**Step 5: 兜底补齐验证**
- 新用户（无行为数据）→ 协同过滤返回0
- 最终应仍返回15个视频（内容匹配10 + 热门2 + 兜底3）

## 十、面试话术参考

> "我实现的推荐系统采用混合推荐策略，结合了三种算法：
>
> **第一，基于内容的标签匹配**，通过8维兴趣向量和视频标签的点积计算匹配度，加上时间衰减的热度因子，保证推荐的相关性和新鲜度。
>
> **第二，基于用户的协同过滤**，通过余弦相似度找到兴趣相近的用户，推荐他们喜欢的内容，解决冷启动后的个性化发现。
>
> **第三，热门探索通道**，强制注入跨标签的全站热门内容，避免信息茧房问题。
>
> **兴趣向量的更新是行为驱动的**：注册时根据用户选择的标签初始化，之后每次播放、点赞、取消点赞都会实时更新Redis中的向量权重，同时每次登录会做衰减处理让长期兴趣逐渐淡化。
>
> **整个系统用C++实现，服务端用epoll+线程池处理高并发，Redis做兴趣向量的实时存储，MySQL持久化用户和视频数据，Nginx-RTMP提供流媒体播放服务。"
