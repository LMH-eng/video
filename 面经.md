## 为什么处理音视频流的时候使用两个队列而不是一个

我的项目是读包数据的时候，判断是视频流还是音频流，这两个类型的数据包会放到两个不同的队列中去进行处理

每个队列都会有消费者进行数据处理

但是如果使用一个队列的话，流程就会变成直接将这个数据包放进这个队列中，进行单线程消费

### 会发生什么问题

**消费速度不同**

因为一帧音频和一帧视频的数据量相差比较大，如果说此时要解码一个音频帧，但是音频帧前排了几个视频包，但是因为是单线程，就会导致音频需要进行阻塞等待，可能就会导致音频卡顿

但是如果使用两个队列，两个是独立的，就不会被阻塞

**两个队列的区别**

因为在处理弱网或者进行流量控制的时候，需要设置不同的阈值，如果是在一个队列里，有音视频两种数据就不好操作，只能统一其中一个阈值了

**音视频同步需要各自独立的时钟**

视频同步逻辑要对比 audio_clock 和 video_pts ，这是两个独立的时间轴，来自两个队列中各帧的 PTS

合并队列后你还是要维护两份时钟，队列的合并没有带来任何简化

## 服务器是怎么搭建的，流地址怎么获取

### 服务器搭建

服务器是在Linux上搭建的，使用的epoll+线程池模型

epoll_wait来阻塞监听事件，有事件就先判断是客户端还是服务端的事件

服务端的事件就调用accept，去使用epoll_ctl来接收客户端连接

如果是客户端，就将事件投递到线程池进行处理

线程池是生产者消费者模型，消费者从任务队列中拿任务进行处理

### 流地址获取

在上传视频的时候，会先发送元数据，里面包含了用户ID和文件名等数据

在服务端这边，会根据用户ID在数据库中查找出用户名，然后RTMP流地址就是根据用户名和文件名进行拼接得来的

服务端在发起请求的时候，服务端会从数据库查出视频列表，然后对每一个视频都构建出一个GIF元数据包，里面就会带上这个RTMP流地址

## 项目中有考虑过倍速追赶吗

当前项目采用的是以音频时钟为主时钟的被动同步策略，并没有引入倍速追赶机制

### 那为什么不做倍速追赶

项目面向的是短视频场景，一般都比较短，音频延迟不会累积到需要倍速处理的程度，丟帧已经能保证用户体验

如果要做倍速追赶，需要在音频重采样层面做 pitch-preserving 的时间拉伸（如 SoundTouch 库），或直接调整采样率，实现复杂度较高

**如果后续接入直播场景，可以引入动态调整音频采样率的追赶策略，或使用FFmpeg的atempo滤镜实现无级变速**

## 怎么在项目中实现多文件批量上传

1.用户可以选择多个文件，然后为每个文件分配唯一的文件id

2.然后还要专门设置一个 批量上传的文件请求结构体

其中包含文件的数量，和存储文件信息的表

3.服务端预分配资源，返回每个文件的就绪状态

4.然后使用多线程分块上传，每一个文件都是一个单独线程，使用线程池进行控制并发，共享TCP连接，通过文件id区分数据块归属

5.所有文件完成后，服务端发送接收确认

6.客户端接收到之后进行UI进度条更新

## 上传过程中某个文件被改了如何处理

当前项目没有做文件完整性校验

如果上传过程中文件被外部程序修改，会出现以下问题：

文件变大：因为客户端的size（）获取的是打开时的快照大小，循环不会读到新增内容，导致服务端收到的文件不完整

文件变小，或者被删除：读的时候会直接返回0，客户端会误以为传输完成了。服务器就会收到截断文件

内容被篡改：数据块内容中途变化，服务端收到损坏文件，但无感知

### 怎么解决

可以在上传的时候，对文件整体进行MD5进行哈希，并且将哈希之后的结果携带在上传请求中

然后在上传的时候客户端边读边计算MD5

上传完成后，客户端发送最终MD5，服务端对已经接受的文件进行单独计算MD5，对两者进行对比

这样就可以判断是否被修改

## 项目中的架构

### MVC

Model-View-Controller（模型-视图-控制器） 模式

这种模式用于应用程序的分层开发

**Model（模型）**模型代表一个存取数据的对象或 JAVA POJO。它也可以带有逻辑，在数据变化时更新控制器

**View（视图）**视图代表模型包含的数据的可视化

**Controller（控制器）**控制器作用于模型和视图上。它控制数据流向模型对象，并在数据变化时更新视图。它使视图与模型分离开

### 流程

用户点击View，View通知Controller，Controller更新Model，Model数据变化后，通知View更新

### 意图

将应用程序分为三个核心组件：模型（Model）、视图（View）和控制器（Controller），以实现关注点分离

### 主要解决的问题

解决了应用程序中业务逻辑、数据和界面显示的耦合问题，使得开发和维护更加清晰和简单

### 使用场景

当需要将数据、业务逻辑和界面显示分离，以便于独立开发和维护时

### 优点

**关注点分离**：将数据、业务逻辑和界面显示分离，降低耦合度

**易于维护**：每个组件负责特定的任务，便于单独开发和维护

**可扩展性**：可以独立地替换或更新模型、视图或控制器

### 缺点

**可能增加复杂性**：对于简单项目，引入MVC可能会增加不必要的复杂性

**性能问题**：如果不正确使用，可能会导致性能问题

**巨型控制器问题：**View往往会和Controller紧紧的绑在一起，各种请求什么的逻辑都在Controller里，可能会产生出一个巨大的Controller

### 使用建议

当开发大型应用程序，需要清晰分离数据、业务逻辑和用户界面时，考虑使用MVC模式

### 使用案例

**Web应用程序**：用户通过浏览器（视图）发送请求，服务器端的控制器处理请求，模型进行数据处理

## 还有没有别的客户端架构

### MVP

**Model**：同上，只管数据

**View：**只负责显示和接收点击，没有任何逻辑，所有的事情都交给Presenter做，通过接口回调Presenter

**Presenter（协调者）**：相当于 MVC 中被剥离了 UI 的 Controller，它包含所有业务逻辑，它持有 View 的接口，从 Model 拿数据，然后调用 View 的接口更新 UI

**逻辑**

用户点击View，View就会调用Presenter的逻辑方法，Presenter去请求Model，Model返回数据给Presenter，Presenter再调用View的接口，最后View更新界面

**优点：**

职责极其清晰：View 变成纯傻瓜，Presenter 变成纯逻辑

将View与Model进行完全解耦，更利于进行代码的修改

**缺点：**

**接口爆炸**：View每需要一个更新动作，就要在接口里定义一个方法，导致接口类极其庞大

**手动同步**：Presenter拿到数据后，必须手动一行行写 view.setXxx()，如果漏写了一行，UI就没更新，维护成本高

### MVVM

就是对MVP的再次升级，它的核心灵魂是**数据驱动**和**响应式编程**

**Model**：同上，只管数据

**View：**同MVP，只负责显示和接收点击，没有任何逻辑，所有的事情都交给Presenter做，通过接口回调Presenter

**ViewModel**：它是 Model 的视图模型，不仅包含逻辑，还持有状态，当 ViewModel 的状态改变时，View 会自动感知并更新

**流程：**

用户点击 View，View调用ViewModel的逻辑方法，ViewModel去请求 Model，Model 返回数据给 ViewModel，ViewModel 更新自身的状态变量，View 监听到状态变化，自动更新界面

**实现基石：**

数据绑定或响应式流

**优点：**

**不再需要手动更新 UI**：ViewModel只管更新数据，View自动跟着变，这被称为“声明式 UI”，代码量大幅减少

**生命周期安全（自动回收）**：会在View销毁时自动解绑，所以不用担心空指针崩溃等问题

**更易于测试**：ViewModel连View的接口都没有，它纯粹就是逻辑和状态，单元测试比MVP还简单

**缺点：**

**调试困难**：因为全是数据驱动，出了Bug很难追踪是哪个状态变化导致的UI异常，调用栈断断续续

**性能开销**：数据绑定和命令机制可能带来额外的性能开销，特别是在复杂应用中

**过度设计**：对于简单应用，使用MVVM可能导致过度设计

## 写DB的方式还是写内存的方式

项目中采用DB持久化+内存缓存的混合架构

| 数据类型     | 存储方式                       | 说明                                     |
| ------------ | ------------------------------ | ---------------------------------------- |
| 用户账户密码 | MySQL `t_UserData`             | 持久化存储，登录注册直接读写             |
| 视频元信息   | MySQL `t_VideoInfo`            | 包含视频的路径，RTMP地址，分类标签，热度 |
| 推荐记录     | MySQL `t_UserRecv`             | 记录已经推荐给用户的视频，避免重复推荐   |
| 点赞数据     | MySQL `t_Like`                 | 持久化点赞状态                           |
| 用户行为日志 | MySQL `t_UserAction`           | 记录交互行为，供协同过滤算法消费         |
| 用户爱好向量 | Redis Hash                     | 内存存储，用于实时计算余弦相似度         |
| 上传状态     | 服务端内存 `map<int,FileInfo>` | 仅上传期间存在，上传完清理               |

MySQL 保证数据不丢失、支持事务和复杂查询

Redis 存爱好向量是因为推荐引擎需要频繁计算用户间余弦相似度，纯 MySQL 查询延迟不可接受，Redis 的 Hash 操作（HGETALL、HSET、HINCRBY）天然适合维度数据的读写

上传状态用内存 map，因为上传是短生命周期操作，不需要持久化

## 抖音断网后下面几个视频还可以再刷一会

### 预加载缓冲

这个下载是在网络断开前，已经把后面的几秒钟视频，甚至是整个视频都下载完了

就像是线程解码视频一样，在生产者和消费者的中间还有一个缓冲队列作为中间蓄水池

这个时候如果断网了，就会将缓冲队列中的剩余数据包进行取出，解码，并且渲染，直到队列耗尽

### 正规做法（抖音等短视频平台的做法）

在网络层不同于我项目使用的RTMP，抖音的做法就是使用HTTP的分段下载，然后会有一个磁盘缓冲，提前下载当前视频，并且下一个视频提前缓存30-50%的内容

就是在播放当前视频的时候，提前下载完当前视频，然后在播放当前视频一半或者80%的时候，提前下载下一个视频30%左右，足够播放前几秒就可以，然后再下一个视频就可以只下载封面和元信息，不下载视频本体了

那断网的时候，就可以从磁盘中找到提前缓存的视频，就下载多少就放多少

## 如何设计视频预加载

服务器会返回推荐的视频列表，然后会将这个列表发给一个**预加载调度器**

这个预加载调度器会根据视频播放的顺序进行优先级判断，对当前正在播放的视频设置最高优先级，就要提前下载全部，对于后面的视频，就是下载封面和前30%视频体，只下载封面，剩下的就是只下载元信息

本地客户端使用线程池，这样用户视频播放和下载就是并行的，播放线程就是只负责解码+渲染，下载线程就只负责从服务端拉数据写进磁盘，这个两个线程只是通过磁盘缓存做数据交接

#### 预加载调度器

主要是实施一个三级缓存策略

L1（内存）：就是当前正在播放的视频的解码队列

因为就是在内存中的，所以用户看起来是没有延迟的

内存空间宝贵，所以最好存储的数据就在20MB以内，只存5~10个已经解码的GOP

L2（磁盘）：存已经下载但是还没有播放的视频文件

因为是存在磁盘中的，用的时候需要先从磁盘中读取，所以会稍微慢一点点，也几乎是秒播

虽然数据是存在磁盘中的，但是不能存太多数据，要设置上限，大约在200MB以内，就是下边的3~5个视频文件

L3（网络）：就是在服务端的视频

因为是在服务器的，所以受网络影响大，延迟就比较高

#### 预加载什么时候触发

可以在点击推荐页的时候就直接下载前几个视频的封面

在滑动到下一个视频的时候，下一个视频优先级变高，提前下载后边的视频

或者当前视频播放到60%左右的时候，开始下载

当用户停留在推荐页的时候，过了3s左右，就开始下载

网络信号比较好或者WiFi连接时就多下载一些

#### 如果缓存满了怎么办

就使用LRU，最近最少使用，将最早的那个不太可能回播的视频删除

#### 为什么要分等级

保证当前视频的观看体验，当前视频的下载应该是优先级最高的

下一个视频不用下全部，防止浪费，而且下载线部分不会特别大影响用户体验

## 项目中为什么使用水平触发

**水平触发模式下：**

只要缓冲区里还有数据，epoll下次循环还会通知

这样就可以收到多少数据就读多少数据，并且搭配EPOLLONESHOT可以解决线程池的并发冲突，而且代码中使用水平触发模式，可以少写一个循环读的代码，也不用设置非阻塞IO

**边缘触发的话：**

epoll只会在缓冲区状态从有到无的这个瞬间通知一次，所以就需要设置循环读EAGAIN，代码复杂度会比较高

而且因为epoll只会提醒一次，所以如果读队列的时候，因为某些原因，比如说文件内容出错，就会导致读一半退出了，然后缓冲区剩下的东西就可能永远也不会被读到，接受的数据也就是不完整的

### 和短视频项目的关系

短视频项目有两个经典的特征：

**数据量极大且持续流式传输：**用户看的视频，传的视频，都是长时间的大流量，如果读的时候有一次没有读完，这些剩下的数据就会被当成下一个数据的包头进行解析，就会出问题

**并发极高：**在高并发的情况下，开发者极有可能会为了降低上下文切换开销，或者防止单个大流量连接霸占 CPU，而在读取循环中做妥协

### 真要使用边缘触发，该怎么设置

使用非阻塞socket，防止最后一次读空然后阻塞线程

就是在循环读数据的时候设置死循环，直到读完返回整个缓冲区

## 为什么不用多个epoll

我的项目中使用的是单epoll+线程池，所有的IO事件都是通过epoll_wait串行获取的，拿到之后才进行判断并进行分发到线程池

虽然业务处理可以并行，但是这个事件socket的收集是串行的

**为什么不用多epoll**

因为我的项目是C/S架构，同时在线人数不会飙升到epoll_wait处理不了的时候，使用单epoll在Linux上管理已经足够了

**多Reactor的优势**

**解耦**：连接建立 和 IO 读写分离

**高并发**：大量连接由多个从 Reactor 分担

**高可用**：一个从 Reactor 卡住，不影响其他连接

主 Reactor 极简，不会被业务阻塞，连接建立永远快速

## 多epoll+多线程池的架构（多Reactor）

Reactor：事件循环 + IO 多路复用（epoll/kqueue/poll），监听事件、分发任务

多epoll+多线程池就是建立主从epoll

主epoll，也就是主Reactor只做一件事，就是接收服务端socket，接收TCP连接

建立连接后，把fd按照某种策略（轮询 / 最少连接数 / 哈希取模）分配给某个 SubReactor

每个SubReactor在自己的线程里跑自己的epoll_wait循环，监听自己负责的那一批fd

事件到来时，SubReactor自己处理轻量逻辑，或交给线程池处理重量逻辑

## 在项目中怎么实现多Reactor

首先就是要先创建多个epoll+线程池，也就是多个SubReactor对象，每个包含自己的 epoll fd、自己的事件循环线程

然后主Reactor建立连接后，会用某种策略进行从Reactor的选择

然后将fd通过epoll_ctl将目标加到目标的从Reactor中的epoll上

**注意**

epoll_ctl 本身是线程安全的，可以在 Main Reactor 线程直接操作SubReactor的epoll fd，不需要额外加锁

线程池可以共享（需要加锁），也可以每个 SubReactor 自带一个

共享更灵活但锁竞争大，独立更简单但线程总数多

## QThreadPool

项目中没有使用QThreadPool，是在服务端自己实现的一个C语言线程池

我了解到这个Qt的线程池，队列是没有边界的，他是有一个动态扩容机制的，这就导致这个队列不会有满的情况，就不会像我自己写的线程池一样队列满生产者阻塞

在线程空闲30s时，会自动退出，我的线程池会有一个管理线程进行缩减

QThreadPool 的优势 ：

**与 Qt 生态完美集成 ：**QFutureWatcher 可以直接绑信号，任务完成时自动触发 UI 更新，不用手动跨线程通信

**开箱即用 ：**一行 QThreadPool::globalInstance()->start(task) 就行，不需要写队列、条件变量、管理线程

**支持优先级调度 ：**高优先级任务优先执行，适合 UI 场景（如当前播放的视频解码优先级高于后台预下载）

QThreadPool 的劣势 ：

**无界队列 ：**任务投递不受限制，如果生产者速度远大于消费者，内存会无限增长直到 OOM。这在服务端是致命的

**对非 Qt 项目不友好 ：**依赖 QCoreApplication 事件循环，纯 C 或非 Qt 项目无法使用

**线程回收是惰性的 ：**空闲 30 秒才退出，突发流量过后线程不会立刻释放，资源占用偏高

**不透明 ：**你看不到内部有多少线程在干活、队列积压了多少，排查瓶颈不方便

### 为什么自己手写

最主要是自己写的线程池中有：有界队列+生产者阻塞可以在服务端收到大量的上传请求的时候，自动阻塞，可以实现限流的功能，维护服务器稳定

而且自己手写的线程池管理线程可以替更加主动的去进行扩容缩减，Qt的线程池线程回收的周期太长了，会占用资源

并且自己手写的线程池可以通过设置存活，忙碌线程数来进行状态监控

## epoll中监听的服务端客户端socket是什么

其实就是客户端和服务端在进行socket()创建socket的时候会为服务端和客户端都创建一个socket（就是fd）

然后服务端会进行监听，就将服务端fd挂在树上监听，当客户端对服务端发起连接，就会被服务端的监听函数监听到，就触发accept

在accept之后会生成一个新的fd，这个fd就是客户端专属的fd，fd会被挂载在树上，以后有针对这个客户端的请求都会触发这个fd，然后被监听到

## 海量消息一把拉会有什么问题？怎么解决？

首先就是服务端会阻塞，因为服务端要给客户端发送1000个GIF的元信息，就会导致别的客户端请求在服务端阻塞了

然后就是客户端会一次将1000个GIF文件都写入内存和磁盘，导致内存和磁盘都爆满

而且传输也需要时间，会导致用户点击推荐后会等好久，体验下降

然后客户端按照接收到的GIF元信息数量创建接收槽位，用于接收传过来的GIF

而且最好在用户滑到这个页的最后5个视频的时候，再自动进行请求，将页号加一

并且可以配合预加载调度器，服务端先将前5个视频的完整GIF和视频体的前30%进行发送

然后后边的15个视频只有GIF封面

客户端预请求的时候也是只传输GIF封面

## PB（Protobuf）

其实就是自定义二进制协议的工业化版本

### 流程

首先要在 .proto文件中描述信息和服务

然后使用protoc编译器生成目标语言的类

然后通过生成的类将对象进行序列化（对象->二进制）或反序列化（二进制->对象）

### PB的可变长度编码

PB的核心编码方式是Varint，就是可变长度整数，它舍弃了C语言中用4个字节存一个int这种固定长度方式，改为：用少量字节存一个小数字，用多量字节存一个大数字

#### 实现的机理

原先一个字节是8位用来存数据

但是Varint就是最高位存的是标志位，用来表示后续的字节是否还有后续字节，1就是有，0就是没有

剩下的7位才是用来存数据

比如300这个数字，就是1 00101100，有效字节就是9位

将有效字节按照7位进行分隔

**低位放在前面，高位有数据标志位就为1**

低位：`1` 010 1100

高位：`0`000 0010

解码：取出两个字节的低 7 位拼接

#### 为什么要压缩

因为业务中的大部分int值都很小，每一个数字都用4字节去存就比较浪费

#### Length-delimited

配合Length-delimited（字符串/字节数组），PB先写Varint长度，再写内容本身，就可以实现字符的可变长度存储

### 项目中为什么没用PB

首先就是自己手写TCP的二进制协议可以帮助我自己去了解本质，比如内存分布，解决粘包

而且项目的体谅比较小，手写自定义结构体还是可控的

PB使用要引入PB编译器，增加了项目的复杂度

## 为什么使用OpenGL而不是SDL的GPU渲染或者是Qt自带的

### Qt Multimedia

这个渲染工具对于RTMP网络流的支持很差，而且底层是黑盒的，出了问题就比较难定位

### SDL GPU渲染

SDL 的 GPU 渲染本质上是把纹理上传和渲染做了封装，底层还是调 OpenGL

而且SDL没办法做到GPU零拷贝，基本上都是数据先拷贝到SDL的内部缓存，然后再拷贝给OpenGL，中间两次拷贝，延迟就会比较大

## 工业级什么时候用OpenGL

1.在渲染高分辨率视频的时候，因为YUV需要被转换成RGB格式，CPU的格式转换会难以承受这么大的数据量，在GPU中可以并行处理

2.需要低延迟的时候，CPU渲染到GPU显示，这个过程会导致延迟变高

3.跨平台的时候，还有移动端就需要用OpenGL

4.后期进行处理，OpenGL支持后期滤镜美颜等操作，而且延迟很低

## 单次上传超大文件怎么做

项目中没有做大文件传输的优化，可能会出现以下问题：

客户端上传是放在子线程的，一个大文件上传可能就会传好几个小时，用户没法暂停

服务端recv阻塞：服务端一次只处理一个数据块，如果其中一个块的recv堵住了，那这个块的fd会一直没办法处理，导致EPOLLONESHOT会一直锁着这个fd，不能被其他线程获取，就会导致后续的文件fd没办法挂载到epoll上

而且因为我的分块上传块比较小，调用开销会特别大

### 如何解决

在写入的时候，可以先攒一些文件块（16块或者32块），然后进行一次写入，减少系统调用

也可以用：

**多 TCP 连接单文件并发上传**

就是客户端设计多线程上传，每个线程都有一个单独的TCP连接，各自负责上传文件的不同区间

但是会出现问题：

1.服务端要感知这是同一个文件的分片，这就需要在文件元信息结构体中添加两个字段：一个是本次上传任务的唯一标识UID，还有就是当前文件是整个文件的第几个分片

服务器就可以通过UID将不同连接上的文件归到一个文件的不同偏移区间

2.服务端采用的写方式是fwrite，这种写方式是追加写，但是四个链接并不是按顺序传到地方的，所以要用pwrite，参数是文件fd，数据块data，长度len，偏移量

## 项目中如果使用智能指针和右值引用会在哪些地方使用

### 智能指针

项目中使用new+delete的对象都可以使用智能指针

unique_ptr：在上传文件的方法中，服务端收到上传请求，就会创建一个FileInfo对象，但是中间要是出错提前return，就会被永远泄露

还有客户端下载GIF时，在接受完最后一块数据只会需要手动调用回收，这时候可以用unique_ptr

shared_ptr：就是跨线程共享生命周期，基本上就是用在一个对象有多个持有者，并且谁最后一个用完不太确定

例如客户端接收GIF文件，存数据的map和UI是两个独立的生命周期：数据块到达，更新map；数据块接收完成，UI更新，并且map上的数据要删除

如果UI线程正在读 label 上的文件信息，而收包线程同时把map里的对应条目删了，可能导致 label 持有的是被释放的旧数据

这样就可以给map和label都来一个shared_ptr，删除map只会让计数器 -1，label还会持有数据，就可以避免之前的状况

### 右值引用

客户端和服务端的SendData都是先创建一个临时的buffer，然后把这个4 字节长度头 + 结构体拼进去，然后再发送send

这个buffer在send之后就销毁了，再次用到需要重新分配

**右值引用：**如果把这个 buffer 的生命周期搬到外层调用的栈上，通过右值引用直接传给 SendData ，每次发送不需要在 SendData 内部重新分配内存和 memcpy 拼接，只是引用一个已经构建好的连续缓冲区

## 多线程并发的方案

**一个任务一个线程**

每个上传任务创建独立线程，各自读文件、各自发包

这种可能会导致因为多个线程共享一个TCP连接调用send函数，其中某一个线程先发了数据包长度，就被切换到另一个线程发送包体

这样会导致两个线程发送的包交错在一起，第一个字节的字节长度没办法解析后一个线程的包体

**解决方式 ：**给整个发送过程加大锁——从写包长前缀到写完数据体，全程持锁。但这样多个线程实际上是排队发，和单线程没区别，还多了锁开销

**每个线程独立的TCP连接**

每个上传任务建一个 TCP 连接，互不干扰，服务端也对应新线程接收，真正实现了并行

**代价 ：**连接数翻倍，服务端要管理多个连接的文件状态。还要处理"同一文件被两个连接同时上传"的冲突——加文件锁或者串行化

**线程池 + 任务队列**

线程池就是生产者消费者模型，生产者将任务放进任务队列中，消费者拿取，但是还是一个线程全程发完一个数据包，还会出现TCP竞争连接的问题

**总结：**

以上的几种情况要么是只实现了伪并发，比如共享TCP连接＋锁，实际上还是串行连接，实现复杂度中等，但是会有锁开销

要么就是每个进程拥有独立的TCP连接，实现了真并发，但是实现复杂度和连接开销都高

## 项目中如何实现的TCP防粘包

我自己项目中使用的是发送端再发送数据之前，会在发送的数据前进行4个字节的拼接，这4个字节就是这个包的总长度，发送的时候会将4 字节的总包长度 + 结构体数据，一起 send

接收方接收数据的时候，也是先接收4个字节，知道包多大，再循环接受直到接受完一整个包

### 别的方案

**分隔符**

缺点是包里不能出现所添加的分隔符，否则要进行转义

二进制数据（视频帧、结构体字段值）转义代价大。适合纯文本协议，不适合二进制协议

**固定包长度**

所有包固定大小，接收端只需要接收固定大小的包就行

缺点是不灵活，而且小数据占用的包也是固定的，会产生浪费

**自定义消息结构TLV**

自定义一个消息结构，由包头和数据组成，其中包头包是固定大小的，而且包头里由一个字段来说明紧随其后的数据有多大

| 方案           | 实现难度 | 二进制安全 | 带宽效率 |
| -------------- | -------- | ---------- | -------- |
| 分隔符         | 低       | x          | 中       |
| 固定包长度     | 最低     | √          | 低       |
| 项目使用的拼接 | 中       | √          | 高       |
| TLV            | 高       | √          | 最高     |

所以我的项目中选择了较为平衡的方式

用户视频播放的时候实际上是：

客户端发送推荐请求；服务端推荐视频，并且返回GIF缩略图，客户端展示GIF预览

用户点击一个GIF，就能拿到这个视频的RTMP流地址，客户端播放器就可以根据这个流对服务器进行拉流，并且是边拉流边播放

## 分块上传怎么设计的，为什么这么设计

项目中将文件按4KB进行分块，先发文件信息包（文件名、大小、类型），然后循环读 4KB 数据填充到包体，发一块累加一次偏移量，更新一次进度条。循环到文件读完后发最后一块，而且最后一块通常小于 4KB，服务端通过块长度 < 4KB判断文件传输结束

### 为什么选择4KB

**磁盘对齐 ：**操作系统磁盘页大小是 4KB，将文件分块为4KB，刚好一页，不需要跨页读两片拼起来，效率最高

**系统调用开销：**因为每次发送数据都会造成send调用开销，如果设置的过低，就会导致send调用过多，系统调用就会显著增加

**进度条更新：**块分的比较小的话，这个进度条更新显示会比较平滑

**断点续传兼容：**块越大越难做断点续传，如果 1MB 一块发到一半断了，这 1MB 得全部重传，4KB 的重传粒度更细

### 其他方案

**512字节**：进度条极致平滑，重传代价极小
缺点：系统调用爆炸（20MB = 40000 个块），系统调用比数据传输本身还耗时

**1MB：**系统调用次数少（20MB = 20 个块），网络带宽利用率高，大块意味着每个 TCP 段的 MSS 都被填满
缺点：进度条跳动大（20 档），断点续传粒度粗

## 为什么使用TCP自定义协议传输，而不是使用HTTP

项目中客户端和服务端之间跑的是一个自定义的结构体类型，这个结构体在内存中就是一段连续的字节，不需要序列化成JSON，发送的时候使用memcpy到发送缓冲区，接收方memcpy回来，内存会自动对齐

如果使用HTTP的时候，就有HTTP头部开销，有时候传输的数据包比较小的话，可能这个头部开销就比数据包大了

**为什么不使用JSON进行格式封装呢**

因为对于音视频这类高频率数据交互场景，JSON 序列化/反序列化的开销远大于二进制，而且包体积也更大（同样是 40 字节的用户名，JSON 还得加上 "m_szUser":" 这些额外字符）

## JSON

json是一种用于网络传输的数据封装格式（当然不用于网络也可以）

json 是多组key-value键值对形成的字符串

### 为什么要用json？

我们项目的网络通信 使用 自定义协议 使用结构体对数据进行封装。传输的时候可能会有些问题 比如 两端 结构体对齐的方式不一致 可能就有问题。

```c++
struct Node
{
	int type;
    int64_t id;
    char name[60];
}
```

这个结构体在windows下是8字节对齐，在linux ubuntu 1604 32位系统下面，按照4字节对齐。会产生读写错误的问题。

同时，随着业务增长需要，可能会有多端开发，多种语言的情况，为了开发方便，我们将其接口统一，使用json会更合适。

### 序列化和反序列化

**序列化**：就是将数据转入到连续的存储区 或者字符串中，序列化有很多种 其中包含json，protobuf，xml...

**反序列化**：将连续的存储区 或者 字符串中内容还原回原来的数据

项目中的序列化就是：用结构体封装几个数据，让他们转入到连续的存储区中

项目中的反序列化就是：服务端收到数据后的拆包操作，比如：

```c++
char* tmp=buf;//拿到数据
int type=*(int*)tmp;//取头四个字节
tmp+=sizeof(type);//偏移位置
```

这样相较于json会更省空间

### json的格式

```c++
{
    "key":"value",
    "type":1001,
    "id":1,
	"name":"张三",
    "hobby":["唱","跳","rap"],
    "data":[{"name":"李白","tel":"123"},{"name":"杜甫","tel":"456"}],
    "email":null
}
```

可以发现json格式有以下特点：

1、易于读和写（qt，windows，linux下面是有json的库）

2、对于一些特殊数据 比如空值 处理会更安全，它可以直接用null表示空值；如果是我们的自定义协议，结构体，就需要用个特殊标志位之类的取表示该值为空，比较麻烦。

### json的缺点

1、冗余数据太多，不太适合延迟低的场合

2、不方便传输富文本（里面可能包含 图片、音频等数据，可能含有0，那么就字符串截断了）。如果非要用这个传输，可以考虑 利用base64编码（不含0的各种字符），base64编码就是把不可直接打印/传输的二进制，转成只含64个安全字符的文本。

举个例子：使用json 完成二维码登录，移动端已经登录，然后客户端 可以通过扫描二维码的方式 快速登录。此时数据传输就可以是json，里面包含type编码格式，data把二位吧通过base64编码后的字符串。

```c++
{
    "type":"base64",
    "data":"abcAbc+=dadf......"
}
```

另一端拿到data数据可以通过base 64进行解码，就可以还原一个二维码的图片。

### json和XML进行对比？

1、格式对比，json更简短，看起来更干净，可读性也高，也能看出json写的东西少，体积也就小。

```
XML：
<person>
  <name>小明</name>
  <age>18</age>
  <hobby>游戏</hobby>
  <hobby>编程</hobby>
</person>
```

2、浏览器对json天然支持，解析极快，XML的话需要复杂的解析器

3、json自带类型，可以是数字、字符串、数组、甚至可以是null，XML所有内容是字符串，想要换类型只能自己转

## JWT

传统的登陆方式，就是服务器会将浏览器信息存储在服务端，服务端会为每一个用户都分配一个唯一的Session ID，这个ID一般存在Cookie或者URL重写的方式传递给浏览器，以后浏览器请求都会带上这个ID，服务器就会根据这个ID进行数据查找

**弊端**

服务器必须要保存用户的状态，就会比较占内存

多服务器要共享session也比较费劲

### JWT的登陆方式

服务器会把用户的信息加密成一串字符串，把这个字符串发给前端，存在浏览器本地存储Localstorage中，以后每次请求都会带上这个字符串，这样服务器就不用查存储，不用保存用户状态，多服务器也可以随便部署

### JWT包含的字段

JWT令牌主要由三部分构成

**头部（Header）**

就是使用的加密算法（HS256），类型（JWT）

**载荷（Payload）**

主要就是存的数据，比如用户的ID，用户姓名，过期时间

**头部和载荷都是使用Base64编码**

**签名（Base64）**

就是使用头部提供的加密算法

先将头部和载荷都进行base64编码后，与选定的密钥一起进行签名，这个过程不是为了加密，而是为了方式数据被篡改，服务器持有密钥，就可以将拿到的头部，载荷，进行相同的签名，然后与签名出的结果和发来的结果比对，最后确定是否通过

## 如果真用HTTP，怎么实现

上传视频就用POST，下载推荐视频就使用GET，服务端推送GIF数据就使用WebSocket，并将请求和响应进行JSON序列化

## 登录注册

登录和注册的时候都是先进行hash再发送，是使用固定盐，加密码进行拼接，然后进行哈希

但是因为项目中使用的是固定盐，所以如果有黑客攻击了数据库，用这一个盐把常用密码字典全算一遍，就能破解出所有使用弱密码的用户

## 优化

**客户端MD5+服务端bcrypt**

这种可以让攻击者哪怕攻击了数据库，也是抓到了经过二次哈希的bcrypt值，要破解依旧需要高成本

但是如果中间人抓到了这个中间的哈希值，就可以进行重放攻击

**直接使用TLS进行客户端服务器加密通信**

客户端直接传明文密码，中间使用TLS进行加密通信，然后服务端使用bcrypt再次哈希，然后存储

## 那使用TCP和自定义协议进行传输，有没有使用加密

项目中没有使用，因为项目的重心是跑通"采集→编码→上传→推荐→播放"的完整链路

目前项目的客户端和服务端都是在本地的，加密是属于安全层面的需求，目前项目还用不上，但是有考虑过真正要进行加密的时候的设计：

## 设计加密方式

**AES对称加密**

这种是比较轻量级的

客户端服务端建立连接后使用ECDH密钥交换，协商出一个会话密钥

然后在send和recv前都使用密钥进行加密

**缺点**

密钥是写在代码中的，反编译可能就直接泄露了

**使用TLS**

直接把裸 TCP 升级成 TLS Socket

用 OpenSSL 的 **SSL_write / SSL_read** 替换 **send / recv**

服务端持有证书+私钥，客户端验证证书，不需要硬编码密钥——每次连接的会话密钥由 TLS 握手自动协商，一次一密

**优点**

企业级安全标准，经过二十年全球攻防验证

防窃听、防篡改、防重放、身份认证，一次性全解决

OpenSSL 库现成的，不用自己写加密

**缺点**

引入 OpenSSL 依赖，增加部署复杂度

TLS 握手有额外延迟（首次连接多 1-2 个 RTT）

证书管理增加运维成本

## 数据库连接池

当前项目只有一个MySQL连接，线程池所有的工作线程，都共用这一个MySQL连接，每次执行查询的时候，都是

线程A查数据库，先上锁，然后执行SQL，然后解锁，只有解锁了别的线程才能使用这个连接

所以说这些数据库操作都是串行的，会导致很多线程在高峰期的时候事件都是花在等待上

### 怎么设计连接池

**整体架构：**

连接池里头的数据结构包括：空闲连接队列（存放当前可用的连接），已经借出的连接数（就是正忙连接数），信号量（初始值 = 池大小，借一次减 1，还一次加 1），互斥锁（用来保护空闲队列）

**使用流程：**

线程需要查询数据库的时候，先查询信号量，如果信号量大于0，说明有空连接，直接拿着用，如果信号量等于0，说明全部借出， 阻塞等待 直到有人归还

加锁，从空闲队列里取出一个连接，解锁

对取出来的连接做心跳检查，mysql_ping，如果连接断了，调用mysql_real_connect自动重连

线程用这个连接执行自己的SQL，这一步完全不需要加锁，因为每个线程独享自己的连接

执行完毕后， 清理结果集残留，然后把连接放回空闲队列，然后唤醒等待的线程

**池大小的设定：**

这个并不是越大越好

MySQL每多一根连接就多一份内存开销（默认每个连接约4MB），太多连接反而会让MySQL调度开销增大

本项目用的是10条

业务以短查询为主（索引命中后几十毫秒返回），连接周转快

线程池最多 200 个线程，但数据库查询不是持续占用的，5-10 个连接足够削峰

再多的话，MySQL 侧会成为瓶颈（连接数 > CPU 核数时，上下文切换开销反而降低吞吐）

## 如果说线程池服务端连接数量上限了怎么办

要是情况是线程池内的连接已经全部借出了

### 扩容

当池中连接全部借出且等待队列开始堆积时，动态创建新连接加入池中。但 不能无上限扩容 ，必须设一个硬上限（比如 20 个），到上限后**线程阻塞等待或者直接快速失败**

扩容的风险在于：你这边加连接，MySQL那边压力也变大，可能把数据库压垮

要是MySQL 服务端的连接数达到上限

### 排队限流

维护一个等待队列，超过一定长度（比如 50 个等待请求）就直接拒绝新的借连接请求，这是 熔断机制 ，防止雪崩

数据库已经慢了 -> 线程都堵在借连接上 -> 更多线程积压 -> 数据库更慢 -> 恶性循环

## 连接池最大连接数应该跟MySQL的最大连接数一样吗

实际上MySQL的最大连接数只是一个边界作用，是数据库能承受的极限并发连接数

每一根 MySQL 连接都是一个独立的线程，都占用内存（默认约 4MB/连接），都会参与 MySQL 内部的调度和锁竞争

而且将连接池的最大值设置成MySQL的最大连接数，意味着这个连接池就直接占用了数据库所有的连接资源，别的服务需要这个数据库的连接就没得用了

### 关键的原因

连接数不是越大吞吐越高

MySQL的执行模型是一连接一线程，当并发连接数超过CPU核数时，大量的CPU时间会消耗在 线程上下文切换 上，而不是执行SQL，单条SQL的延迟也会因为竞争CPU和锁资源而显著增加

### 那怎么设定

连接池大小 = (核心数 × 2 + 有效磁盘数)

但是实际使用中，大多数还是试错试出来的

就是当事件执行过程中，经常发生"借连接等待"，说明池小了

但是要是MySQL的CUO占用很高，但是每秒查询数（QPS）不高，就是连接太多了

## 为什么数据库不用JSON存

### 格式问题

项目中确实可以用JSON存，因为JSON最主要就是比较灵活，  ，项目中对于用户的爱好就是写定了几个维度，所以目前是固定的模式，用这种原子的列更好一些

### 运算问题

在推荐算法里有点赞操作，点赞的时候会把用户对响应标签的视频的喜爱值给加10，如果就是用列存储，那直接执行SQL这个过程是原子操作，MySQL内部直接对那一列的数值做加法，不需要把数据读到应用层

但是如果是JSON存储的数据，要进行修改的时候，就是

1.先读取这个JSON的hobby的列，读出原始的字符串

2.然后进行解析，JSON_EXTRACT(hobby, '$.food')，这一步就是查找food的值，并返回

3.然后再对返回的值进行+操作

4.使用JSON_SET再次解析原JSON，然后将里面的food的值修改为刚才执行+操作的数，并且重新序列化为新字符串

5.再写回hobby

这个过程需要额外的锁去保障安全性，而且过程需要走一遍序列化和反序列化，开销会比较大

### 存储效率和安全

相对于列的存储，JSON的存储十分占用空间，一个INT在MySQL只会占用4个字节，但是一个"food": 5，都要占用9个字节，一整个记录的JSON hobby字段可能要几百字节，内存占用相当高

而且在安全方面，INT列就是INT，写入字符串会报错，JSON 列里你可以往food字段塞一个字符串 "apple"， MySQL不会报错，但你的推荐算法跑余弦相似度时就会出问题，而且是运行时才出问题，不是编译/写入时

## 什么时候用JSON存

动态Schema（数据库模式），列没办法提前确定

配置项/元数据，每个人一套配置，字段不统一

一次读取、整体使用，不参与 SQL 运算

## 双Token鉴权

这个流程就是使用两个Token来平衡安全性

这两个Token，一个是Access Token，一个是Refresh Token

|          | Access Token     | Refresh Token                |
| -------- | ---------------- | ---------------------------- |
| 有效期   | 短，大概5-15分钟 | 长，7-30天                   |
| 作用     | 访问业务的凭证   | 用来换新的Access Token       |
| 携带方式 | 每次请求都带     | 只在 Access Token 过期时才用 |
| 泄露后果 | 影响窗口短       | 需要立即吊销                 |

### 完整流程

用户登录，服务端验证密码

服务端签发：一个Access Token+一个Refresh Token

客户端把两个Token都存好，每次请求都带上Access Token

服务端验证：解析Token，检查签名，检查是否过期，取出userId

如果说Access Token过期了，服务端就返回401

客户端自动拿Refresh Token去换新的Access Token

服务端验证这个Refresh Token，然后签发新的Access Token（这个过程中也有可能刷新Refresh Token）

如果 Refresh Token 也过期了，跳转登录页，用户重新输入密码

### 为什么要两个Token

因为 Access Token每次都发在网络里，被截获的风险高。有效期短，即使泄露了也很快失效。Refresh Token很少发送（只在换 Token 时用一次），而且只在服务端和客户端之间传，暴露面小···

### Token都存在哪

Access Token：这个不用存在服务端，这个就是一个令牌，JWT，里边是由用户id和用户信息的Base64编码然后加上服务端的密钥，进行哈希签名后在服务端，服务端拆开进行校验，所以不用存

Refresh Token：这个真需要存，因为这个需要吊销能力，如果用户改密码或者账号被盗了，必须让旧的Refresh Token立即失效，不能再让他换新的Access Token了

而Refresh Token一般是随机字符串（不是 JWT），所以需要在服务端保存

一般存在数据库中，客户端发来 Refresh Token -> 服务端去 Redis 查这个 key 在不在 -> 在且没过期 -> 签发新的 Access Token -> 可能也刷新 Refresh Token

## 一致性哈希

也是一种负载均衡算法

正常的哈希，运算都是hash（数据）%服务器数（也就是节点数）

如果节点总数变化，别的数据都要重新计算，就会造成大量的缓存失效

### 一致性哈希怎么做的

一致性哈希主要是将节点，数据都映射到一个环形的哈希空间 （0~2³²）32是无符号整型的32位

再把数据 key 做哈希，在环上**顺时针找到第一个遇到的节点**，数据就归属该节点

这样在修改节点数量的时候，仅会有一小部分的数据需要重新分配

### 优化：虚拟节点

在节点少的时候，可能会导致部分节点负载极高

这时候可以引入虚拟节点：就是一个物理节点映射成多个虚拟节点铺满这个哈希环，让数据均匀分布

实际上就是因为在一致性哈希中，节点和数据都只进行了一次哈希，然后直接映射到环上，有可能节点分布是不均匀的，如果使用虚拟节点，就可以让每个节点都尽可能铺满这个环，然后数据映射的时候可能会更均匀一些

## HTTP中TLS和TCP可以同时握手吗，TCP Fast Open

在操作中，由于TLS是建立在TCP的连接之上的。再者说，TCP三次握手的前两次是不可以携带数据的。

但是，在TCP存在 TCP Fast Open。

1、TCP Fast Open，是绕过TCP三次握手，就可以发数据，可以减少TCP建立连接的延迟。 流程如下：

（1）客户端发送SYN，包含Fast Open，Fast Open内部的cookie为空。

（2）如果服务器支持Fast Open，生成cookie，并将这个cookie放在Fast Open中，发回客户端

（3）客户端收到后，本地缓存Fast Open中的Cookie

随后，客户端手中就存在了cookie，在后续过程中，客户端可以在发送连接的SYN中的 Fast Open中填

入这个cookie，就可以携带数据了。流程如下

（1）客户端发送SYN，Fast Open中添加本地缓存的cookie和一些数据

（2）服务端收到后，验证cookie后，验证成功，就可以将数据发送给应用程序了。 如果验证失败，仅

仅回复SYN + ACK走普通流程。

（3）如果客户端未收到第一个SYN中数据的回复，接下来也会走普通连接的流程，重新发送数据。

此时配合TLS1.3的RTT连接，就可以同时握手了。

## 在TIME_WAIT的时候，收到相同四元组的SYN会发生什么

1、此时分为两种情况：合法SYN（seq大于服务端期待的序列号）和非法SYN（seq小于服务端期待的序

列号）

（1）合法SYN：重新使用当前四元组连接，跳过2MSL，直接SYN_REVD状态。

（2）非法SYN：由于序号非法，会重新发送一个最后一次挥手的ACK，客户端收到后，看到不合理，发

送RST断开连接。

2、在TIME_WAIT状态，收到RST会咋样？

（1）还是看内核参数 tcp_rfc1337，如果是0，收到RST直接关闭，如果是1，丢弃报文。

（2）但是TIME_WAIT是存在实际意义的，不建议直接关闭

## TCP连接，断电和进程崩溃的区别

（1）断电：一瞬间什么都不存在了，（在没有开启保活机制的前提下）服务端如果没有心跳机制，会一

直保持这个连接到程序退出。要是存在心跳机制则会检测到客户端异常，此时发送RST断开连接。

（2）进程崩溃：内核会代替当前进程发送FIN，优雅断开连接。

1、当有数据往来的时候，客户端崩溃，如何处理？

（1）客户端主机崩溃后恢复：无论如何，都会恢复RST断开连接。 情况：①客户端内核不保留任何连接

信息，没有进程绑定这个端口，客户端内核会回复RST。②有进程绑定端口，但是由于崩溃导致socket

丢失，找不到，会回复RST重置连接。

（2）客户端主机一直崩溃：超时重传次数存在阈值，超过这个阈值，自动关闭。

## 拔掉网线，TCP还存在吗

对于TCP连接来说，存在于内核层，当拔掉网线之后，不会对内核层有任何影响，TCP连接状态不会改

变。

1、拔掉网线后，还有数据传输：①如果说拔掉网线后，迅速插上，首先服务端收不到任何客户端的确认

信息，会超时重传，此时插上网线，由于网络连接还是ESTABLISH，可以正常回复服务端信息，正常通

信。②拔掉之后，一直插，服务端的超时重传次数存在限制，会判断出这个TCP连接出现问题，自动发

送RST报文，断开连接。

2、拔掉网线后，没有数据传输：由于keepalive保活机制，会检测到客户端异常，发送RST断开这个连

接。

3、keepalive工作原理：当长时间没有数据往来，会开设一个定时时间，每隔一段时间发送探测报文，

如果连续几次没有回应，说明TCP连接死亡，RST销毁连接

1、TCP两个队列满了，还没有做任何处理，SYN丢弃。

## 内存池

### 为什么引入内存池

为了减少系统调用，避免内存碎片，提高分配速度，内存池可以减少外部碎片

### 内存池怎么实现的

**定长内存池（固定大小分配）**

就是先一次性向系统申请一大块内存，直接用malloc或者new去申请

然后再把这块大内存切成固定的小块，然后用这些小块串成一个空闲链表

分配的时候，直接从链表头摘下一个节点，直接返回

然后释放的时候，把节点重新挂回链表头

**变长内存池（可变大小分配）**

规定一系列标准的大小，比如8，16...2048字节

内部维护16个定长内存池，也就是自由链表，其中的每个节点分别就是对应以上大小

当用户申请N字节时

N>2048，就直接malloc

N<=2048，就将N向上取整到最近的标准大小，用自由链表中的节点进行空间分配

### 内存池多线程与高性能

上面实现的内存池在单线程下非常快，但在多线程下是线程不安全的，如果给alloc和dealloc加互斥锁，锁竞争会极大拖慢性能

**怎么优化**

全局内存池：所有线程共享，由一把全局锁保护

线程本地缓存：每个线程自己维护一个小的内存池

线程申请内存时，优先从自己的线程池中拿，这个过程不用加锁

如果线程自己的内存池空了，就向全局内存池批量申请一批内存，放入自己的内存池中，这个过程要加锁

线程释放内存的时候，也是先归还到自己的内存池中，这个过程也不加锁

然后如果线程自己的内存池中内存比较多，就批量归还给全局内存池

### 内存池的内存归还

很多时候线程只是将自己的空闲内存还给自由链表，但是内存池不会自己调用free或者munmap把内存还给操作系统

1.可以记录每个大块内存都有多少小块的空闲内存，然后当这些小块都空闲了，就一整个将内存还给操作系统

2.将内存划分成页（4KB），然后用链表进行切割，维护一个页到内存池的映射，如果一整页都空闲了，调用mmap解除映射还给操作系统

## 可以通过给函数加const实现重载吗
