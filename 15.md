# 笔记15：短视频平台核心交互系统深度设计方案

## 模块一：高并发视频评论系统

1. 功能目标：建立支持毫秒级同步、敏感词过滤及物理存储一致性的视频下方交互式文字评论模块。
2. 输入输出：
   - 发表评论请求（RQ）：用户唯一标识（4字节整数）、视频唯一标识（4字节整数）、评论正文（1024字节定长字符数组，UTF-8编码）。
   - 发表评论回复（RS）：处理状态码（1字节：0-成功，1-含敏感词，2-频率受限）、新评论ID（4字节整数）、服务器时间戳（8字节长整数）。
   - 拉取评论请求（RQ）：视频ID（4字节）、请求页码（2字节）、每页条数（2字节）。
   - 拉取评论回复（RS）：总条数（4字节）、结果列表（由多个结构块组成：用户名20字节、头像索引2字节、内容1024字节、评论ID 4字节、发表时间10字节字符串）。
3. 核心执行流程：
   - 提交环节：客户端捕获输入 -> 剔除首尾空白符 -> 长度校验（1-300字） -> 构建TCP数据包 -> 服务端通过线程池接收 -> 调用正则扫描算法进行敏感词脱敏（匹配则直接阻断并回包） -> 向数据库评论表插入记录 -> 获取自增主键ID -> 在全局在线用户映射表中检索所有“当前正在观看该视频”的连接描述符集合 -> 将新评论封装为广播包即时推送。
   - 展示环节：客户端进入视频界面 -> 发送拉取请求 -> 服务端执行SQL联表查询（评论表与用户表，依据视频ID过滤） -> 按时间维度降序排列 -> 限制结果集偏移量（Limit/Offset） -> 循环序列化为二进制流发送 -> 客户端UI层清空现有缓冲区 -> 遍历二进制流解析出结构体内容 -> 动态创建评论子部件并按序排布在滚动区域。
   - 清理环节：当用户点击删除按钮 -> 客户端发送包含评论ID与用户ID的请求 -> 服务端校验评论所属权 -> 执行物理删除 -> 触发数据库级联任务或异步清理关联的子级数据。
4. 数据结构设计：
   - 数据库表（t_comment）：id (主键自增), u_id (用户外键), v_id (视频外键), content (VARCHAR 1024), ctime (DATETIME), is_del (TINYINT 标记删除)。
   - 内存缓冲区：服务端维护视频标识到在线连接列表的映射结构，记录每个视频当前的实时观众，用于精准广播。
5. 边界与异常处理：
   - 并发碰撞：同一秒内多人发表时，利用数据库自增主键保证顺序唯一。
   - 断网容错：发送评论后启动3秒超时计时器，若无回包，客户端UI显示错误状态并提示重试。
   - 脏数据清理：若视频被删除，服务端启动清理任务，依据视频ID删除所有关联评论。
6. 依赖说明：
   - SQL存储引擎：用于评论的持久化及分页检索。
   - 正则过滤模块：内置敏感词词库，执行高效的字符串扫描。

## 模块二：用户社交关系（关注/粉丝）系统

1. 功能目标：构建用户间的定向关联映射，实时同步社交计数器，并动态干预推荐算法权重。
2. 输入输出：
   - 社交操作请求：发起者ID（4字节）、目标博主ID（4字节）、操作码（1字节：1-关注，0-取消关注）。
   - 社交操作回复：状态码（1字节：0-操作成功，1-目标不存在，2-禁止自关注）。
   - 状态同步请求：发起者ID（4字节）、博主ID列表。
   - 状态同步回复：关系掩码列表（1字节掩码：0-无关系，1-已关注，2-粉丝，3-互相关注）。
3. 核心执行流程：
   - 建立/解除关系：服务端接收请求 -> 验证标识合法性 -> 开启数据库事务 -> 检查关系表中是否存在对应记录 -> 执行插入或删除操作 -> 触发计数器同步更新双向的粉丝与关注数值 -> 提交事务 -> 向博主推送新增粉丝的系统通知。
   - 权重干预逻辑：当用户关注博主后 -> 服务端提取该博主的兴趣分类标签 -> 在该用户的兴趣特征向量中对相应标签赋予高额权重分值 -> 在后续推荐检索中，优先合并查询已关注博主的最新视频数据。
   - 状态展示逻辑：打开他人主页 -> 并行发送主页信息请求与社交状态请求 -> 服务端检索双向关注表 -> 判断匹配结果（双向关注则返回对应掩码） -> 客户端根据返回值切换UI按钮文字与样式。
4. 数据结构设计：
   - 数据库表（t_relation）：id (主键), follower_id (关注者ID), following_id (被关注者ID)。建立唯一联合索引防止数据重复。
   - 高速缓存：服务端为每个用户维护粉丝数与关注数的数值映射，支持高频原子增减操作。
5. 边界与异常处理：
   - 自我关注：在逻辑层强制拦截发起者ID与目标ID相同的情况。
   - 数据一致性：若数据库更新成功但缓存失效，需提供校准机制通过数据库统计值重新覆盖缓存。
   - 隐私账户：若涉及私密权限，关注流程需增加待审批状态位，由博主确认后方可建立关联。
6. 依赖说明：
   - 高性能缓存服务：用于存储高频读写的粉丝与关注计数，缓解主数据库压力。
   - 数据库事务组件：保证社交关系建立与计数更新的原子性。
