# 14 推荐算法与点赞功能重构

## 一、问题诊断

### 1.1 点赞状态丢失（核心 Bug）

**现象**：点赞后关闭视频，再次打开同一个视频，点赞状态消失（按钮恢复未点赞）。

**根因**：`MainDialog::slot_videoClicked()` 优先使用缓存 RTMP 地址直接播放，**跳过了 PlayRq 请求**。而点赞状态（`isLiked` + `likeCount`）只通过 PlayRq 响应获取，直接播放路径没有这个信息。

```cpp
// 原代码（有Bug）
void MainDialog::slot_videoClicked(int index) {
    QString rtmpUrl = m_videoLabels[index]->getRtmpUrl();
    if(!rtmpUrl.isEmpty()) {
        Q_EMIT SIG_PlayVideo(rtmpUrl, videoId);  // ❌ 直接播放，不获取点赞状态
        return;
    }
    Q_EMIT SIG_PlayRq(videoId, m_id);  // ✅ 但只在RTMP缓存空时才走
}
```

推荐刷新后每个视频都缓存了 RTMP URL，所以**后续所有播放都走第一条路径**，永远不再发 PlayRq，点赞状态永远丢失。

**修复**：`slot_videoClicked()` 改为**始终发送 PlayRq**，每次播放都获取最新点赞状态。

### 1.2 推荐列表不显示赞数

**现象**：推荐视频网格只有 GIF 缩略图，没有点赞数预览。看不到哪个视频更火。

**根因**：`STRU_VIDEO_INFO` 结构体没有 `likeCount` 字段，推荐回复不携带赞数。

**修复**：协议增加 `likeCount` 字段，服务端查询 `t_UserLike` 表填充赞数，客户端在缩略图右下角叠加显示红心+赞数。

### 1.3 注册兴趣权重过高

**现象**：注册时选了"美食"（初始50分），日常点赞"音乐"类视频（每次+10分）。需要点赞6次音乐（60分）才能超过注册时选的美食（50分）。这不合理——抖音的做法是行为兴趣远比注册时填的兴趣重要。

**根因**：原权重体系：

| 行为 | 分值 |
|------|------|
| 注册选中 | 50（太强） |
| 观看视频 | +1（太弱） |
| 点赞视频 | +10（偏弱） |

**修复**：重构权重体系，贴合抖音风格：

| 行为 | 新分值 | 旧分值 | 变化 |
|------|--------|--------|------|
| 注册选中 | **10** | 50 | ↓ 降低5倍 |
| 观看视频 | **+5** | +1 | ↑ 提升5倍 |
| 点赞视频 | **+20** | +10 | ↑ 提升2倍 |
| 取消点赞 | **-20** | -10 | ↑ 提升2倍 |
| 登录衰减 | **×0.9** | 无 | 新增 |

**效果**：注册选"美食"(10分)，点赞1个"音乐"视频就到20分，已经超过美食。2次点赞=40分，远超注册兴趣。登录每次衰减10%，3次登录后注册兴趣只剩约7.3分，几乎可以忽略。

### 2.1 STRU_VIDEO_INFO 增加 likeCount（两端 packdef.h）


## 四、服务端改动

### 4.1 推荐权重重构（clogic.cpp）

| 方法 | 旧值 | 新值 | 说明 |
|------|------|------|------|
| `initUserHobbyToRedis` | 选中=50 | 选中=**10** | 注册兴趣只是冷启动线索，不应长期主导 |
| `LikeRq` delta | +10/-10 | **+20/-20** | 行为权重提升，2次点赞即超过注册兴趣 |
| `PlayRq` delta | +1 | **+5** | 观看也是有意义的兴趣信号 |
| 新增 `decayUserHobby` | 无 | **×0.9** | 每次登录衰减10%，注册兴趣3次登录后约剩7.3 |

### 4.2 新增 decayUserHobby 方法

```cpp
void CLogic::decayUserHobby(int userId)
{
    if(!m_redisConnected) return;
    string key = "user:hobby:" + to_string(userId);
    map<string, double> hobbyMap;
    m_redis->getZSetValsWithScores(key, hobbyMap);

    for(auto& p : hobbyMap) {
        double newScore = p.second * 0.9;
        m_redis->appondOneToZSet(key, p.first, (int)round(newScore));
    }
}
```

在 `LoginRq` 登录成功后调用，衰减注册兴趣。

### 4.3 RecommendRq 增加点赞总数查询

推荐回复填充每个视频时，额外查询 `t_UserLike` 表获取该视频的累计点赞数：

```cpp
int vid = rs.videoList[count].id;
char sqlLikeCount[256] = "";
sprintf(sqlLikeCount, "SELECT COUNT(*) FROM t_UserLike WHERE videoId=%d;", vid);
list<string> lstLikeCount;
if(m_sql->SelectMysql(sqlLikeCount, 1, lstLikeCount) && lstLikeCount.size() > 0) {
    rs.videoList[count].likeCount = atoi(lstLikeCount.front().c_str());

```

降级路径（时间倒序）和协同过滤路径都加了这段查询。

## 五、权重对比：注册兴趣 vs 行为兴趣

### 场景举例

**用户 A**：注册选了"美食"，日常点赞"音乐"类视频

| 时间线 | 美食分数 | 音乐分数 | 推荐倾向 |
|--------|----------|----------|----------|
| 注册完成 | **10** | 0 | 偏美食（冷启动） |
| 登录1次 | **9**（衰减） | 0 | 偏美食 |
| 点赞1个音乐视频 | 9 | **20** | **偏音乐** ✓ |
| 点赞2个音乐视频 | 9 | **40** | **偏音乐** ✓ |
| 登录3次 | **~7.3** | 40 | **偏音乐** ✓ |

旧权重体系下（美食=50，点赞+10）：需要点赞6个音乐视频(60)才能超过美食(50)。

新权重体系下（美食=10，点赞+20）：点赞1个音乐视频(20)就超过美食(10)。

## 八、踩坑记录

1. **缓存RTMP路径导致点赞丢失**：这是最核心的坑。推荐刷新后缓存了 RTMP URL，后续播放都走缓存直接播放，跳过了 PlayRq 请求。而点赞状态只在 PlayRq 响应中返回。修复：始终发 PlayRq。

2. **协议结构体大小变化必须两端同步**：`STRU_VIDEO_INFO` 增加了 `likeCount` 字段后，`STRU_RECOMMEND_RS` 的整体大小变了。如果只改了一端的 packdef.h，TCP 协议解析会错位，整个推荐功能崩溃。**两端必须同时更新**。

3. **注册兴趣衰减不能直接乘浮点数存整数**：Redis ZSet 的 score 是浮点数，但 `appondOneToZSet` 接收 int。衰减后的 score 用 `round()` 转为整数再存回。

4. **点赞按钮 checkable 的样式切换**：`QPushButton::setChecked(true)` 只改变按钮的 pressed 状态，不会自动改变样式。需要在 `setCurrentVideo()` 中手动根据 `isLiked` 设置不同的 `setStyleSheet` 和 `setText`。
