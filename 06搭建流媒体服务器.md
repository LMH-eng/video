### 二、nginx.conf RTMP 核心配置
nginx
rtmp {
    server {
        listen 1935;          # RTMP默认端口
        # 直播配置：项目名videotest
        application videotest{
            live on;
        }
        # 点播配置：项目名vod
        application vod{
            play /home/lmh/video;
        }
    }
}
直播地址：rtmp://IP/videotest
点播地址：rtmp://IP/vod/视频全名.mp4

### 七、初学者常见困惑：直播 vs 点播到底有什么区别？

#### 1. 本质区别一句话
- **直播（Live）**：主播在推流，观众实时接收，**没有完整文件**，边产生边传输，错过了就没了
- **点播（VOD）**：视频文件已经存在服务器上，观众**随时点开随时看**，可以拖进度条

#### 2. 从配置角度理解
| | 直播 application videotest | 点播 application vod |
|---|---|---|
| 关键指令 | `live on;` | `play /home/lmh/video;` |
| 数据来源 | 推流端实时推过来的流 | 服务器本地磁盘上的文件 |
| 需要推流吗 | **必须**，没人推就没有流 | **不需要**，文件已经在那了 |
| 观众能拖进度条吗 | 不能（实时流，没有"过去"） | 能（文件有完整时长） |
| 错过开头能回看吗 | 不能（除非服务器开了录制） | 能，从头播放即可 |

#### 3. 从流程角度理解
```
直播流程：推流端 → ffmpeg推RTMP流 → Nginx-RTMP模块 → 拉流端实时观看
点播流程：视频文件已在磁盘 → 观众请求 → Nginx-RTMP模块读取文件 → 传输给观众
```

#### 4. 初学者最容易搞混的点
- **直播不是"把视频文件放在服务器上让人看"**，那是点播
- **直播必须有推流动作**，推流断了观众就看不到了
- 点播的 `play` 指令指定的是**服务器本地目录**，不是URL
- 直播的 `live on` 不需要指定目录，因为数据来自网络推流

#### 5. RTMP 地址规则再理解
```
直播：rtmp://192.168.18.129/videotest
       ↑ IP          ↑ application名（对应nginx.conf里的application videotest）
       没有文件名！因为流是实时的，不是文件

点播：rtmp://192.168.18.129/vod/103.mp4
       ↑ IP          ↑ application名  ↑ 文件名（对应/home/lmh/video/103.mp4）
       必须有文件名！因为是从磁盘读取指定文件
```

### 八、面试高频问题

#### Q1：直播和点播的区别？
**答**：直播是实时推流、实时拉流，数据边产生边传输，观众看到的是"当下"，不能回退；点播是视频文件已存储在服务器，观众可以随时点开、拖拽进度条、暂停，本质是"读取文件"。

#### Q2：RTMP 是什么？为什么直播常用 RTMP？
**答**：RTMP（Real-Time Messaging Protocol）是 Adobe 设计的实时消息传输协议，基于 TCP，默认端口 1935。直播常用它的原因：
- 延迟低（1-3秒），适合实时场景
- 基于 TCP 传输稳定，不丢包
- 生态成熟，推流端（OBS/ffmpeg）和播放端（ffplay/VLC）都支持
- 缺点：不支持加密（RTMPS 才支持），在浏览器中需要 Flash（已淘汰），所以现在浏览器端更多用 HLS/FLV over HTTP

#### Q3：为什么点播不用 HTTP 而用 RTMP？
**答**：实际上生产环境中点播**更多用 HTTP**（如 Nginx 静态文件服务、HLS .m3u8），更简单、兼容性更好。本项目用 RTMP 做点播是为了**学习和统一协议**，RTMP 模块的 `play` 指令可以方便地读取本地文件并通过 RTMP 协议传输，适合课堂实验。实际工程中点播推荐 HTTP + HLS/DASH。

#### Q4：直播推流用 ffmpeg 命令各参数含义？
```bash
ffmpeg -re -i /home/lmh/116.mp4 -vcodec copy -acodec copy -f flv rtmp://localhost/videotest
```
- `-re`：按原始帧率读取（模拟实时推流，不加这个 ffmpeg 会以最快速度推完）
- `-i`：输入文件
- `-vcodec copy`：视频不重新编码，直接拷贝（省CPU，低延迟）
- `-acodec copy`：音频不重新编码，直接拷贝
- `-f flv`：输出封装格式为 FLV（RTMP 协议要求 FLV 封装）
- 最后是推流目标地址

#### Q5：直播的延迟主要来自哪里？如何降低？
**答**：延迟来源：
1. **编码延迟**：推流端编码耗时 → 用 `-vcodec copy` 不重编可避免
2. **网络传输延迟**：TCP 三次握手 + 丢包重传 → 优化网络、用 CDN
3. **服务端缓冲**：RTMP 服务器默认有缓冲 → 可调小 `buffer` 配置
4. **播放端缓冲**：播放器为流畅播放会缓冲几秒 → 可调小播放器缓冲

降延迟方案：推流端不重编码 + 服务端减小缓冲 + 播放端减小缓冲 + 网络优化，可做到 1 秒左右延迟。如果要求更低（<500ms），需考虑 WebRTC。

#### Q6：HLS 和 RTMP 的区别？为什么生产环境直播常用 HLS？
**答**：
| | RTMP | HLS |
|---|---|---|
| 协议 | TCP 长连接 | HTTP 短连接（拉 .m3u8 + .ts 切片） |
| 延迟 | 1-3秒 | 10-30秒（切片机制导致） |
| 浏览器支持 | 需 Flash（已淘汰） | 原生支持（Safari/Edge）或 hls.js |
| 防火墙穿透 | 1935 端口可能被封 | 走 HTTP/HTTPS 80/443，几乎不被封 |
| CDN 友好 | 不友好（长连接） | 非常友好（就是普通 HTTP 文件） |

生产环境选 HLS 是因为**兼容性和 CDN 分发**，牺牲延迟换通用性。低延迟场景用 RTMP 或 WebRTC。

#### Q7：nginx-rtmp 模块中 `live on` 和 `play` 的区别？
**答**：
- `live on`：开启直播模式，服务器接收推流并转发给拉流端，数据不落盘（除非配置 `record`）
- `play /path`：开启点播模式，服务器从指定目录读取文件，按需传输给客户端，不需要推流

#### Q8：为什么 Nginx 以 nobody 用户运行会导致点播 404？
**答**：Linux 文件权限是**逐级检查**的。Nginx worker 进程默认以 `nobody` 用户运行，要访问 `/home/lmh/video/103.mp4`，需要依次穿透 `/home` → `/home/lmh` → `/home/lmh/video/`，任何一级目录没有 `x`（执行/进入）权限，nobody 就无法到达目标文件。所以只 `chmod 755 /home/lmh/video` 不够，必须 `chmod 755 /home/lmh` 开放父目录的进入权限。

#### Q9：如果让你设计一个短视频平台的后端流媒体架构，你会怎么做？
**答**（分层思路）：
1. **上传层**：用户上传视频 → 转码服务（ffmpeg）生成多清晰度版本（1080p/720p/480p）+ HLS 切片
2. **存储层**：原始文件存对象存储（如 MinIO/OSS），切片存 CDN
3. **分发层**：点播走 HTTP + HLS/MP4，通过 CDN 加速；直播走 RTMP 推流 → 转码 → HLS 分发
4. **调度层**：Nginx 做负载均衡，调度到不同的转码/流媒体节点
5. **关键点**：点播和直播用不同协议栈，点播重存储和CDN，直播重低延迟和转码

### 九、实战踩坑：RTMP 点播无法拖动进度条

#### 问题描述
将播放器路径从本地文件切换为 RTMP 点播地址（`rtmp://192.168.18.129:1935/vod/103.mp4`）后，
视频可以正常播放，但**拖动进度条无效**——松开鼠标后画面不跳转。

#### 原因分析
- **不是代码改动导致的**：进度条拖拽相关代码（eventFilter、slot_sliderReleased、seek 函数）均未修改
- **根本原因**：Nginx-RTMP 模块的 VOD（点播）功能对 **seek（跳转/拖动进度条）支持有限**
- 本地 MP4 文件：FFmpeg 的 `av_seek_frame()` 可以随意跳转到任意时间点
- RTMP VOD 流：数据通过网络从 Nginx 实时拉取，nginx-rtmp 模块对 seek 的实现不完善

#### 验证方法
在 Windows CMD 中用 ffplay 测试同一个 RTMP 地址：
```bash
ffplay rtmp://192.168.18.129:1935/vod/103.mp4
```
如果 ffplay 也无法拖动进度条 → 确认是 RTMP 协议限制，非代码问题

#### 解决方案：点播改用 HTTP 协议

##### 为什么 HTTP 对 seek 友好？
```
HTTP 方式：
客户端请求 → "给我第 60 秒开始的数据"（Range 请求）
→ Nginx 直接定位到文件的字节偏移量返回
→ 本质就是读本地文件的一个区间，想跳哪跳哪

RTMP VOD 方式：
客户端通过 RTMP 协议发 seek 命令
→ Nginx-RTMP 模块内部处理（取决于模块实现是否完善）
→ 很多实现对 seek 支持不完整或有限制
```

##### 改造工作量评估：非常小（仅需改 2 处）

| 改什么 | 怎么改 | 难度 |
|--------|--------|------|
| **Nginx 配置** | 在 nginx.conf 的 `http {}` 块中新增一个 server 块 | 复制粘贴 |
| **宏定义 `_DEF_PATH`** | 把 `rtmp://...` 改成 `http://...` | 改 1 行 |
| **播放器代码** | **零改动**（FFmpeg 的 avformat_open_input 原生支持 HTTP） | 无 |

##### 具体修改内容预览

**1. Nginx 新增配置**（nginx.conf 的 http 块内）：
```nginx
server {
    listen 8080;
    location /vod/ {
        alias /home/lmh/video/;
    }
}
```

**2. 宏定义改为 HTTP 地址**：
```cpp
#define _DEF_PATH "http://192.168.18.129:8080/vod/103.mp4"
```

**3. 播放器所有代码不动**（信号槽、解码线程、seek 跳转、UI 全部兼容）

##### 注意事项
- 改完后需要重启 Nginx 使新配置生效
- 确保 8080 端口未被占用且防火墙放行
- 视频目录权限仍然需要保持 `chmod 755 /home/lmh` 和 `chmod -R 755 /home/lmh/video`
