# 打开、播放、暂停、跳转 —— 功能实现与踩坑全记录

## 一、整体架构：谁干什么？

我们的播放器有三个关键角色：

```
┌─────────────────────────────────────────────────────┐
│  PlayerDialog（UI层）                                │
│  - 负责界面：按钮、进度条、画面显示                     │
│  - 接收用户操作，调用 VideoPlayer 的接口               │
│  - 接收 VideoPlayer 的信号，更新界面                   │
└──────────────┬──────────────────────────────────────┘
               │ 调用 play() / pause() / stop() / seek()
               │ 接收 SIG_getOneImage / SIG_PlayerStateChanged / SIG_TotalTime
┌──────────────▼──────────────────────────────────────┐
│  VideoPlayer（播放引擎，继承QThread）                 │
│  - run() 是一个独立线程，负责读取视频文件、解码         │
│  - 内部启动 video_thread（视频解码线程）               │
│  - 内部启动 SDL 音频回调（音频解码）                   │
│  - 通过信号把画面/状态/总时长发给 UI 层                │
└─────────────────────────────────────────────────────┘
```

**核心原则：UI 线程和播放线程各干各的，通过信号槽通信，不要跨线程直接操作。**


## 二、打开视频

### 流程

```
用户点击"打开"按钮
  → on_pb_open_clicked()
    → 弹出文件选择对话框
    → 如果当前有视频在播放：stop(false) + wait() 等旧线程结束
    → delete 旧的 VideoPlayer，new 一个新的
    → 重新连接信号槽
    → setFileName(path)  ← 这里面会 start() 启动线程
    → 手动设置状态为 Playing
```

### 关键代码

```cpp
void PlayerDialog::on_pb_open_clicked()
{
    // 1. 选文件
    QString path = QFileDialog::getOpenFileName(...);
    if(path.isEmpty()) return;

    // 2. 如果旧视频还在播，先停掉
    if(m_player->isRunning())
    {
        m_player->stop(false);  // 非阻塞式停止
        m_player->wait();       // 等线程真正结束
    }

    // 3. 销毁旧的，创建新的
    delete m_player;
    m_player = new VideoPlayer;

    // 4. 重新连接信号槽（新的对象，旧的连接失效了）
    connect(m_player, SIGNAL(SIG_getOneImage(QImage)), ...);
    connect(m_player, SIGNAL(SIG_PlayerStateChanged(int)), ...);
    connect(m_player, SIGNAL(SIG_TotalTime(qint64)), ...);

    // 5. 设置文件名并启动播放
    m_player->setFileName(path);
    slot_PlayerStateChanged(PlayerState::Playing);
}
```

### 为什么要 delete 再 new，而不是复用？

因为 `VideoPlayer::run()` 里面会 `memset(&m_videoState, 0, ...)` 清空状态。如果复用同一个对象，线程刚结束就 `start()`，Qt 可能报 "QThread: Destroyed while thread is still running"。delete + new 是最干净的做法。


## 三、播放与暂停

### 流程

```
播放：m_player->play()   → 设置 isPause=false，状态切为 Playing
暂停：m_player->pause()  → 设置 isPause=true，状态切为 Pause
```

### 原理

播放线程（run、video_thread、audio_callback）里都有暂停检查：

```cpp
// run() 主循环里
if(m_videoState.isPause)
{
    SDL_Delay(10);  // 暂停时啥也不干，就等
    continue;
}
```

暂停的本质就是：**线程还在跑，但遇到 `isPause=true` 就跳过所有工作，只睡觉。** 这样恢复播放时不需要重新初始化任何东西，把 `isPause` 设回 `false` 就行了。


## 四、跳转（进度条拖动）—— 重点中的重点

### 4.1 跳转的完整流程

```
用户点击/拖动进度条
  → eventFilter 拦截鼠标事件
    → 鼠标按下：计算位置 → 设置滑块值 → m_sliderPressed=true（定时器不再覆盖滑块）
    → 鼠标移动：计算位置 → 设置滑块值（滑块跟着鼠标走）
    → 鼠标松开：m_sliderPressed=false → m_player->seek(滑块值 * 1000000)

seek() 函数：
  → 设置 seek_req=1, seek_pos=目标位置（微秒）

run() 主循环检测到 seek_req：
  → av_seek_frame() 跳转到目标位置
  → 往音频队列和视频队列各塞一个 FLUSH 包
  → 设置 seek_flag_audio=1, seek_flag_video=1

video_thread 收到 FLUSH 包：
  → avcodec_flush_buffers() 清空解码器缓存
  → 跳过所有 pts < seek_time 的帧，直到找到目标帧

audio_callback → audio_decode_frame 收到 FLUSH 包：
  → avcodec_flush_buffers() 清空解码器缓存
  → 跳过所有 pts < seek_time 的帧
```

### 4.2 为什么需要 FLUSH 包？

跳转之后，解码器里面还缓存着旧位置的帧。如果不清空，你会看到/听到跳转前的一小段画面和声音。FLUSH 包就是一个"信号"，告诉解码线程："把缓存清了，从新位置开始。"

### 4.3 为什么需要 seek_flag？

`av_seek_frame()` 只能跳到最近的关键帧（I帧），不一定是目标位置。关键帧到目标位置之间还有几帧 B帧/P帧，这些帧的 pts 比目标位置小。`seek_flag` 的作用就是：**跳过这些 pts 不够的帧，直到找到 pts >= seek_time 的帧才显示。**


## 五、进度条拖动 —— 踩坑全记录

这是整个开发过程中最折磨人的部分，反复改了好几次。下面按时间顺序讲清楚每次的问题和原因。

### 5.1 第一版：事件过滤器 + return true

**做法**：给进度条装事件过滤器，拦截鼠标事件，`return true` 表示"我处理了，不用再往下传了"。

```cpp
bool PlayerDialog::eventFilter(QObject *watched, QEvent *event)
{
    if (watched == ui->slider_videoProcess)
    {
        if (event->type() == QEvent::MouseButtonPress)
        {
            // 计算位置，设值
            return true;  // ← 事件到此为止，不再传给 QSlider
        }
        // ... MouseMove、MouseButtonRelease 同理
    }
    return QDialog::eventFilter(watched, event);
}
```

**问题**：`return true` 把鼠标事件吞掉了，QSlider 内部根本收不到这些事件，所以 QSlider **不会发出** `sliderPressed` 和 `sliderReleased` 信号。

**后果**：
- `slot_sliderPressed()` 没被调用 → `m_sliderPressed` 始终是 `false`
- 定时器每 500ms 执行 `slot_TimerTimeOut()`，发现 `m_sliderPressed == false`，就把滑块拽回当前播放位置
- 用户看到的现象：**点了进度条，滑块闪一下就弹回去了**

**为什么这么解决不行？** 因为 `return true` 的意思是"这个事件我处理完了，不需要别人再处理"。但 QSlider 的 `sliderPressed`/`sliderReleased` 信号是在 QSlider 自己处理鼠标事件时发出的，你把事件拦截了，QSlider 就没机会发信号了。

### 5.2 第二版：ClickSlider 子类化

**做法**：创建 `ClickSlider` 类继承 `QSlider`，重写 `mousePressEvent`/`mouseMoveEvent`/`mouseReleaseEvent`，在重写的方法里手动 `emit sliderPressed()` 和 `emit sliderReleased()`。

```cpp
class ClickSlider : public QSlider
{
    void mousePressEvent(QMouseEvent *event) override
    {
        // 计算位置，设值
        emit sliderPressed();  // ← 手动发信号
    }
    // ...
};
```

**问题**：`.ui` 文件里写的还是 `<widget class="QSlider" name="slider_videoProcess">`，Qt 的 `uic` 工具根据 `.ui` 文件生成代码，生成的还是 `QSlider*`，不是 `ClickSlider*`。

**后果**：ClickSlider 的代码写好了，但**从来没有被创建过**。运行时进度条仍然是普通 QSlider，鼠标事件重写完全没生效。

**为什么？** Qt Designer 的 `.ui` 文件决定了控件的类型。你只在头文件里定义了 ClickSlider，但没告诉 `.ui` 文件用 ClickSlider 替代 QSlider，所以生成的代码还是创建 QSlider。

### 5.3 第三版：修改 .ui 提升 ClickSlider

**做法**：把 `.ui` 文件里的 `QSlider` 改成 `ClickSlider`，加上 `<customwidgets>` 声明。

**问题**：编译报错 —— `incomplete type 'QMouseEvent'`、`incomplete type 'QStyle'`。

**原因**：在头文件里重写虚函数时，参数类型（QMouseEvent）和使用的类（QStyle）必须有完整的定义，不能只是前向声明。`<QSlider>` 头文件里只有 QMouseEvent 和 QStyle 的前向声明，没有完整定义。

**修复**：加上 `#include <QMouseEvent>` 和 `#include <QStyle>`。

**但最终这个方案还是被放弃了**，因为用户反馈跳转功能完全失败。可能的原因是 ClickSlider 的 `event->accept()` 和 `emit sliderPressed()` 的交互方式在某些 Qt 版本下行为不一致。

### 5.4 最终版：事件过滤器 + 直接调用 slot 函数（✅ 成功）

**做法**：回到事件过滤器方案，但**不再依赖 QSlider 的信号**，而是直接调用 `slot_sliderPressed()` 和 `slot_sliderReleased()`。

```cpp
bool PlayerDialog::eventFilter(QObject *watched, QEvent *event)
{
    if (watched == ui->slider_videoProcess)
    {
        if (event->type() == QEvent::MouseButtonPress)
        {
            QMouseEvent *me = static_cast<QMouseEvent*>(event);
            if (me->button() == Qt::LeftButton)
            {
                int val = QStyle::sliderValueFromPosition(
                    ui->slider_videoProcess->minimum(),
                    ui->slider_videoProcess->maximum(),
                    me->x(), ui->slider_videoProcess->width());
                ui->slider_videoProcess->setValue(val);
                slot_sliderPressed();  // ← 直接调用，不靠信号
                return true;
            }
        }
        else if (event->type() == QEvent::MouseMove)
        {
            QMouseEvent *me = static_cast<QMouseEvent*>(event);
            if (me->buttons() & Qt::LeftButton)
            {
                int val = QStyle::sliderValueFromPosition(
                    ui->slider_videoProcess->minimum(),
                    ui->slider_videoProcess->maximum(),
                    me->x(), ui->slider_videoProcess->width());
                ui->slider_videoProcess->setValue(val);
                return true;
            }
        }
        else if (event->type() == QEvent::MouseButtonRelease)
        {
            QMouseEvent *me = static_cast<QMouseEvent*>(event);
            if (me->button() == Qt::LeftButton)
            {
                slot_sliderReleased();  // ← 直接调用，不靠信号
                return true;
            }
        }
    }
    return QDialog::eventFilter(watched, event);
}
```

**为什么这个方案能成功？**

关键区别在于 `m_sliderPressed` 是怎么被设为 `true` 的：

| 方案 | m_sliderPressed 怎么变 true | 结果 |
|------|---------------------------|------|
| 事件过滤器 + return true（第一版） | 靠 QSlider 的 sliderPressed 信号 → 但信号被拦截了，发不出来 | ❌ 永远是 false，定时器一直拽回滑块 |
| ClickSlider 子类（第二/三版） | 靠手动 emit sliderPressed() → 但 .ui 没提升 / 编译问题 | ❌ 代码没生效 |
| **事件过滤器 + 直接调 slot（最终版）** | **事件过滤器里直接调 slot_sliderPressed()** | ✅ 立刻变 true，定时器停止覆盖 |

**核心教训**：`return true` 拦截事件后，QSlider 的信号机制就失效了。既然你拦截了事件，就要自己负责把信号要做的事情也做了——直接调用 slot 函数，绕过信号机制。


## 六、m_sliderPressed 的作用

这个变量是整个跳转功能的关键开关：

```
m_sliderPressed = false  →  定时器每500ms更新滑块位置（跟着播放进度走）
m_sliderPressed = true   →  定时器跳过更新（用户在拖动，别打扰）
```

**时序**：

```
1. 用户按下鼠标 → m_sliderPressed = true → 定时器不再覆盖滑块
2. 用户拖动鼠标 → 滑块跟着鼠标走（事件过滤器里 setValue）
3. 用户松开鼠标 → m_sliderPressed = false → seek 跳转 → 定时器恢复更新
```

**如果 m_sliderPressed 没被设为 true 会怎样？**

用户拖动滑块的同时，定时器每 500ms 把滑块拽回当前播放位置。用户看到的就是：滑块拖到哪就弹回哪，完全没法操作。


## 七、QStyle::sliderValueFromPosition 是什么？

普通 QSlider 点击轨道时，默认行为是：滑块往点击方向移动一页（PageStep），而不是直接跳到点击位置。

`QStyle::sliderValueFromPosition(min, max, x, width)` 的作用是：**根据鼠标点击的 x 坐标，计算出对应的滑块值**。这样点击轨道的任何位置，滑块都能直接跳过去。

```cpp
// 举例：进度条范围 0~300秒，宽度 600像素
// 用户点击在 x=300 的位置（正中间）
int val = QStyle::sliderValueFromPosition(0, 300, 300, 600);
// val = 150，即跳到 2分30秒 的位置
```


## 八、总结：跳转功能为什么这么折腾

| 尝试 | 失败原因 | 根本教训 |
|------|---------|---------|
| 事件过滤器 + return true | 拦截了事件，QSlider 信号发不出 | return true 会阻断控件的内部信号机制 |
| ClickSlider 子类 | .ui 文件没提升，代码没生效 | .ui 文件决定控件类型，头文件定义不够 |
| ClickSlider + .ui提升 | 编译报错 incomplete type | 头文件中重写虚函数需要完整的类型定义 |
| **事件过滤器 + 直接调slot** | **成功** | **拦截事件后，要自己负责做信号该做的事** |

**一句话总结**：当你拦截了 Qt 控件的事件（return true），就等于告诉 Qt "这事我来处理"。那你就要把原来信号该干的活也一并做了——直接调用 slot 函数，不依赖信号。
